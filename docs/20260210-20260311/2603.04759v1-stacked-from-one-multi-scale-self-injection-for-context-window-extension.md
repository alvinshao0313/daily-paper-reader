## 摘要
当代大语言模型（LLMs）有限的上下文窗口仍然是其在不同领域广泛应用的主要瓶颈。虽然在长上下文数据上进行持续预训练提供了一种直接的解决方案，但它会产生高昂的数据获取和计算成本。为了应对这一挑战，我们提出了 SharedLLM，这是一个基于多粒度上下文压缩和查询感知信息获取的新颖框架。SharedLLM 由两个堆叠的短上下文 LLM 组成：一个作为压缩器的底层模型和一个作为解码器的顶层模型。底层模型将长输入压缩为紧凑的多粒度表示，然后将其转发给顶层模型进行上下文感知处理。为了最大化效率，这种信息传递仅发生在最底层，从而绕过了冗长的前向传递和冗余的交叉注意力操作。整个过程（其中顶层和底层模型源自相同的底层 LLM 层）被称为“自注入”（self-injection）。为了支持这种架构，一种专门的基于树的数据结构实现了上下文信息的有效编码和查询感知检索。尽管仅在 8K token 的序列上进行训练，SharedLLM 仍能有效地泛化到超过 128K token 的输入。在多项长上下文建模和理解基准测试中，SharedLLM 取得了优于或等同于强基准模型的性能，在效率和准确性之间达到了最佳平衡。此外，这些设计选择使 SharedLLM 能够显著减少内存占用，并带来显著的推理加速（比流式架构快 2 倍，比编码器-解码器架构快 3 倍）。

## Abstract
The limited context window of contemporary large language models (LLMs) remains a primary bottleneck for their broader application across diverse domains. Although continual pre-training on long-context data offers a straightforward solution, it incurs prohibitive data acquisition and computational costs. To address this challenge, we propose~\modelname, a novel framework based on multi-grained context compression and query-aware information acquisition. SharedLLM comprises two stacked short-context LLMs: a lower model serving as a compressor and an upper model acting as a decoder. The lower model compresses long inputs into compact, multi-grained representations, which are then forwarded to the upper model for context-aware processing. To maximize efficiency, this information transfer occurs exclusively at the lowest layers, bypassing lengthy forward passes and redundant cross-attention operations. This entire process, wherein the upper and lower models are derived from the same underlying LLM layers, is termed~\textit{self-injection}. To support this architecture, a specialized tree-based data structure enables the efficient encoding and query-aware retrieval of contextual information. Despite being trained on sequences of only 8K tokens, \modelname~effectively generalizes to inputs exceeding 128K tokens. Across a comprehensive suite of long-context modeling and understanding benchmarks, \modelname~achieves performance superior or comparable to strong baselines, striking an optimal balance between efficiency and accuracy. Furthermore, these design choices allow \modelname~to substantially reduce the memory footprint and yield notable inference speedups ($2\times$ over streaming and $3\times$ over encoder-decoder architectures).

### 1. 核心问题与整体含义
*   **核心问题**：当前大语言模型（LLMs）在处理超长上下文（如 128K+ tokens）时，面临计算复杂度高（$O(T^2)$）、内存占用大以及持续预训练成本昂贵的问题。
*   **研究背景**：现有的扩展方法中，持续预训练需要海量长文本数据和算力；流式处理（Streaming）虽省内存但推理慢且不兼容 FlashAttention；编解码架构（Encoder-Decoder）则存在特征对齐困难和推理冗余。
*   **整体含义**：本文提出 **SharedLLM** 框架，通过“自注入”机制将同一个短上下文模型堆叠为压缩器和解码器，利用树状结构实现多粒度、查询感知的上下文压缩，旨在以极低的训练成本实现高效的长文本外推。

### 2. 方法论
*   **核心思想**：**自注入（Self-Injection）**。利用预训练模型的前 $M$ 层作为底层模型（压缩器），剩余层作为顶层模型（解码器）。由于两者源自同一模型，隐藏空间天然对齐，无需复杂的对齐训练。
*   **关键技术细节**：
    *   **上下文树（Context Tree）**：将长文本切分为块，构建二叉树。根节点代表整个块（粗粒度），子节点代表子序列（细粒度）。
    *   **查询感知动态检索**：在推理时，根据当前查询（Query）与树节点的语义相似度（余弦相似度），动态决定展开哪些节点。相关部分保留细粒度信息，不相关部分仅保留粗粒度摘要。
    *   **多粒度压缩**：对树中保留的节点进行均匀下采样（Down-sampling），层级越高压缩率越大，从而大幅减少 KV Cache 数量。
    *   **位置感知交叉注意力**：在顶层模型中引入交叉注意力层，并使用块级旋转位置编码（RoPE）来维持全局时序信息。

    *   **指令微调（SFT）**：使用 LongAlpaca 数据集及部分 Llama-3 语料进行微调。
*   **Benchmark**：
    *   **LongBench**：涵盖单/多文档 QA、摘要、少样本学习、代码补全等 14 个任务。
    *   **InfiniBench**：测试超过 100K 长度的极端长文本理解能力。
*   **对比方法**：包括位置插值（PI）、YaRN、CEPE（编解码类）、StreamingLLM（流式类）、Activation Beacon、SnapKV 和 OmniKV 等。

*   **消融实验**：针对注入层位置（顶层 vs 底层）、查询感知机制的有无、噪声注入的影响、块位置编码的作用等进行了详细对比。
*   **充分性评价**：实验设计较为全面，不仅对比了性能指标（PPL, F1），还深入分析了推理延迟和内存占用。对比基准涵盖了近年来长文本领域的主要技术路线，实验结果具有较强的说服力和客观性。

### 6. 主要结论与发现
*   **极强的外推能力**：SharedLLM 仅需在 8K 长度上训练，即可在 128K 长度下保持性能稳定，且 PPL 优于 CEPE 和 Activation Beacon。

### 7. 优点
*   **架构优雅（自注入）**：巧妙利用模型自身层级进行堆叠，极大地降低了新参数的学习难度和对齐成本。
*   **多粒度平衡**：通过上下文树结构，在保留全局语义（粗粒度）和局部细节（细粒度）之间找到了良好的平衡。
*   **推理友好**：兼容 FlashAttention，且通过大幅压缩 KV Cache 解决了长文本推理的内存瓶颈。

### 8. 不足与局限
*   **数据依赖风险**：论文提到由于版权问题缺少 Book-S3 数据集，导致其 PPL 结果在某些维度上受到影响，这暗示了模型性能对高质量长文本语料仍有一定依赖。
*   **检索开销**：虽然整体速度快，但动态树搜索需要额外的一次短前向传递来计算相似度，这在极短文本处理时可能产生额外开销。
*   **复杂逻辑挑战**：对于需要同时精确引用全文多个微小细节的极端复杂逻辑推理任务，多粒度压缩可能会导致部分非查询相关的细节丢失。
