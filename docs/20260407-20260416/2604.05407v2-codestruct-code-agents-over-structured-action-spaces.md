## 摘要
基于大语言模型（LLM）的代码智能体通常将代码库视为非结构化文本，通过脆弱的字符串匹配来应用编辑，而这种方式经常因格式偏移或歧义模式而失败。我们提出将代码库重新构架为结构化动作空间，使智能体在命名的 AST（抽象语法树）实体上而非文本片段上进行操作。我们的框架 CODESTRUCT 提供了用于检索完整语法单元的 readCode，以及用于对语义程序元素应用经过语法验证的转换的 editCode。在 SWE-Bench Verified 基准测试中，针对六种 LLM 的评估显示，CODESTRUCT 将 Pass@1 准确率提高了 1.2-5.0%，同时对大多数模型而言，Token 消耗减少了 12-38%。在基于文本的界面下经常无法生成有效补丁的模型受益最为显著：GPT-5-nano 的准确率提升了 20.8%，其空补丁失败率从 46.6% 降至 7.2%。在 CodeAssistBench 上，我们同样观察到一致的准确率提升（+0.8-4.4%），且成本降低高达 33%。研究结果表明，结构感知界面为代码智能体提供了更可靠的基础。

## Abstract
LLM-based code agents treat repositories as unstructured text, applying edits through brittle string matching that frequently fails due to formatting drift or ambiguous patterns. We propose reframing the codebase as a structured action space where agents operate on named AST entities rather than text spans. Our framework, CODESTRUCT, provides readCode for retrieving complete syntactic units and editCode for applying syntax-validated transformations to semantic program elements. Evaluated on SWE-Bench Verified across six LLMs, CODESTRUCT improves Pass@1 accuracy by 1.2-5.0% while reducing token consumption by 12-38% for most models. Models that frequently fail to produce valid patches under text-based interfaces benefit most: GPT-5-nano improves by 20.8% as empty-patch failures drop from 46.6% to 7.2%. On CodeAssistBench, we observe consistent accuracy gains (+0.8-4.4%) with cost reductions up to 33%. Our results show that structure-aware interfaces offer a more reliable foundation for code agents.

### 1. 核心问题与整体含义（研究动机）
目前的 LLM 代码智能体（如 SWE-agent）在处理代码仓库时存在**抽象层级不匹配**的问题：它们将代码视为扁平的“非结构化文本”，通过行号或字符串匹配进行读写。
*   **脆弱性：** 微小的格式变化（如空格、缩进）会导致字符串匹配失败（"no occurrence"）。
*   **低效性：** 为了修改几行代码，智能体往往需要读取和重新生成大量无关的上下文，浪费 Token。
*   **易错性：** 文本编辑无法保证语法正确性，常导致语法错误或生成“空补丁”。
**核心思想：** 将代码库视为由命名实体（类、函数、方法）组成的结构化空间，让智能体直接在抽象语法树（AST）节点上操作，而非操作字符序列。

### 2. 方法论：结构化动作空间
CODESTRUCT 引入了两个核心的结构感知原语（Primitives），通过 **Model Context Protocol (MCP)** 集成到智能体中：

*   **`readCode`（结构化检索）：**
    *   **多级模式：** 支持目录浏览、文件摘要（仅显示函数/类签名）和实体检索。
    *   **语义定位：** 智能体通过选择器（如 `file.py::ClassName::method`）获取完整的语法单元，避免了行号截断或读取过多无关代码。
*   **`editCode`（结构化编辑）：**
    *   **原子操作：** 提供 `insert`（插入）、`replace`（替换）和 `removal`（删除）操作。
    *   **语法验证：** 在提交修改前，工具会自动验证修改后的 AST 是否符合语法规范，拒绝产生语法错误的编辑。
    *   **格式保持：** 自动处理缩进和上下文对齐，使智能体只需关注逻辑变更。

*   **数据集/基准测试：**

*   **评估模型：** 涵盖了 GPT-5 系列（nano, mini, full）和 Qwen3 系列（8B, 32B, 480B-Coder）。

### 6. 主要结论与发现
*   **性能提升：** 在 SWE-Bench Verified 上，Pass@1 准确率提升了 **1.2%–5.0%**。
*   **效率优化：** 大多数模型的 Token 消耗降低了 **12%–38%**，推理成本降低高达 **33%**。
*   **小模型红利：** **GPT-5-nano** 的表现最为惊人，准确率提升了 **20.8%**，其空补丁失败率从 46.6% 骤降至 7.2%。这表明结构化接口能有效弥补小模型在处理复杂文本格式时的脆弱性。
*   **可靠性：** 结构化操作显著减少了因字符串匹配失败导致的无效尝试。

### 7. 优点与亮点
*   **解耦语义与表示：** 将智能体的“语义意图”与代码的“文本实现”分离，使智能体能像人类开发者一样思考（按函数名操作而非按行号）。
*   **即插即用：** 采用 MCP 协议，无需重新训练模型或修改智能体的规划逻辑即可集成。
*   **语法护栏：** 内置的语法验证机制极大提高了补丁的质量，减少了智能体在错误路径上的无效循环。

### 8. 不足与局限
*   **跨文件分析缺失：** 目前 CODESTRUCT 主要在单文件 AST 范围内操作，尚未显式建模跨文件的继承关系或调用图（虽然这在文本基准中也是普遍限制）。
*   **语言覆盖偏差：** 虽然在 CodeAssistBench 测试了多语言，但核心的 Bug 修复评估（SWE-Bench）仍集中在 Python。
*   **语法依赖：** 该方法要求源代码本身必须是语法正确的才能解析 AST，对于已经存在严重语法错误的代码库可能无法直接启动。
