## 摘要
由大语言模型（LLM）驱动的编程智能体日益依赖于模型上下文协议（MCP）等工具调用协议，在开发者的工作站上执行文件读写。当写入操作因内容过滤、截断或会话中断而失败时，智能体通常无法获得结构化信号，导致草稿丢失并因盲目重试而浪费 Token。我们提出了 Resilient Write，这是一个 MCP 服务器，它在智能体与文件系统之间构建了一个六层持久化写入层。这六层——预检风险评分、事务性原子写入、断点续传分块、结构化类型错误、带外暂存器存储以及任务连续性移交信封——彼此正交且可独立采用。每一层都对应于 2026 年 4 月一次真实智能体会话中观察到的具体故障模式，在该会话中，内容安全过滤器静默拒绝了一份包含脱敏 API 密钥前缀的草稿。在使用该系统撰写本文的过程中，还衍生出了分块预览、格式感知验证和日志分析这三个额外工具。包含 186 个测试的测试集验证了各层的正确性；与朴素基准和防御性基准的定量对比表明，该系统使恢复时间缩短了 5 倍，智能体自我纠错率提升了 13 倍。Resilient Write 已基于 MIT 协议开源。

## Abstract
LLM-powered coding agents increasingly rely on tool-use protocols such as the Model Context Protocol (MCP) to read and write files on a developer's workstation. When a write fails - due to content filters, truncation, or an interrupted session - the agent typically receives no structured signal, loses the draft, and wastes tokens retrying blindly. We present Resilient Write, an MCP server that interposes a six-layer durable write surface between the agent and the filesystem. The layers - pre-flight risk scoring, transactional atomic writes, resume-safe chunking, structured typed errors, out-of-band scratchpad storage, and task-continuity handoff envelopes - are orthogonal and independently adoptable. Each layer maps to a concrete failure mode observed during a real agent session in April 2026, in which content-safety filters silently rejected a draft containing redacted API-key prefixes. Three additional tools - chunk preview, format-aware validation, and journal analytics - emerged from using the system to compose this paper. A 186-test suite validates correctness at each layer, and quantitative comparison against naive and defensive baselines shows a 5x reduction in recovery time and a 13x improvement in agent self-correction rate. Resilient Write is open-source under the MIT license.

### 1. 核心问题与整体含义
*   **研究背景**：随着 Claude Code、Cursor 等 LLM 编程智能体的普及，智能体通过模型上下文协议（MCP）在本地文件系统执行读写已成为常态。
*   **核心问题**：现有的写入路径极其脆弱。当写入操作因内容安全过滤器（如误触 API 密钥检测）、文件截断或会话中断而失败时，智能体通常只能收到模糊的错误提示甚至完全没有信号，导致其陷入盲目重试、浪费 Token 且丢失进度的困境。
*   **整体含义**：本文旨在为 LLM 智能体构建一个“持久化写入表面”，通过结构化的反馈和事务性保障，提升智能体在复杂现实环境中的自主生存和纠错能力。

### 2. 方法论
论文提出了一个包含六个正交层级的架构，各层可独立采用：
*   **L0: 预检风险评分 (Risk Scoring)**：在写入前使用确定性正则分类器对内容评分，识别可能触发下游过滤器的模式（如 API 密钥、PII 等），提前预警。
*   **L1: 事务性原子写入 (Safe Write)**：采用“临时文件 -> fsync -> 哈希校验 -> 原子重命名”流程，确保文件要么完整写入，要么保持原样，防止损坏。
*   **L2: 断点续传分块 (Chunking)**：支持将大文件拆分为编号分块，支持幂等重试和连续性校验，仅需重试失败的分块。
*   **L3: 结构化类型错误 (Typed Errors)**：将模糊的字符串错误替换为机器可读的 JSON 信封，包含错误类型（如 `blocked`）、原因提示及重试配额。
*   **L4: 内容寻址暂存器 (Scratchpad)**：提供带外存储，用于存放不宜进入代码库的敏感数据或二进制大对象。
*   **L5: 任务接力信封 (Handoff)**：生成 `HANDOFF.md`，记录任务状态、文件哈希和后续步骤，以便新会话能无缝接管中断的任务。
*   **扩展工具**：包括分块预览、格式感知验证（LaTeX/Python/JSON 等语法检查）和日志分析。

### 3. 实验设计

### 6. 主要结论与发现
*   **性能提升**：相比朴素方案，Resilient Write 将智能体的**恢复时间缩短了 5 倍**。
*   **可靠性增强**：将数据丢失概率降低了 50 倍，并将智能体的**自纠错成功率提升了 13 倍**（从 5% 提升至 65%）。

*   **实践价值**：通过 `CLAUDE.md` 等约定文件，无需修改智能体核心代码即可实现集成。

### 7. 优点
*   **实用性极强**：直接针对 LLM 编程中最高频的“静默失败”和“内容过滤”痛点。
*   **架构优雅**：六层设计彼此独立，开发者可以根据需求只启用部分功能（如仅 L1+L5）。
*   **无状态重试管理**：通过在响应中嵌入 `retry_budget`，实现了透明且无状态的重试控制。
*   **开源贡献**：提供了完整的开源实现和详细的 MCP 协议集成方案。

### 8. 不足与局限
*   **作用域限制**：目前仅限于单工作区根目录，不支持跨工作区的复杂编排。
*   **缺乏跨文件事务**：虽然单个文件写入是原子的，但不支持多个文件同时写入的“全有或全无”事务（Write-Ahead Logging 尚在未来计划中）。
*   **本地化局限**：日志和暂存器均存储在本地，不支持多机协作或分布式状态同步。
*   **分类器覆盖面**：L0 层的风险评分依赖于预定义的正则模式，对于新型敏感数据格式或非英语 PII 的识别可能存在漏报。
