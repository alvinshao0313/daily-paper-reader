## 摘要
我们提出了 ATLAS-RTC，这是一种用于自回归语言模型的运行时控制系统，可在解码过程中强制执行结构化输出。ATLAS-RTC 在每一步监控生成过程，利用轻量级信号检测与输出契约的偏离，并应用针对性的干预措施，如偏置、掩码和回滚。与事后验证或静态约束解码不同，它以闭环方式运行，能够在错误显现之前进行修正。在结构化生成和工具调用任务中，ATLAS-RTC 将首次尝试成功率提高了 20 到 37.8 个百分点，在失败主导的场景下，延迟降低了高达 88%。结果表明，许多失败源于解码伪影而非对任务的误解，这促使运行时控制成为大语言模型系统中一个独立的层。

## Abstract
We present ATLAS-RTC, a runtime control system for autoregressive language models that enforces structured output during decoding. ATLAS-RTC monitors generation at each step, detects drift from output contracts using lightweight signals, and applies targeted interventions such as biasing, masking, and rollback. Unlike post-hoc validation or static constrained decoding, it operates in a closed loop, enabling correction before errors materialize. Across structured generation and tool-calling tasks, ATLAS-RTC improves first-attempt success rates by 20 to 37.8 percentage points, with up to 88% latency reduction in failure-dominated settings. Results show that many failures arise from decoding artifacts rather than task misunderstanding, motivating runtime control as a distinct layer in LLM systems.

### 2. 方法论
*   **核心思想**：将自回归生成建模为一个**受控随机过程**。在每个解码步骤，控制器观察轨迹、评估风险并施加干预。

    *   **阶梯式干预策略**：根据风险等级 $\rho_t$ 采取不同强度的行动：
        1.  `noop`：无操作。
        2.  `bias`：Logit 偏置。
        3.  `temperature`：调整温度以减少方差。

        5.  `correct`：**中途回滚（Rollback）与重定向**。利用 KV-Cache 截断技术，回退 $n$ 个 Token 并重新引导生成。

### 6. 主要结论与发现
*   **性能提升**：在 JSON 模式任务中，首试成功率从 56.7% 提升至 76.7%（+20pp）；在工具调用任务中，从 28.3% 提升至 58.3%（+30pp）。
*   **特定任务突破**：搜索工具的成功率从 0%（Baseline 完全无法处理格式）提升至 90%。
*   **延迟优化**：在结构化任务中，由于避免了生成冗长的错误字符串，平均延迟反而下降了（最高下降 87.9%）。
*   **核心发现**：许多智能体失败源于“解码伪影”（如模型喜欢说“好的，这是你要的 JSON...”这种前导词），而非模型不懂任务。

### 7. 优点与亮点
*   **阶段感知控制**：这是该方法最大的亮点。它不在“值生成区”乱加干预，保留了模型的表达能力，只在结构关键点“亮剑”。
*   **中途回滚机制**：不同于传统的“错了就重来”，ATLAS-RTC 能在生成中途发现苗头不对并局部修正，极大地节省了算力。
*   **闭环治理**：将生成过程视为可控系统，填补了提示词工程和事后验证之间的空白。

### 8. 不足与局限
*   **语义 vs. 结构**：系统只能保证“格式正确”，无法保证“内容真实”。如果模型生成了格式完美的错误数据，系统无法识别。
*   **特定模式下的退化**：在处理包含多行字符串的复杂 Schema 时表现不佳。
*   **语义缺失导致的失败**：如果模型漏掉了必要字段（语义缺失），强制结构约束反而可能导致更严重的报错。
*   **开销风险**：在模型本身表现已经很好的任务中，逐 Token 的监控会带来额外的计算延迟。
