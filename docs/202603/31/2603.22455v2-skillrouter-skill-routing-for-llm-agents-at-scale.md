## 摘要
可重用技能使大语言模型（LLM）智能体能够将特定任务的程序、工具功能（affordances）和执行指南封装成模块化的构建块。随着技能生态系统增长到数万个条目，在推理时暴露所有技能变得不可行。这产生了一个技能路由问题：给定一个用户任务，系统必须在下游规划或执行之前识别出相关的技能。现有的智能体技术栈通常依赖于渐进式披露，仅暴露技能名称和描述，而隐藏完整的实现主体。我们在一个基于 SkillsBench 衍生的基准测试上检查了这一设计选择，该基准包含约 8 万个候选技能，针对具有严重重叠的大型技能注册库这一具有实际重要性的场景。在该场景下的代表性稀疏、稠密和重排序基准模型中，隐藏技能主体会导致路由准确率下降 31 到 44 个百分点，这表明在这一设定下，完整的技能文本是关键的路由信号，而非次要的元数据改进。受此发现启发，我们提出了 SkillRouter，这是一个紧凑的 1.2B 参数全文本检索与重排序流水线。SkillRouter 在我们的基准测试中实现了 74.0% 的 Hit@1——这是我们评估的基准模型中表现最强的平均 top-1 路由性能——同时其参数量减少了 13 倍，运行速度比最强的基础流水线快 5.8 倍。在一项针对四个编程智能体的补充性端到端研究中，路由性能的提升转化为任务成功率的提高，且对于能力更强的智能体，提升效果更为显著。

## Abstract
Reusable skills let LLM agents package task-specific procedures, tool affordances, and execution guidance into modular building blocks. As skill ecosystems grow to tens of thousands of entries, exposing every skill at inference time becomes infeasible. This creates a skill-routing problem: given a user task, the system must identify relevant skills before downstream planning or execution. Existing agent stacks often rely on progressive disclosure, exposing only skill names and descriptions while hiding the full implementation body. We examine this design choice on a SkillsBench-derived benchmark with approximately 80K candidate skills, targeting the practically important setting of large skill registries with heavy overlap. Across representative sparse, dense, and reranking baselines on this setting, hiding the skill body causes a 31--44 percentage point drop in routing accuracy, showing that full skill text is a critical routing signal in this setting rather than a minor metadata refinement. Motivated by this finding, we present SkillRouter, a compact 1.2B full-text retrieve-and-rerank pipeline. SkillRouter achieves 74.0% Hit@1 on our benchmark -- the strongest average top-1 routing performance among the baselines we evaluate -- while using 13$\times$ fewer parameters and running 5.8$\times$ faster than the strongest base pipeline. In a complementary end-to-end study across four coding agents, routing gains transfer to improved task success, with larger gains for more capable agents.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）智能体生态的发展，开发者将复杂的工具调用和操作流程封装为“技能（Skills）”。当技能库规模达到数万级时，将所有技能放入上下文是不现实的，因此需要**技能路由（Skill Routing）**：即根据用户任务从海量库中检索出最相关的技能。
*   **研究动机**：现有的智能体框架通常只向路由系统暴露技能的名称和描述（元数据），而隐藏具体的实现代码（主体）。
*   **核心发现**：在具有高度功能重叠的大规模技能库中，仅靠元数据会导致路由准确率大幅下降（31-44%）。论文强调，**技能实现全文（Full Text）**是精准路由的关键信号。

### 2. 方法论
论文提出了 **SkillRouter**，一个轻量级（1.2B参数）的全文检索与重排序流水线。
*   **核心思想**：采用“双编码器检索 + 交叉编码器重排序”的两阶段架构，且两个阶段均利用技能的完整文本。
*   **关键技术细节**：
    *   **双编码器（SR-Emb-0.6B）**：基于Qwen3-Emb-0.6B微调，使用InfoNCE损失函数。通过LLM生成合成查询进行训练，并采用硬负采样（语义近邻、词法匹配、同类别干扰）。
    *   **交叉编码器（SR-Rank-0.6B）**：对检索出的Top-20候选进行精排。
    *   **训练优化**：
        1.  **假负例过滤（False-Negative Filtering）**：通过名称去重、文本重叠度（Jaccard相似度）和嵌入相似度过滤掉库中功能等价的技能，避免干扰对比学习。
        2.  **列表式重排序损失（Listwise Reranking Loss）**：采用列表式交叉熵而非点对点分类，使模型在相似候选之间进行细粒度比较。

### 3. 实验设计
*   **基准测试（Benchmark）**：基于 SkillsBench 构建，包含 75 个核心查询（24 个单技能，51 个多技能任务）。
*   **技能库规模**：约 80,000 个候选技能。
*   **场景划分**：
    *   **Easy 模式**：自然技能库。
    *   **Hard 模式**：额外加入 780 个由 LLM 生成的“功能相似但错误”的干扰项。
*   **对比方法**：

### 6. 主要结论与发现
*   **全文信号至关重要**：隐藏技能主体会导致路由性能崩溃。
*   **小模型胜过大模型**：经过特定任务微调的 1.2B SkillRouter 在路由准确率（74.0% Hit@1）上超过了参数量大 13 倍的 16B 基准模型。
