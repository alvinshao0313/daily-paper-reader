## 摘要
智能体强化学习（Agentic RL）通过在数据收集的 rollout 阶段和策略训练阶段之间交替，使大语言模型（LLM）能够解决复杂任务。在 rollout 过程中，智能体生成轨迹（trajectories），即 LLM 与外部工具之间的多步交互。然而，频繁的工具调用会导致长尾轨迹生成，从而成为 rollout 的瓶颈。这源于忽略轨迹上下文的以步骤为中心（step-centric）的设计，为长尾轨迹生成引发了三个系统问题：排队延迟、干扰开销和单 token 时间膨胀。我们提出了 Heddle，这是一个以轨迹为中心（trajectory-centric）的系统，旨在优化智能体 rollout 执行的时间、位置和方式。Heddle 集成了三个核心机制：利用运行时间预测和渐进优先级进行轨迹级调度，以最小化累积排队；通过预排序动态规划和工具调用空闲期间的机会性迁移实现轨迹感知放置，以最小化干扰；以及轨迹自适应资源管理器，动态调整模型并行度以加速长尾轨迹的单 token 时间，同时保持短轨迹的高吞吐量。对多种智能体强化学习工作负载的评估表明，Heddle 有效地消除了长尾瓶颈，与最先进的基准方案相比，端到端 rollout 吞吐量提高了多达 2.5 倍。

## Abstract
Agentic Reinforcement Learning (RL) enables LLMs to solve complex tasks by alternating between a data-collection rollout phase and a policy training phase. During rollout, the agent generates trajectories, i.e., multi-step interactions between LLMs and external tools. Yet, frequent tool calls induce long-tailed trajectory generation that bottlenecks rollouts. This stems from step-centric designs that ignore trajectory context, triggering three system problems for long-tail trajectory generation: queueing delays, interference overhead, and inflated per-token time. We propose Heddle, a trajectory-centric system to optimize the when, where, and how of agentic rollout execution. Heddle integrates three core mechanisms: trajectory-level scheduling using runtime prediction and progressive priority to minimize cumulative queueing; trajectory-aware placement via presorted dynamic programming and opportunistic migration during idle tool call intervals to minimize interference; and trajectory-adaptive resource manager that dynamically tunes model parallelism to accelerate the per-token time of long-tail trajectories while maintaining high throughput for short trajectories. Evaluations across diverse agentic RL workloads demonstrate that Heddle effectively neutralizes the long-tail bottleneck, achieving up to 2.5$\times$ higher end-to-end rollout throughput compared to state-of-the-art baselines.

#### 1. 核心问题与研究动机
*   **研究背景**：智能体强化学习（Agentic RL）通过 LLM 与外部工具的多步交互（轨迹生成）来解决复杂任务。研究发现，Rollout（数据收集）阶段占据了整个训练时间的 80% 以上，是主要的性能瓶颈。
*   **核心问题**：现有的 RL 框架（如 Verl, Slime）采用“以步骤为中心”（Step-centric）的设计，将每一步交互视为独立请求。这种设计忽略了轨迹的整体上下文，导致长尾轨迹（步骤多、耗时长）面临严重的**排队延迟**、**计算干扰**以及**单 token 生成速度慢**的问题，产生显著的“掉队者”（Straggler）效应，拖慢整个集群的进度。

#### 2. 核心方法论
Heddle 提出了“以轨迹为中心”（Trajectory-centric）的设计，从三个维度优化 Rollout：
*   **轨迹级调度（解决“何时”执行）**：
    *   **渐进式预测**：利用轻量级回归模型，结合初始 Prompt 和运行时上下文，在每一步交互后动态更新轨迹长度预测。
    *   **渐进优先级调度（PPS）**：根据预测长度赋予长尾轨迹更高优先级，并引入**抢占机制**，确保长轨迹优先执行，减少累积排队时间。
*   **轨迹感知放置（解决“何处”执行）**：
    *   **预排序动态规划**：基于“最优分区在排序序列中是连续的”这一数学特性，通过 DP 算法将长短轨迹隔离，降低长尾轨迹受到的干扰系数（$\alpha$）。
    *   **机会性迁移**：在工具调用（Tool Call）的空闲间隙，利用 GPU-Direct RDMA 异步迁移 KV Cache，修正因预测偏差导致的负载不均，且不阻塞执行主路径。
*   **轨迹自适应资源管理（解决“如何”配置）**：
    *   **异构并行配置**：打破传统的同构配置，为长尾轨迹分配高模型并行度（MP）以降低延迟，为短轨迹分配低 MP 以保证高吞吐。
    *   **排序初始化模拟退火**：在巨大的搜索空间中快速寻找近乎最优的 GPU 资源分配方案。

#### 6. 主要结论与发现
*   **性能提升**：Heddle 在端到端 Rollout 吞吐量上比现有最先进基准（Verl, Slime）提升了 **1.4x 至 2.5x**。
*   **规模效应**：模型规模越大，计算干扰越严重，Heddle 的优化效果越明显。

*   **开销可控**：预测和迁移的开销基本能被工具执行时间掩盖，对主路径影响极小。

#### 7. 优点与亮点
*   **视角创新**：将 LLM 推理优化从传统的“请求/步骤”粒度提升到了“轨迹”粒度，更符合智能体交互的本质。
*   **系统与算法结合**：利用数学引理简化了复杂的放置问题，并结合模拟退火和动态规划，在工程实现上非常优雅。
*   **资源利用率高**：通过异构并行和机会性迁移，极大地减少了 GPU 在等待长尾任务时的空闲时间。

#### 8. 不足与局限
*   **工具执行时间依赖**：机会性迁移依赖于工具执行（Tool Call）的时间窗口，如果工具执行极快（如微秒级），迁移开销可能无法完全掩盖。

*   **应用限制**：目前主要针对同步 RL 训练场景，虽然文中讨论了异步 RL 的兼容性，但在超大规模异步集群上的表现仍需进一步验证。
