## 摘要
随着大语言模型（LLM）智能体越来越多地部署在软件工程等开放式领域，它们经常会遇到缺乏关键上下文的描述不明确（underspecified）的指令。虽然人类开发人员通常通过提出澄清性问题来解决描述不明确的问题，但目前的智能体在很大程度上是针对自主执行进行优化的。在这项工作中，我们系统地评估了 LLM 智能体在 SWE-bench Verified 的描述不明确变体上的澄清寻求能力。我们提出了一种具有不确定性感知能力的多智能体脚手架（scaffold），它明确地将描述不明确检测与代码执行解耦。我们的结果表明，使用 OpenHands + Claude Sonnet 4.5 的多智能体系统实现了 69.40% 的任务解决率，显著优于标准的单智能体设置（61.20%），并缩小了与在完全明确指令下运行的智能体之间的性能差距。此外，我们发现该多智能体系统表现出良好的不确定性校准，在简单任务上节省查询，而在更复杂的问题上主动寻求信息。这些发现表明，当前的模型可以转变为主动的协作伙伴，智能体能够独立识别何时提出问题，以在现实世界的描述不明确任务中获取缺失的信息。

## Abstract
As Large Language Model (LLM) agents are increasingly deployed in open-ended domains like software engineering, they frequently encounter underspecified instructions that lack crucial context. While human developers naturally resolve underspecification by asking clarifying questions, current agents are largely optimized for autonomous execution. In this work, we systematically evaluate the clarification-seeking abilities of LLM agents on an underspecified variant of SWE-bench Verified. We propose an uncertainty-aware multi-agent scaffold that explicitly decouples underspecification detection from code execution. Our results demonstrate that this multi-agent system using OpenHands + Claude Sonnet 4.5 achieves a 69.40% task resolve rate, significantly outperforming a standard single-agent setup (61.20%) and closing the performance gap with agents operating on fully specified instructions. Furthermore, we find that the multi-agent system exhibits well-calibrated uncertainty, conserving queries on simple tasks while proactively seeking information on more complex issues. These findings indicate that current models can be turned into proactive collaborators, where agents independently recognize when to ask questions to elicit missing information in real-world, underspecified tasks.

### 1. 研究动机与背景
*   **核心问题**：当前的 LLM 编程智能体（如 OpenHands、SWE-agent）大多被优化为“自主执行者”，但在处理现实世界任务（如 GitHub Issues）时，初始指令往往是**描述不明确（Underspecified）**的。
*   **研究动机**：人类开发者在遇到模糊需求时会通过提问来澄清意图，而智能体通常会盲目假设并直接执行，导致任务失败。论文旨在探索如何让智能体具备“不确定性感知”能力，主动识别信息缺失并向用户寻求澄清。

### 2. 方法论
论文提出了一种**不确定性感知多智能体脚手架（Uncertainty-Aware Multi-Agent Scaffold）**，其核心思想是将“任务执行”与“需求检测”解耦：
*   **核心架构（UA-MULTI）**：
    *   **主智能体（Main Agent）**：负责具体的编程任务，拥有文件读写、Bash 脚本执行等工具权限。
    *   **意图智能体（Intent Agent）**：充当“监视器”。在每一轮对话中，它会分析当前的状态历史（包括用户指令、主智能体的动作和观察结果），判断是否存在歧义或关键信息缺失。
*   **关键流程**：
    1.  如果意图智能体检测到不确定性，它会强制主智能体的下一个动作必须是“询问用户（clarify）”。
    2.  如果意图智能体认为信息充足，主智能体则继续执行代码逻辑。
*   **对比架构（UA-SINGLE）**：作为对比，研究者还测试了单智能体模式，即在同一个模型中通过 Prompt 提醒其在每一步自发检查不确定性。

### 3. 实验设计
*   **数据集**：使用 **SWE-bench Verified** 的变体。
    *   **原始数据**：包含 500 个经过人工验证的真实 GitHub Issues。
    *   **欠规范化处理**：利用 GPT-4o 将原始 Issue 总结为简短、缺失关键细节（如堆栈跟踪、具体文件路径）的版本。
*   **实验场景与 Benchmark**：
    *   **FULL**：提供完整 Issue 信息（基准上限）。
    *   **HIDDEN**：仅提供模糊信息，禁止提问。
    *   **INTERACTIVE BASELINE**：提供模糊信息，但硬编码要求智能体在开始前必须提问一次。
    *   **UA-SINGLE/UA-MULTI**：论文提出的不确定性感知方案。
*   **用户模拟器**：使用 **GPT-5.1** 扮演用户，它持有完整的 Issue 信息，仅在智能体提问时提供对应答案。

### 4. 资源与算力
*   **模型后端**：所有实验均使用 **Claude Sonnet 4.5** 作为智能体骨干模型。

*   **迭代限制**：每个任务最多允许 100 次迭代。

### 5. 实验数量与充分性
*   **实验规模**：在全部 500 个 SWE-bench Verified 任务上进行了完整测试。
*   **充分性**：

### 6. 主要结论与发现
*   **性能提升**：UA-MULTI 实现了 **69.40%** 的解决率，显著优于 UA-SINGLE（61.20%），并几乎追平了拥有完整信息的 FULL 基准（70.80%）。
*   **不确定性校准**：多智能体系统表现出良好的校准性。在简单任务上它会减少提问，而在复杂任务上提问率显著升高。
*   **提问策略**：UA-MULTI 倾向于进行**多次、简短且针对性强**的迭代式提问（平均每任务 3.06 次），而单智能体往往在任务后期才意识到需要提问，此时往往为时已晚。
*   **解耦的必要性**：实验证明，即使是顶尖模型（如 Sonnet 4.5），在同时处理代码执行和自我监控时也会表现不佳，必须通过架构解耦来释放其潜力。

### 7. 优点
*   **架构创新**：通过简单的多智能体协作逻辑，解决了 LLM 容易“过度自信”并盲目假设的顽疾。
*   **现实意义**：针对软件工程中常见的“需求模糊”痛点，提供了可落地的智能体协作框架。
*   **深入分析**：不仅关注解决率，还深入探讨了智能体提问的时机、频率与任务难度的相关性。

### 8. 不足与局限
*   **用户模拟器偏差**：实验依赖 GPT-5.1 模拟用户，可能无法完全反映真实人类用户的多样性、情绪或不配合的情况。
*   **推理成本高昂**：多智能体架构导致 Token 消耗翻倍，对于大规模部署可能存在经济压力。
*   **模型泛化性**：实验仅在 Claude Sonnet 4.5 上进行，尚不清楚较弱的开源模型（如 Llama-3）是否能通过该架构获得类似的校准能力。
*   **环境依赖**：实验环境（OpenHands）与测试床（/testbed）的目录差异有时会误导智能体，虽然通过 Guardrails 进行了修正，但仍显示出系统复杂性带来的脆弱性。
