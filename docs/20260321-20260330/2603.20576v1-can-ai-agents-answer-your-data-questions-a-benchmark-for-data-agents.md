## 摘要
各行各业的企业用户越来越依赖AI智能体通过自然语言查询其数据。然而，构建可靠的数据智能体仍然具有挑战性，因为现实世界的数据通常分散在多个异构数据库系统中，存在不一致的引用，且信息隐藏在非结构化文本中。现有的基准测试仅解决了该问题的部分环节——例如，将自然语言问题转换为SQL查询，或针对上下文中提供的小型表格回答问题——但并未评估跨多个数据库系统进行数据集成、转换和分析的完整流程。为了填补这一空白，我们提出了数据智能体基准测试（Data Agent Benchmark, DAB），该基准基于对六个行业企业数据智能体工作负载的形成性研究。DAB包含涉及12个数据集、9个领域和4个数据库管理系统的54个查询。在DAB上，表现最好的前沿模型（Gemini-3-Pro）仅达到了38%的pass@1准确率。我们对五个前沿大语言模型（LLM）进行了基准测试，分析了它们的失败模式，并为未来的数据智能体开发总结了核心启示。我们的基准测试和实验代码已发布在 github.com/ucbepic/DataAgentBench。

## Abstract
Users across enterprises increasingly rely on AI agents to query their data through natural language. However, building reliable data agents remains difficult because real-world data is often fragmented across multiple heterogeneous database systems, with inconsistent references and information buried in unstructured text. Existing benchmarks only tackle individual pieces of this problem -- e.g., translating natural-language questions into SQL queries, answering questions over small tables provided in context -- but do not evaluate the full pipeline of integrating, transforming, and analyzing data across multiple database systems. To fill this gap, we present the Data Agent Benchmark (DAB), grounded in a formative study of enterprise data agent workloads across six industries. DAB comprises 54 queries across 12 datasets, 9 domains, and 4 database management systems. On DAB, the best frontier model (Gemini-3-Pro) achieves only 38% pass@1 accuracy. We benchmark five frontier LLMs, analyze their failure modes, and distill takeaways for future data agent development. Our benchmark and experiment code are published at github.com/ucbepic/DataAgentBench.

*   **核心问题**：现有的基准测试（如Text-to-SQL或Table-QA）过于局限，仅测试单一数据库的查询生成或小规模上下文推理，无法评估智能体在**集成、转换和分析跨异构系统数据**时的端到端能力。
*   **整体含义**：本文提出了首个针对真实、复杂数据任务的基准测试 **DAB (Data Agent Benchmark)**，旨在系统性地识别数据智能体的失效模式，推动其在企业级环境中的应用。

### 2. 方法论
*   **形成性研究（Formative Study）**：通过对6个行业（科技、金融、医疗等）的企业级工作负载进行调研，总结出真实数据查询的四大挑战：
    1.  **多数据库集成**：需跨SQL（PostgreSQL, SQLite等）和NoSQL（MongoDB）系统操作。
    2.  **格式错误的连接键**：同一实体的ID在不同系统中格式不同（如前缀差异、空格干扰）。
    3.  **非结构化文本转换**：关键属性嵌入在评论或描述文本中，需提取后才能进行过滤或聚合。
    4.  **领域知识**：需要特定行业的逻辑（如金融波动率计算公式）。
*   **DAB构建流程**：
    *   **数据扰动**：选取12个开源数据集，手动移除直观列，将其值“重嵌入”到自由文本中，并人为制造连接键的不一致性。
    *   **异构部署**：将数据分布在PostgreSQL、MongoDB、DuckDB和SQLite中。
    *   **确定性评估**：从原始干净数据中推导标准答案，验证脚本通过子字符串匹配等方式进行自动化、确定性的评分。

### 3. 实验设计
*   **Benchmark构成**：包含12个数据集、54个自然语言查询，涵盖9个领域。
*   **对比方法**：评估了5个前沿大模型：**GPT-5.2、GPT-5-mini、Gemini-3-Pro、Gemini-2.5-Flash、Kimi-K2**。
*   **智能体架构**：采用 **ReAct** 模式（推理+行动循环），为智能体提供四种工具：`list_db`（查看表）、`query_db`（执行查询）、`execute_python`（数据处理）、`return_answer`（提交答案）。
*   **评估指标**：使用 **pass@k**（k=1到50），即在k次尝试中至少成功一次的概率。

### 4. 资源与算力
*   **实验成本**：总计进行了13,500次试验（50次独立重复/查询/模型），API调用总成本约为 **3,150美元**。
*   **算力细节**：论文未提及模型训练算力（属于闭源模型推理评估），但提到Python代码执行是在配置了Python 3.12、Pandas和Pyarrow的 **Docker 容器环境**中运行，单个工具调用超时限制为600秒。

### 5. 实验数量与充分性
*   **实验规模**：54个查询虽然在数量上少于某些Text-to-SQL榜单，但每个查询都涉及多步推理和跨库操作，复杂度极高。
*   **充分性**：每个查询重复50次实验，极大地降低了LLM随机性对结果的影响，提供了稳定的pass@k曲线。
*   **客观性**：通过GPT-5作为裁判对1,147个失败轨迹进行了详细的**失效模式分类（Failure Taxonomy）**，分析过程严谨且具有统计意义。

### 6. 主要结论与发现
*   **性能低迷**：最强模型 Gemini-3-Pro 的 **pass@1 仅为 38%**，即使尝试50次，最高准确率也不超过69%。有一个数据集（patents）所有模型均未攻克。
*   **失效模式**：**85%的错误源于“规划错误”或“实现错误”**，而“数据源选择错误”仅占15%。这说明模型能找到数据，但不知道怎么算或算不对。
*   **工具局限**：所有智能体在处理文本提取时都过度依赖正则表达式（Regex），面对复杂的自然语言日期或格式时表现极差。
*   **探索策略**：表现较好的模型通常花费约20%的步骤进行数据探索（如查看表结构和采样）。

### 7. 优点
*   **真实性强**：基于真实企业调研而非学术假设，捕捉到了数据“脏、乱、散”的本质特征。
*   **异构性突破**：首次在基准测试中强制要求智能体同时处理SQL和NoSQL方言。
*   **失效分析深入**：不仅给出了分数，还通过分类学解释了智能体在哪个环节“掉链子”，对开发者极具指导价值。

### 8. 不足与局限
*   **模型覆盖**：由于API额度限制，未包含Anthropic的Claude系列模型。
*   **验证偏差**：验证脚本采用子字符串匹配，可能存在“召回率高但精确率低”的偏见（即智能体返回一堆错误答案中夹杂一个正确答案也会被判对）。
*   **静态环境**：为了保证确定性，去除了实时API集成测试，无法评估智能体处理动态外部数据的能力。
