## 摘要
AI 智能体越来越多地通过模型上下文协议（MCP）调用工具，并通过智能体对智能体（A2A）模式向其他智能体进行委托，然而这两种协议均未验证智能体身份。对约 2,000 个 MCP 服务器的扫描发现，所有服务器都缺乏身份验证。在我们的调研中，尚未发现任何已实现的协议能同时结合公钥可验证委托、持有者端衰减、表达性链式策略、跨 MCP/A2A/HTTP 的传输绑定以及面向溯源的完成记录。我们引入了调用绑定能力令牌（IBCTs），这是一种将身份、衰减授权和溯源绑定融合到单个追加型令牌链中的原语。IBCTs 以两种传输格式运行：紧凑模式（用于单跳场景的签名 JWT）和链式模式（带有 Datalog 策略的 Biscuit 令牌，用于多跳委托）。我们提供了 Python 和 Rust 的参考实现，并具有完全的跨语言互操作性。紧凑模式的验证时间分别为 0.049 毫秒（Rust）和 0.189 毫秒（Python），在实际的 MCP-over-HTTP 部署中，相比无认证模式仅增加 0.22 毫秒的开销。在 Gemini 2.5 Flash 的真实多智能体部署中，AIP 增加了 2.35 毫秒的开销（占端到端总延迟的 0.086%）。针对 600 次攻击尝试的对抗性评估显示其拒绝率为 100%，其中两类攻击（委托深度违规和通过空上下文规避审计）被 AIP 的链式委托模型唯一捕获，而无签名或普通 JWT 部署均无法检测到这些攻击。

## Abstract
AI agents increasingly call tools via the Model Context Protocol (MCP) and delegate to other agents via Agent-to-Agent (A2A), yet neither protocol verifies agent identity. A scan of approximately 2,000 MCP servers found all lacked authentication. In our survey, we did not identify a prior implemented protocol that jointly combines public-key verifiable delegation, holder-side attenuation, expressive chained policy, transport bindings across MCP/A2A/HTTP, and provenance-oriented completion records. We introduce Invocation-Bound Capability Tokens (IBCTs), a primitive that fuses identity, attenuated authorization, and provenance binding into a single append-only token chain. IBCTs operate in two wire formats: compact mode (a signed JWT for single-hop cases) and chained mode (a Biscuit token with Datalog policies for multi-hop delegation). We provide reference implementations in Python and Rust with full cross-language interoperability. Compact mode verification takes 0.049ms (Rust) and 0.189ms (Python), with 0.22ms overhead over no-auth in real MCP-over-HTTP deployment. In a real multi-agent deployment with Gemini 2.5 Flash, AIP adds 2.35ms of overhead (0.086% of total end-to-end latency). Adversarial evaluation across 600 attack attempts shows 100% rejection rate, with two attack categories (delegation depth violation and audit evasion through empty context) uniquely caught by AIP's chained delegation model that neither unsigned nor plain JWT deployments detect.

**研究动机：** 现有的身份验证方案（如 OAuth 2.1、DIDs、Macaroons 等）无法同时满足智能体所需的七项核心属性：公钥验证、持有者端权限衰减、表达性策略、跨协议绑定、溯源感知、无重型基础设施依赖以及生命周期感知。本论文旨在填补这一空白，提出一种统一的身份与授权协议。

#### 2. 论文提出的方法论
论文引入了**调用绑定能力令牌（IBCT, Invocation-Bound Capability Tokens）**，其核心思想是将身份、可衰减的授权和执行溯源融合到一个追加型的令牌链中。

*   **核心技术细节：**
    *   **两种运行模式：**
        *   **紧凑模式（Compact Mode）：** 使用 EdDSA 签名的 JWT，适用于单跳 MCP 调用，便于现有库集成。
        *   **链式模式（Chained Mode）：** 基于 Biscuit 令牌格式，利用 Datalog 策略语言实现多跳委托。每一跳都会追加一个签名块，且权限只能缩小（衰减）不能扩大。
    *   **身份方案：** 支持基于 DNS 的 `aip:web`（适用于长期智能体）和自证明的 `aip:key`（适用于临时生成的子智能体）。
    *   **三类数据块：** 权限块（Block 0，定义初始权限）、委托块（Block N，记录委托路径与约束）、完成块（Block N+1，记录执行结果哈希与成本）。
    *   **策略配置文件：** 提供简单（Simple）、标准（Standard）和高级（Advanced）三级 Datalog 策略，平衡了易用性与表达力。

*   **极低开销：** Rust 环境下紧凑模式验证仅需 0.049ms；在多智能体任务中，AIP 带来的总延迟约为 2.35ms，远小于 LLM 推理时间（约 2747ms）。

#### 7. 优点
*   **协议融合：** 首次将身份、授权和溯源（Completion Blocks）整合进单一令牌，解决了智能体协作中的信任链条问题。
*   **高性能与互操作性：** 提供了高性能的 Rust 实现和易用的 Python SDK，且两者完全兼容。
*   **离线衰减：** 允许持有者在不联系授权服务器的情况下，线下缩小权限并向下委托，非常适合分布式智能体环境。

#### 8. 不足与局限
*   **部署环境局限：** 实验主要在单机 localhost 环境下完成，缺乏跨数据中心、高并发生产环境下的网络抖动和延迟数据。
*   **自报告风险：** “完成块”目前主要由执行智能体自签名报告，虽然实现了不可篡改，但无法防止智能体恶意撒谎（除非引入第三方审计）。
*   **撤销机制缺失：** AIP v1 主要依赖短寿命令牌，尚未建立完善的实时撤销（CRL/OCSP）基础设施。
*   **DNS 依赖：** `aip:web` 模式继承了 DNS 系统的安全风险（如劫持），尽管有文档自签名保护，但仍存在信任根风险。
