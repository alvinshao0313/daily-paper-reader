## 摘要
尽管大语言模型（LLMs）发展迅速，但复杂软件工作流的鲁棒自动化仍是一个悬而未决的问题。在长程（long-horizon）场景中，智能体经常受到级联错误和环境随机性的困扰；动态界面中的一次失误就可能导致任务失败，进而引发幻觉或陷入反复试错。本文介绍了“环境地图”（Environment Maps）：一种持久的、与智能体无关的表示方式，通过将屏幕录像和执行轨迹等异构证据整合到结构化图中，来缓解这些失败。该表示由四个核心组件组成：(1) 上下文（抽象位置），(2) 动作（参数化可供性），(3) 工作流（观测到的轨迹），以及 (4) 隐性知识（领域定义和可重用程序）。我们在 WebArena 基准测试的五个领域中评估了该框架。配备环境地图的智能体实现了 28.2% 的成功率，几乎是受限于会话上下文的基线模型（14.2%）的两倍，且优于能够访问用于生成环境地图的原始轨迹数据的智能体（23.3%）。通过在模型和环境之间提供结构化接口，环境地图为长程规划奠定了持久的基础，且具有可解释性、可编辑性和增量完善性。

## Abstract
Although large language models (LLMs) have advanced rapidly, robust automation of complex software workflows remains an open problem. In long-horizon settings, agents frequently suffer from cascading errors and environmental stochasticity; a single misstep in a dynamic interface can lead to task failure, resulting in hallucinations or trial-and-error. This paper introduces $\textit{Environment Maps}$: a persistent, agent-agnostic representation that mitigates these failures by consolidating heterogeneous evidence, such as screen recordings and execution traces, into a structured graph. The representation consists of four core components: (1) Contexts (abstracted locations), (2) Actions (parameterized affordances), (3) Workflows (observed trajectories), and (4) Tacit Knowledge (domain definitions and reusable procedures). We evaluate this framework on the WebArena benchmark across five domains. Agents equipped with environment maps achieve a 28.2% success rate, nearly doubling the performance of baselines limited to session-bound context (14.2%) and outperforming agents that have access to the raw trajectory data used to generate the environment maps (23.3%). By providing a structured interface between the model and the environment, Environment Maps establish a persistent foundation for long-horizon planning that is human-interpretable, editable, and incrementally refinable.

*   **核心问题**：尽管大语言模型（LLM）在短期任务中表现出色，但在处理复杂软件工作流的长程（Long-horizon）任务时，常因**级联错误**、**环境随机性**（如UI变动）以及**缺乏持久的环境理解**而失败。
*   **研究动机**：现有的方法（如UI转换图、预测模型或技能库）往往彼此孤立，且通常与特定智能体绑定或非持久化。作者旨在创建一种**与智能体无关、持久且结构化**的环境表示，将零散的交互经验转化为可查询、可编辑的知识库，以支持智能体进行更稳健的规划。

#### 2. 方法论：环境地图框架
环境地图被定义为一个四元组 $M = (C, A, W, K)$，其核心思想是通过五阶段流水线将原始观测（如屏幕录像、DOM树）转化为结构化图：
*   **核心组件**：
    1.  **上下文 (Contexts, C)**：基于URL模式识别的抽象页面或视图（如“搜索页”、“设置页”）。
    2.  **动作 (Actions, A)**：参数化的可供性（Affordances），例如将“点击张三”抽象为“点击 {用户名}”，并记录观察到的取值范围。
    3.  **工作流 (Workflows, W)**：从成功轨迹中提取的可重用步骤序列。
    4.  **隐性知识 (Tacit Knowledge, K)**：包括领域定义（如缩写含义）和操作规程（如过滤问题的具体步骤）。
*   **构建流水线**：
    *   **阶段1-2**：解析原始事件流，提取已执行动作和界面中潜在的未执行动作。
    *   **阶段3-4**：利用LLM进行动作参数化泛化，并根据URL模式聚类生成上下文和隐性知识。
    *   **阶段5**：合并重复项，建立跨实体的可追溯链接，生成最终的JSON格式地图。

#### 3. 实验设计
*   **基准测试**：使用 **WebArena**，涵盖 812 个自然语言任务。
*   **实验场景**：包括 5 个交互式 Web 环境：电子商务（Magento）、内容管理系统（CMS）、代码托管（GitLab）、论坛（Reddit）和地图（OpenStreetMap）。
*   **对比方法**：
    1.  **Baseline (No Map)**：仅提供当前页面观测（可访问性树）和任务目标。
    2.  **Trajectory Access**：允许智能体访问原始的人类操作轨迹文件（JSONL格式），但无结构化地图。
    3.  **Environment Map Access**：允许智能体访问基于相同轨迹数据生成的结构化环境地图。
*   **智能体配置**：基于 `claude-3.5-sonnet` (claude-sonnet-4-5)，配备文件读取、搜索和 Bash 工具。

#### 4. 资源与算力
*   **算力消耗**：论文**未提及**具体的 GPU 训练算力，因为该方法基于现成的 LLM API，无需对模型进行策略微调（Policy Retraining）。
*   **构建成本**：作者强调了该方法的经济性。构建一个环境地图的成本约为 **1 到 4 美元**，耗时约 **13 到 31 分钟**。相比之下，微调中型 LLM 通常需要数十到数百个 GPU 小时。

#### 5. 实验数量与充分性
*   **实验规模**：在 812 个任务上进行了完整测试，涵盖了不同的意图模板和难度等级。
*   **充分性**：实验设计较为全面，不仅对比了成功率，还分析了**泛化能力**（区分有无演示轨迹的任务）、**工具调用模式**以及通过 HAR 追踪分析的**导航效率**（回溯率）。
*   **客观性**：使用了 WebArena Verified 评估框架，通过 HTTP 存档（HAR）检查确保评估的确定性和可重复性，避免了传统评估中常见的幻觉误判。

#### 6. 主要结论与发现
*   **性能显著提升**：配备环境地图的智能体成功率达到 **28.2%**，远高于基线（14.2%）和原始轨迹访问组（23.3%）。
*   **结构化优于原始数据**：即使底层数据相同，将数据组织成“地图”比直接提供原始轨迹更能提升智能体表现，尤其是在 GitLab 和 CMS 等高分支、UI 复杂的环境中。
*   **泛化能力强**：在没有直接演示轨迹的任务上，环境地图依然能提供 26.9% 的成功率（基线仅 12.6%），证明了其提供的页面上下文和动作模式具有通用价值。
*   **行为改变**：使用地图的智能体更倾向于通过文件工具主动查询环境知识（平均每任务 12.1 次调用），而基线组几乎不使用此类工具。

#### 7. 优点
*   **智能体无关性**：地图作为外部知识库存在，模型升级时无需重新训练，可直接复用。
*   **可解释与可编辑**：地图以人类可读的 JSON 格式存储，人类专家可以轻松诊断错误、修改动作定义或更新过时的信息。
*   **高效性**：相比于强化学习或微调，构建地图的成本极低，且能显著降低智能体在执行时的盲目探索（回溯率降低）。

#### 8. 不足与局限
*   **环境局限性**：目前的地图主要针对单一站点，尚未解决跨站点工作流和共享实体抽象的问题。
*   **状态验证不足**：虽然记录了动作的前置条件和效果，但目前多为描述性文字，尚未实现基于实时状态观测的自动化验证。
*   **时效性挑战**：Web 环境演进较快，地图可能会过时。虽然作者提到可以增量更新，但尚未提出自动检测和修复结构错误的成熟机制。
*   **简单环境收益有限**：在 Reddit 等导航路径较浅的环境中，结构化地图相比原始轨迹的优势并不明显。
