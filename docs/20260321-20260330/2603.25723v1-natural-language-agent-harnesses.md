## 摘要
智能体的性能日益依赖于测试框架工程（harness engineering），然而测试框架的设计通常隐藏在控制器代码和特定于运行时的约定中，这使得它难以作为科学对象进行迁移、比较和研究。我们探究智能体测试框架的高层控制逻辑是否可以外置为一种可移植的可执行构件。我们引入了自然语言智能体测试框架（NLAHs），它以可编辑的自然语言表达测试框架的行为；以及智能测试框架运行时（IHR），这是一个通过显式契约、持久化构件和轻量级适配器来执行这些测试框架的共享运行时。在编程和计算机使用基准测试中，我们对运行可行性、模块消融以及从代码到文本的测试框架迁移进行了受控评估。

## Abstract
Agent performance increasingly depends on \emph{harness engineering}, yet harness design is usually buried in controller code and runtime-specific conventions, making it hard to transfer, compare, and study as a scientific object. We ask whether the high-level control logic of an agent harness can instead be externalized as a portable executable artifact. We introduce \textbf{Natural-Language Agent Harnesses} (NLAHs), which express harness behavior in editable natural language, and \textbf{Intelligent Harness Runtime} (IHR), a shared runtime that executes these harnesses through explicit contracts, durable artifacts, and lightweight adapters. Across coding and computer-use benchmarks, we conduct controlled evaluations of operational viability, module ablation, and code-to-text harness migration.

#### 1. 论文的核心问题与整体含义（研究动机和背景）
现代智能体的成功不仅取决于基础模型，更取决于其外围的**测试框架（Harness）**——即负责多步推理、工具调用、内存管理、任务委派和停止准则的控制层。然而，目前的测试框架逻辑通常深埋在控制器代码、框架默认设置或特定运行时的约定中，导致其难以迁移、难以公平比较，也难以作为独立的科学对象进行消融研究。论文提出的核心问题是：**能否将智能体测试框架的高层控制逻辑外部化为一种可移植、可执行的自然语言构件？**

#### 2. 论文提出的方法论
论文提出了两项核心技术：
*   **自然语言智能体测试框架 (NLAHs)：** 一种结构化的自然语言表示，用于表达测试框架的控制逻辑。它包含以下核心组件：
    *   **契约 (Contracts)：** 规定输入输出要求、格式约束、验证门控和停止规则。
    *   **角色 (Roles)：** 定义具有非重叠职责的角色（如规划者、执行者、验证者）。
    *   **阶段结构 (Stage Structure)：** 明确的工作流拓扑（如 规划 -> 执行 -> 验证 -> 修复）。
    *   **适配器与脚本 (Adapters & Scripts)：** 用于执行确定性操作（如测试、解析）的钩子。
    *   **状态语义 (State Semantics)：** 规定跨步骤持久化的构件和路径寻址方式。
    *   **失败分类 (Failure Taxonomy)：** 定义驱动恢复机制的命名失败模式。
*   **智能测试框架运行时 (IHR)：** 一个共享的运行时环境，直接解释并执行 NLAHs。
    *   它在循环中引入一个 LLM 作为解释器，根据 NLAH 逻辑、当前环境状态和“运行时章程”（Runtime Charter）决定下一步行动。
    *   它将模型调用提升为受契约约束的“智能体调用”，支持多智能体协作和文件支持的状态管理。

#### 6. 论文的主要结论与发现
*   **行为改变显著：** 引入 NLAH 和 IHR 显著改变了智能体的行为模式（如 Token 消耗和调用次数大幅增加），而不仅仅是简单的提示词包装。
*   **结构并非越多越好：** 增加复杂的控制结构（如多候选搜索）并不总是能提高成功率，有时反而会因为与评估标准的对齐偏差而导致性能下降。
*   **模块化可行：** 证明了测试框架逻辑可以像软件模块一样进行组合和消融，其中“自我进化”和“文件支持状态”模块表现出较好的增益。
*   **代码到文本迁移有效：** 在 OSWorld 中，NLAH 版本甚至优于原生代码版本，因为它将可靠性机制从脆弱的界面修复转向了持久化的运行时状态和构件验证。

#### 7. 优点
*   **解耦设计：** 成功将高层设计模式与底层执行细节分离，使测试框架成为可研究的独立对象。
*   **可移植性：** 自然语言形式使得复杂的智能体工作流更容易在不同系统间迁移和复用。
*   **高可观察性：** 通过 IHR 提供的显式契约和文件状态，使得智能体的决策过程和中间产物更易于审计。

#### 8. 不足与局限
*   **精度损失：** 自然语言在表达某些极度精确的控制逻辑时可能不如代码严谨。
*   **运行时污染：** 强大的运行时章程可能会吸收部分本应属于测试框架文本的行为，导致责任边界模糊。
*   **成本开销：** 引入 IHR 解释层和多步验证显著增加了推理成本（Token 消耗和延迟）。
*   **样本规模：** 实验仅在子集上运行，完整基准测试的结果尚待验证。
