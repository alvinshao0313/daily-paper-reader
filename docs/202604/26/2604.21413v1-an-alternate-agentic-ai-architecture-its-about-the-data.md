## 摘要
在过去几年中，“智能体 AI”（agentic AI）的主流叙事是大语言模型应通过动态选择工具、发布子查询并综合结果来编排信息访问。我们认为这种方法存在误导：企业面临的并非推理能力不足，而是数据集成问题。企业是以数据为中心的：关键信息分散在异构系统（如数据库、文档和外部服务）中，每个系统都有其特定的查询语言、模式（schema）、访问控制和性能约束。相比之下，当代基于 LLM 的架构针对非结构化文本的推理进行了优化，并将企业系统视为语料库或由黑盒组件调用的外部工具。这导致了富模式、受控且对性能要求极高的数据系统与以文本为中心、概率性的 LLM 架构之间的不匹配，从而导致透明度有限、正确性保证薄弱以及性能不可预测。在本文中，我们提出了 RUBICON，这是一种基于数据管理原则的替代架构。我们没有将编排工作委托给不透明的智能体，而是引入了 AQL（智能体查询语言），这是一种小型的、显式的查询代数——Find、From 和 Where——通过特定源的封装器执行，这些封装器负责强制执行访问控制、模式对齐和结果归一化。所有中间结果均可见且可检查。复杂问题被分解为结构化、可审计的查询计划，而非隐藏的 LLM 调用链。我们的论点很简单：企业 AI 不是提示工程问题，而是一个系统问题。通过重新引入显式查询结构、基于封装器的中介以及基于成本的优化，我们在获得智能体搜索广度的同时，保留了企业环境中的可追溯性、确定性和信任。

## Abstract
For the last several years, the dominant narrative in "agentic AI" has been that large language models should orchestrate information access by dynamically selecting tools, issuing sub-queries, and synthesizing results. We argue this approach is misguided: enterprises do not suffer from a reasoning deficit, but from a data integration problem.   Enterprises are data-centric: critical information is scattered across heterogeneous systems (e.g., databases, documents, and external services), each with its own query language, schema, access controls, and performance constraints. In contrast, contemporary LLM-based architectures are optimized for reasoning over unstructured text and treat enterprise systems as either corpora or external tools invoked by a black-box component. This creates a mismatch between schema-rich, governed, performance-critical data systems and text-centric, probabilistic LLM architectures, leading to limited transparency, weak correctness guarantees, and unpredictable performance.   In this paper, we present RUBICON, an alternative architecture grounded in data management principles. Instead of delegating orchestration to an opaque agent, we introduce AQL (Agentic Query Language), a small, explicit query algebra - Find, From, and Where - executed through source-specific wrappers that enforce access control, schema alignment, and result normalization. All intermediate results are visible and inspectable. Complex questions are decomposed into structured, auditable query plans rather than hidden chains of LLM calls.   Our thesis is simple: enterprise AI is not a prompt engineering problem; it is a systems problem. By reintroducing explicit query structure, wrapper-based mediation, and cost-based optimization, we obtain the breadth of agentic search while preserving traceability, determinism, and trust in enterprise environments.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心矛盾**：当前“以 LLM 为中心”的智能体架构（如 ReAct、AutoGPT）将 LLM 作为核心编排器，通过动态选择工具和推理来解决问题。作者认为这在企业环境下是误导性的。
*   **企业现状**：企业数据具有**异构性**（分散在 ERP、CRM、数据库中）、**模式丰富性**（Schema-rich）、**严格的访问控制**和**高性能要求**。
*   **现有方案缺陷**：
    *   **Text-to-SQL 失效**：在处理包含特定术语、冗余模式和复杂逻辑的企业私有数据时，Text-to-SQL 的准确率会从基准测试的 85% 骤降至 50% 以下。
    *   **黑盒编排**：LLM 驱动的编排过程不透明、不可重复，且容易产生幻觉或遗漏关键数据源。
    *   **数据丢失**：将结构化数据转化为文本输入 LLM 会丢失原始的结构信息和约束。

### 2. 方法论：RUBICON 架构与 AQL
RUBICON 放弃了让 LLM 自由发挥的思路，转而采用“以数据为中心”的系统工程方法：
*   **核心思想**：将检索视为虚拟数据集成问题。数据保留在原位（In-situ），通过封装器（Wrappers）将其抽象为关系视图。
*   **关键技术细节**：
    *   **封装器（Wrappers）**：为每个数据源（如 SQL 数据库、邮件系统、视频流）编写特定的转换器，将原生 API 映射为统一的关系表（行和列）。
    *   **AQL（Agentic Query Language）**：一种简化的 SQL 子集，作为用户与封装器之间的通用语言。其核心语法为：`FIND <列> FROM <表> WHERE <自然语言谓词>`。它支持聚合（Aggregates）和连接（Joins）。
    *   **执行模式**：
        *   **交互模式**：用户逐条执行 AQL，中间结果以显式表格形式呈现，可检查、可追溯。
        *   **编译模式**：将一系列 AQL 指令转化为复合执行计划，利用关系数据库的**查询优化器**（如调整 Join 顺序）来降低成本和延迟。

### 3. 实验设计
*   **场景与数据集**：构建了一个模拟企业环境的基准测试，包含 5 个异构数据源：维基百科、大学数据仓库（97 张表）、实验室网站、Gmail 邮件系统、LLM 内部知识。
*   **Benchmark 设计**：设计了 7 个专家级查询（Q1-Q7），每个查询都必须集成且仅集成两个特定的数据源才能完成。
*   **对比方法**：
    *   **Vanilla LLM**：直接使用自然语言（NL）或 AQL 提示词。
    *   **LangChain ReAct Agent**：具备工具调用能力的智能体。
    *   **底层模型**：GPT-5-mini、Gemini-3-flash-preview、Claude-Sonnet-4.6。

### 4. 资源与算力
*   论文主要基于商业模型的 API 进行推理实验（GPT-5-mini 等）。
*   **未明确说明**具体的本地 GPU 算力消耗，因为该研究侧重于推理架构而非模型训练。实验记录了 Token 消耗、API 报告的货币成本和延迟（TTFT）。

### 5. 实验数量与充分性
*   **实验规模**：针对 7 个复杂查询，在 3 种模型、2 种提示风格（NL/AQL）、2 种架构（Vanilla/ReAct）下进行了多组对比。
*   **充分性评价**：虽然查询数量较少（7 个），但每个查询都经过精心设计以测试“多源协调能力”。作者指出，这种小规模但全披露的测试能更清晰地诊断协调失败的原因。此外，文中提到正在进行一项包含 100 个真实企业查询的大规模测试。

### 6. 主要结论与发现
*   **准确率鸿沟**：所有以 LLM 为中心的架构（Vanilla 和 ReAct）在所有模型下的**准确率均为 0%**。它们要么遗漏数据源，要么无法正确完成跨源 Join。
*   **RUBICON 表现**：通过显式的 AQL 计划，RUBICON 达到了 **100% 的准确率**。
*   **成本与延迟**：ReAct 智能体由于反复的推理-行动循环，导致 Token 消耗呈指数级增长（Gemini-AQL 模式下平均输入 Token 超过 46 万），而 RUBICON 的 Token 消耗和成本保持在极低且确定的水平。
*   **协调是瓶颈**：失败的主因不是语言理解能力不足，而是**协调失效**（无法确保数据源的完整覆盖和确定性集成）。

### 7. 优点
*   **透明度与可审计性**：中间结果是可见的表格，而非隐藏的对话状态，符合企业合规要求。
*   **确定性**：通过结构化查询消除了 LLM 编排的随机性。
*   **性能优化**：引入了数据库领域的查询优化思想（如 Plan A vs Plan B 的成本差异分析），能显著降低大规模检索的成本。
*   **安全性**：封装器直接继承底层系统的访问控制逻辑。

### 8. 不足与局限
*   **基准测试规模**：目前公开的实验仅 7 个查询，样本量较小，可能无法覆盖所有边缘情况。
*   **人工成本**：需要企业人员为私有数据源手动编写封装器（Wrapper），这增加了部署初期的工作量。
*   **谓词依赖**：AQL 中的 `WHERE` 子句仍依赖 LLM 将自然语言转化为原生查询，若谓词逻辑极其复杂，仍可能存在转化风险。
*   **优化器实现**：论文中对优化器的讨论偏向理论，尚未展示自动代价估算（Cost-based Optimization）的完整自动化实现。
