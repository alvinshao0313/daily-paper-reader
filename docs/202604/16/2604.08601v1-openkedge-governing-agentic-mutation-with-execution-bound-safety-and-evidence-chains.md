## 摘要
自主 AI 智能体的兴起暴露了以 API 为中心的架构中的一个根本缺陷：概率性系统在缺乏足够上下文、协调或安全保证的情况下直接执行状态变更。我们提出了 OpenKedge，这是一种将状态变更重新定义为受控过程而非 API 调用直接结果的协议。OpenKedge 要求执行者提交声明式意图提案，并在执行前根据确定性推导出的系统状态、时间信号和策略约束对其进行评估。获批的意图被编译为执行合约，严格限制允许的操作、资源范围和时间，并通过临时的、面向任务的身份进行强制执行。这将安全性从反应式过滤转变为预防性的、受执行限制的强制执行。至关重要的是，OpenKedge 引入了“意图到执行证据链”（IEEC），通过密码学方式将意图、上下文、策略决策、执行边界和结果链接到一个统一的谱系中。这使状态变更转变为一个可验证且可重构的过程，从而实现了确定性的可审计性以及对系统行为的推理。我们在多智能体冲突场景和云基础设施变更中对 OpenKedge 进行了评估。结果表明，该协议在保持高吞吐量的同时，能够确定性地仲裁竞争意图并限制不安全执行，为大规模安全运行智能体系统奠定了原则性基础。

## Abstract
The rise of autonomous AI agents exposes a fundamental flaw in API-centric architectures: probabilistic systems directly execute state mutations without sufficient context, coordination, or safety guarantees. We introduce OpenKedge, a protocol that redefines mutation as a governed process rather than an immediate consequence of API invocation. OpenKedge requires actors to submit declarative intent proposals, which are evaluated against deterministically derived system state, temporal signals, and policy constraints prior to execution. Approved intents are compiled into execution contracts that strictly bound permitted actions, resource scope, and time, and are enforced via ephemeral, task-oriented identities. This shifts safety from reactive filtering to preventative, execution-bound enforcement. Crucially, OpenKedge introduces an Intent-to-Execution Evidence Chain (IEEC), which cryptographically links intent, context, policy decisions, execution bounds, and outcomes into a unified lineage. This transforms mutation into a verifiable and reconstructable process, enabling deterministic auditability and reasoning about system behavior. We evaluate OpenKedge across multi-agent conflict scenarios and cloud infrastructure mutations. Results show that the protocol deterministically arbitrates competing intents and cages unsafe execution while maintaining high throughput, establishing a principled foundation for safely operating agentic systems at scale.

### 2. 方法论：核心思想与技术细节
OpenKedge 弃用了直接调用 API 的模式，引入了以下核心流程：
*   **意图提案（Intent Proposals）**：智能体不直接发送命令，而是提交声明式的意图。
*   **治理层评估**：
    *   **上下文扩展**：从系统实时状态中提取相关信号（如资源依赖、流量指标）。
    *   **策略评估**：使用 **Policy-as-Code**（如 Cedar 语言）对意图进行确定性评估，解决权限和冲突问题。
*   **执行合约（Execution Contracts）**：获批的意图被编译为合约 $C = (a, r, t)$，明确规定允许的操作 $a$、资源范围 $r$ 和时间窗口 $t$。
*   **任务导向身份（Task-Oriented Identity）**：通过动态生成的临时凭证（如 AWS STS 令牌）强制执行合约，确保执行权限不会溢出。
*   **意图到执行证据链（IEEC）**：这是系统的骨干，通过密码学手段将“意图-上下文-决策-合约-结果”链接成不可篡改的事件流。
*   **确定性状态推导**：系统状态不是直接修改的，而是通过对事件流进行函数式折叠（Fold）推导出来的，保证了审计的确定性。

### 6. 主要结论与发现
*   **确定性仲裁**：OpenKedge 能根据权限、信任分和时间戳有效解决多智能体冲突，避免了分布式竞态条件。
*   **安全围栏**：通过执行合约和临时身份，成功拦截了所有模拟的“幻觉指令”，将故障影响范围（Blast Radius）限制在原子级别。
*   **可解释性**：IEEC 证据链使得每一个状态变更不仅“发生了什么”清晰可见，且“为什么发生”和“如何被授权”均可追溯。
*   **高性能**：在保证强治理的前提下，系统仍能支持大规模企业级吞吐量。

### 7. 优点
*   **范式转移**：将安全从“事后过滤”提升为“事前治理”和“执行绑定”，从根本上解决了 API 架构对智能体的不友好性。
*   **零信任执行**：引入任务导向身份，即使智能体逻辑被劫持或产生严重幻觉，也无法执行合约之外的操作。
*   **密码学溯源**：IEEC 提供了比传统审计日志（如 CloudTrail）更深层的因果谱系。

### 8. 不足与局限
*   **策略编写依赖**：目前高度依赖人工编写的确定性策略（如 Cedar 规则），对人类的前瞻性要求较高。
*   **治理层风险**：OpenKedge 假设治理层本身（Policy Engine, IEEC）是可信且安全的，如果治理层基础设施被攻破，整个协议将失效。
*   **应用限制**：对于极低延迟（微秒级）要求的变更场景，增加的治理环节可能会带来不可接受的开销。
*   **学习成本**：将现有 API 驱动的智能体迁移到意图驱动的 OpenKedge 协议需要对现有工具链进行适配。
