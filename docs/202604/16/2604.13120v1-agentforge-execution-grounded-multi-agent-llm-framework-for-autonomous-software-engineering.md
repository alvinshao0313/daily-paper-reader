## 摘要
大语言模型能够生成看似合理的代码，但无法验证其正确性。现有的多智能体系统往往只是模拟执行，或者将验证视为可选环节。我们引入了“基于执行的验证”作为核心原则：每一次代码更改在传播之前都必须通过沙盒执行的验证。我们在 AGENTFORGE 中实现了这一原则，这是一个多智能体框架，其中规划者（Planner）、编码者（Coder）、测试者（Tester）、调试者（Debugger）和评论者（Critic）智能体通过共享内存和强制性的 Docker 沙盒进行协作。我们将基于大语言模型的软件工程形式化为代码库状态上的迭代决策过程，其中执行反馈提供了比“下一个 Token 概率”更强的监督信号。AGENTFORGE 在 SWE-BENCH Lite 上实现了 40.0% 的解决率，比单智能体基准线高出 26-28 个百分点。消融实验证实，执行反馈和角色分解均能独立提升性能。该框架已在 https://github.com/raja21068/AutoCodeAI 开源。

## Abstract
Large language models generate plausible code but cannot verify correctness. Existing multi-agent systems simulate execution or leave verification optional. We introduce execution-grounded verification as a first-class principle: every code change must survive sandboxed execution before propagation. We instantiate this principle in AGENTFORGE, a multi-agent framework where Planner, Coder, Tester, Debugger, and Critic agents coordinate through shared memory and a mandatory Docker sandbox. We formalize software engineering with LLMs as an iterative decision process over repository states, where execution feedback provides a stronger supervision signal than next-token likelihood. AGENTFORGE achieves 40.0\% resolution on SWE-BENCH Lite, outperforming single-agent baselines by 26--28 points. Ablations confirm that execution feedback and role decomposition each independently drive performance. The framework is open-source at https://github.com/raja21068/AutoCodeAI.

### 1. 核心问题与研究动机
*   **核心问题**：大语言模型（LLM）虽然在代码生成方面表现出色，但往往无法验证生成代码的正确性。现有的多智能体框架多依赖于模拟执行或将验证视为可选步骤，导致在处理复杂的真实软件工程任务（如多文件修复、回归测试）时可靠性不足。
*   **研究背景**：软件修复需要一个“计划-实现-执行-测试-修订”的闭环。如果没有真实的执行反馈，模型无法检测回归错误或验证功能正确性，模拟执行产生的系统性误差会随流程传播。

### 2. 方法论
AgentForge 的核心思想是将软件工程建模为一个**基于执行验证的迭代细化过程**（MDP 建模）。
*   **五大专业智能体角色**：
    *   **Planner（规划者）**：生成结构化的执行计划。
    *   **Coder（编码者）**：生成最小化的统一差异补丁（Unified Diffs）。
    *   **Tester（测试者）**：合成可执行的测试用例。
    *   **Debugger（调试者）**：利用执行反馈迭代修复失败。
    *   **Critic（评论者）**：验证最终结果并决定是否存入记忆。
*   **关键技术细节**：
    *   **强制性 Docker 沙盒**：所有代码变更必须在资源受限（512MB RAM, 0.5 CPU）、网络隔离的 Docker 容器中运行，提供真实的执行信号。
    *   **双重检索机制**：结合了**情节记忆**（以往解决的任务补丁）和**实时代码库索引**（当前代码上下文），存储于 ChromaDB 向量数据库中。
    *   **迭代修复循环**：Tester 和 Debugger 构成闭环，最多允许 $N_{retry}=3$ 次尝试。
    *   **差异化编辑**：采用 Diff 格式而非全文件重写，降低了 Token 成本和错误发生概率。

### 6. 主要结论与发现
