## 摘要
我们推出了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（RL）后训练的基准测试，旨在探究 LLM 智能体能否自主设计、实现并运行完整的 RL 流水线以提升基础模型。这一能力至关重要，因为 RL 后训练正日益成为模型对齐和专业化的核心驱动力，然而现有的基准测试在很大程度上仍是静态的：仅靠监督微调（SFT）即可取得显著成果，导致交互式 RL 工程化能力未得到充分测试。Agent^2 RL-Bench 通过三个层级的六个任务解决了这一问题——从静态的基于规则的训练到带有轨迹采集的闭环在线 RL——每个层级都增加了前一层级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。通过对涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈进行评估，我们发现智能体取得了显著的交互式增益：在 ALFWorld 上，一个纯 RL 智能体通过 SFT 预热和带有在线 rollout 的 GRPO，性能从 5.97 提升至 93.28；但在其他任务上仅取得微小进展（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）。此外，驱动模型的选择对交互式任务影响巨大：在相同的脚手架（scaffold）下，更换驱动模型可使交互式改进从接近零提升至 +78 个百分点。更广泛地，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线 RL 仅在 ALFWorld 上作为最终最佳路径取得了成功。代码开源地址：https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

#### 1. 论文的核心问题与整体含义（研究动机和背景）
强化学习（RL）后训练（如 InstructGPT, DeepSeek-R1）是提升大模型对齐和推理能力的关键，但其开发过程极度依赖专家经验。论文提出了**智能体化 RL 后训练（Agentic RL Post-training）**的概念：即 LLM 智能体能否在有限预算内，自主完成从设计、实现、调试到运行完整 RL 流水线的全过程。
*   **研究动机**：现有的智能体基准测试（如 SWE-bench）多侧重于代码修复或静态任务，而现有的后训练基准测试（如 PostTrainBench）往往通过简单的监督微调（SFT）就能取得高分，无法真正测试智能体处理**在线数据采集、轨迹奖励管理和闭环 RL 训练**的工程能力。

#### 2. 论文提出的方法论
论文构建了 **Agent^2 RL-Bench**，其核心思想是**训练结构覆盖（Training Structure Coverage）**，将任务分为三个递进的难度等级：
*   **L1：静态规则任务**（如 GSM8K, HumanEval）：测试基础的数据构建和监督训练。
*   **L2：静态评判任务**（如 AlpacaEval）：引入非确定性的 LLM 评判信号。
*   **L3：交互式 Rollout 任务**（如 ALFWorld, WebShop, DeepSearchQA）：这是核心挑战，要求智能体实现环境交互、多步轨迹采集和闭环在线 RL。
*   **技术细节**：
    *   **隔离工作区**：为智能体提供只读的基础模型和训练数据，以及可写的代码/输出目录。
    *   **评分 API**：智能体可多次提交模型进行评估，获取反馈并迭代。
    *   **运行时检测与事后分析**：自动记录所有代码修订和提交记录，并生成结构化的运行报告，用于诊断智能体的搜索策略和失败原因。

#### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、指令遵循（AlpacaEval）、具身交互（ALFWorld）、电商搜索（WebShop）和深度搜索问答（DeepSearchQA）。
*   **智能体系统（Scaffolds）**：对比了 OpenHands、OpenCode、Claude Code、Codex CLI 和 Gemini CLI。
*   **驱动模型（Driver LLMs）**：包括 GPT-5.4、Claude 3.5/4.6 系列、GPT-4o、Gemini 2.5-flash 等。
*   **对比方法**：主要对比不同“智能体脚手架+驱动模型”组合在 12 小时固定工程预算下的表现，并设置了 SFT-only、RL-only 等不同操作模式的消融实验。

#### 4. 资源与算力
*   **算力设备**：论文在附录中提到使用了 **NVIDIA H200 GPU**（部分实验为 2×H200）。
*   **训练时长**：每个任务设定了严格的 **12 小时或 24 小时** 工程预算。
*   **成本估算**：以 Claude Code 为例，完成 19 个智能体运行的总 Token 成本约为 796 美元。

#### 5. 实验数量与充分性
*   **实验规模**：
    *   针对 Qwen2.5-7B-Instruct 进行了多系统对比（部分任务重复 10 次取平均）。
    *   针对 Qwen3-8B-Base 进行了受控变量研究，涵盖 3 种脚手架与 6 种驱动模型的组合。
    *   进行了详细的模式消融（SFT vs RL vs Multi-task）。
*   **充分性与公平性**：实验设计较为客观，通过隔离测试集防止了数据泄露，并引入了种子扫描（Seed Sweep）来量化评估噪声（如 DeepSearchQA 的 ±3pp 波动）。但 8B 模型的受控研究多为单次运行（Single-run），在统计严谨性上略有不足。

#### 6. 论文的主要结论与发现
*   **交互式增益显著但非均一**：在 ALFWorld 任务中，智能体能将模型性能从 5.97 提升至 93.28（通过 GRPO 算法），但在 DeepSearchQA 上几乎停滞。
*   **驱动模型是决定性因素**：在相同脚手架下，更换驱动模型可使 ALFWorld 的改进幅度从 0 变为 +78pp。
*   **SFT 仍是主流选择**：在固定预算下，智能体更倾向于构建稳定的监督流水线，仅在 ALFWorld 上成功运行了复杂的在线 RL。
*   **对齐差距的闭合依赖堆栈**：使用未对齐的 Base 模型时，强力智能体堆栈能将其提升至 Instruct 级别，而弱智能体甚至会导致性能倒退。

#### 7. 优点
*   **诊断性强**：不仅关注最终分数，还通过运行时检测分析智能体的“心路历程”和失败模式（如 OOM 恢复、格式对齐）。
*   **结构化设计**：三层难度设计清晰地界定了“自动化微调”与“自动化 RL 工程”的界限。
*   **真实模拟**：允许智能体多次提交和调试，贴合现实中算法工程师的工作流程。

#### 8. 不足与局限
*   **评估噪声**：部分交互式任务（如 DeepSearchQA）的评估方差较大，导致微小改进难以区分是算法优化还是随机性。
*   **实验覆盖不完全**：8B 模型的受控实验并非全因子设计（未测试所有脚手架与驱动的组合），且缺乏多次重复的置信区间。
*   **任务局限**：目前仅涵盖 6 个任务，且 ALFWorld 的最高分部分依赖于 SFT 组合流水线而非纯粹的在线 RL 发现。
