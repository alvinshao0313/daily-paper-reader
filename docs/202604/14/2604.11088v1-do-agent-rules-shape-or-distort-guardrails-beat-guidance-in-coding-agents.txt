## Abstract 

Developers increasingly guide AI coding agents through natural language instruction files (e.g., 

CLAUDE.md , .cursorrules ), yet no con-trolled study has measured whether these rules ac-tually improve agent performance or which prop-erties make a rule beneficial. We scrape 679 such files (25,532 rules) from GitHub and conduct the first large-scale empirical evaluation, running over 5,000 agent runs with a state-of-the-art coding agent on SWE-bench Verified. Rules improve performance by 7–14 percentage points, but ran-dom rules help as much as expert-curated ones— suggesting rules work through context priming 

rather than specific instruction. Negative con-straints (“do not refactor unrelated code”) are the only individually beneficial rule type, while positive directives (“follow code style”) actively hurt—a pattern we analyze through the lens of potential-based reward shaping (PBRS). More-over, individual rules are mostly harmful in isola-tion yet collectively helpful, with no degradation up to 50 rules. These findings expose a hidden reliability risk—well-intentioned rules routinely degrade agent performance—and provide a clear principle for safe agent configuration: constrain what agents must not do, rather than prescribing what they should .

## 2. Related Work 

Coding agents. SWE-agent (Yang et al., 2024), Devin (Cognition Labs, 2024), and OpenDevin (Wang et al., 2024) represent the current generation of autonomous cod-ing agents. These systems accept persistent configuration through system prompts or rule files, but the effect of these configurations has not been systematically studied. Arize AI (2025) reported over 15% accuracy improvement from iter-atively optimized rules on SWE-bench Lite using the Cline agent. Our results are consistent in direction; we addition-ally disentangle the contributions of context priming versus rule content, and identify which rule properties matter most. 

Prompt engineering and optimization. Extensive work has studied how prompt content affects LLM behavior (Reynolds & McDonell, 2021; Wei et al., 2022; Zhou et al., 2023). Min et al. (2022) found that in-context learning benefits more from demonstration format than content, and Webson & Pavlick (2022) showed that irrelevant prompts can be as effective as relevant ones—both motivate our prim-ing hypothesis. Automatic optimization methods like DSPy (Khattab et al., 2023) and APE (Zhou et al., 2023) search for optimal prompts programmatically. As we show in Sec-tion 6.1, the returns to content optimization may be limited for agent rule files—what matters more is the type of rule (constraint vs. directive). 

Agent scaffold optimization. A growing body of work optimizes agent systems through scaffolding, prompts, and rules rather than (or in addition to) modifying model weights (Gao et al., 2025). EvolveR (Wu et al., 2026) distills reusable strategic principles from agent interaction trajec-tories and injects them into agent context—a dynamic ana-logue of the static rule files we study. Nie et al. (2026) analyze iterative generative optimization and identify three critical design decisions, noting that the starting artifact 

(initial code, prompt, or rules) determines the reachable so-lution space. Our findings (Section 6) directly inform these systems: if the priming effect dominates, auto-optimizers need not search the full semantic space; if constraints outper-form directives, the search narrows to negative rules; and if ensemble resilience holds, greedy single-rule optimization may miss emergent collective effects. 

Reward shaping. Ng et al. (1999) established that potential-based reward shaping preserves optimal policies in RL. MiRA (Wang et al., 2026) uses milestone-based sub-goal rewards to improve long-horizon agent RL training, and PRS (Su et al., 2025) proposes progressive shaping from simple to complex objectives (e.g., formatting compliance before answer correctness). We extend the PBRS frame-work (Section 4) analogically to LLM instruction following, testing whether the qualitative insights (state-dependence, shaping vs. distortion) transfer to the agent rule setting.

## 4. Theoretical Framework: PBRS 

We borrow vocabulary from potential-based reward shap-ing (PBRS) in reinforcement learning (Ng et al., 1999) to classify agent rules. The central insight of PBRS is that auxiliary signals which depend on the agent’s state preserve optimal behavior (shaping), while state-independent signals can arbitrarily distort it. We draw the following analogy: • State-dependent rules (e.g., “if tests fail, fix the root cause”) condition on the agent’s current state, analo-gous to valid potential-based shaping. These include tool/process rules that specify when to take what ac-tion. • State-independent rules (e.g., “write clear code”) ap-ply a constant bias regardless of state, analogous to non-potential-based perturbation. These include style and architecture rules that apply uniformly. This classification generates three testable hypotheses: 

P1 State-dependent rules (tool/process) should help more than state-independent ones (style/architecture). 

P2 Constant-bias rules accumulate and degrade perfor-mance at scale (phase transition). 

P3 Shaping terms compose additively: the effect of rules A+B approximates the sum of their individual effects. We test each in Section 6; where the hypotheses fail, the failures reveal how LLM in-context processing differs from RL policy updates. 

## 5. Experimental Setup 

Agent and benchmark. We use Claude Code with Claude Opus 4.6 via AWS Bedrock as our agent—one of the top-performing coding agent systems at time of writing—and SWE-bench Verified (Jim ´enez et al., 2024) as our bench-mark. Each task consists of a real GitHub issue; the agent must produce a patch that passes the repository’s test suite. Evaluation uses the official SWE-bench Docker harness with 3Do Agent Rules Shape or Distort? 

per-repository containers ensuring correct dependencies and Python versions. 

Rule injection. Claude Code automatically reads a

CLAUDE.md file from the working directory at session start. We inject different rule conditions by placing dif-ferent CLAUDE.md files before each agent run, requiring zero modifications to the agent itself. This mirrors the real-world usage pattern where developers write and modify rule files for their projects. 

Discriminative task selection. Running all 500 SWE-bench Verified tasks is prohibitively expensive ( ∼$5 per task-condition). Moreover, preliminary experiments showed that 47% of tasks are always solved and 27% never solved regardless of rules—contributing no statistical signal. We screen 500 tasks with 3 baseline repetitions each (1,500 screening runs), selecting 58 discriminative tasks where the agent succeeds 1–2 out of 3 times (30–70% baseline pass rate). These are tasks where rules have room to help or hurt. 

Paired design. All conditions within each experiment use the same task set (58 tasks for Experiments 1–3; 35 for Experiment 4), enabling paired statistical comparisons. We use McNemar’s exact test (McNemar, 1947) for pairwise comparisons and Cochran’s Q test (Cochran, 1950) for multi-condition comparisons. 

Experiment summary. Our four experiments total 3,544 agent runs: Experiment 1 (rule sources: 8 conditions × 58 tasks = 464 runs), Experiment 2 (rule counts: 19 conditions 

× 58 tasks = 1,102 runs), Experiment 3 (rule types: 5 con-ditions × 58 tasks = 290 runs), and Experiment 4 (ablation, distortion, superposition: 1,688 runs on 35 tasks). Including 1,500 screening runs, the total is over 5,000 agent runs at an experimental cost of approximately $2,000 in API calls. 

## 6. Experiments and Results 

6.1. Experiment 1: Do Rules Help? 

We compare 8 rule conditions on 58 tasks (Figure 1a). The conditions test different aspects of rule quality: curated 

(hand-crafted best-practice rules), random (3 randomly-selected community rule files), popular (top-starred com-munity rules), matched (rules from the same programming domain), mismatched (wrong-domain rules, e.g., React rules for Python tasks), native-only (CLAUDE.md files without format conversion), and shuffled (same rules as popular, with sentences randomly reordered). 

Finding 1: Rules help, but content doesn’t matter. Ev-ery rule condition outperforms the 50.0% baseline by 7– 14pp (Figure 1a, Table 2), yet no condition is significantly different from any other (Cochran’s Q = 4 .70 , p = 0 .697 ). The closest to significance is random vs. baseline (McNe-mar p = 0 .077 ; 12 tasks helped, 4 hurt). Critically, random rules tie with curated rules at 63.8%. Mismatched (wrong-domain) rules outperform matched (same-domain) rules (58.6% vs. 56.9%). Shuffled rules—with sentences ran-domly reordered, destroying semantic coherence—perform identically to their ordered counterparts (both 56.9%). While no individual comparison reaches p < 0.05 at our sample size, the consistency is notable: all 7 non-baseline conditions show positive improvement (range 6.9–13.8pp). A binomial test on the sign of improvement (7/7 positive) yields p = 0 .008 , providing strong evidence that rules gen-uinely help despite individual comparisons lacking power. This pattern of content-independent improvement is consis-tent with context priming : the presence of any structured instructions activates a more careful problem-solving mode in the agent, regardless of their specific content. This par-allels findings by Min et al. (2022) that in-context learning benefits from demonstration format more than content , and by Webson & Pavlick (2022) that even irrelevant prompts can be as effective as relevant ones. 

6.2. Experiment 2: Phase Transition in Rule Count 

We test whether performance degrades with increasing rule counts: 0, 1, 3, 5, 10, 20, and 50 rules, each sampled with 3 different random seeds ( n = 58 tasks per condition). 

Finding 2: No phase transition; ensemble resilience. 

Pass rates remain in the 59–67% range regardless of rule count (Figure 3a). Fifty rules slightly outperform zero rules (66.7% vs. 60.3%), refuting P2: non-potential-based biases do not accumulate to degrade performance in this setting. The dominant source of variance is which rules are sam-pled (seed variance up to 17pp at count=1), not how many .Cochran’s Q = 22 .12 , p = 0 .227 across all 19 conditions, confirming no significant difference. This ensemble resilience suggests that the LLM effectively filters irrelevant or conflicting rules, treating the rule file as a noisy signal from which it extracts a general behavioral prior. The lack of degradation at 50 rules is practically important: developers need not worry about pruning their rule files.

6.4. Experiment 4: Shaping vs. Distorting 

We perform three fine-grained analyses on 35 discriminative tasks using a curated set of 18 rules (Section A) to under-stand the mechanisms by which individual rules help or hurt. 

4a: Per-rule ablation. Starting from the full 18-rule cu-rated set (65.7% pass rate), we remove each rule one at a time and measure the change (Figure 1b). Of 18 rules, 3 are 

shaping (removing hurts), 4 are distorting (removing helps), and 11 are inert (|∆| ≤ 5pp). 

Finding 4: Negative constraints shape; positive direc-tives distort. The pattern is striking (Figure 1b, Table 1): all three shaping rules are negative constraints (“do not X”), while all four distorting rules are positive directives 

(“do X”). While individual rules rarely reach significance at 

n = 35 , the perfect separation between polarity and effect direction is itself significant: Fisher’s exact test on the 2 ×2contingency table (3 shaping rules all negative, 4 distorting rules all positive) yields p = 0 .029 , confirming this is a systematic relationship rather than coincidence. The most impactful single rule—“do not refactor unrelated code”— shows a 20pp drop when removed (McNemar p = 0 .016 ,the only individually significant result). The interpretation maps onto two complementary mecha-nisms. Positive directives like “read test files” or “follow code style” broaden the agent’s action space by adding pre-scribed activities that compete for attention with the core task—analogous to reward distortion in RL, where a mis-aligned auxiliary reward diverts the policy away from the true objective. Negative constraints like “do not refactor unrelated code” prune the action space by eliminating costly deviations—analogous to state-dependent shaping that ac-tivates only when the agent is about to err, reducing the chance of over-engineering without interfering with pro-ductive behavior. The base agent already possesses strong coding capabilities; it does not need to be told what to do, but benefits from being told what not to do. 

4b: Distortion on solved tasks. We isolate 17 tasks that the baseline agent solves reliably (88.2% on re-run), then apply each of the 18 rules individually to measure distortion potential. As shown in Figure 4(a), 14 of 18 rules break at least 2 previously-solved tasks. The worst offenders—“understand full context,” “keep functions concise,” “don’t install dependencies”—each break 4/17 tasks (24%). Finding 5: Most rules are individually harmful but collectively helpful. This is perhaps our most counter-intuitive result. Rules that are individually harmful (14/18 break at least 2 tasks) somehow combine to produce a 65.7% pass rate—well above the 50% baseline. The mechanism appears to be distortion averaging : each rule biases the agent in a different direction, and when combined, these biases partially cancel out while the shared beneficial effect (the priming signal) accumulates. 

4c: Superposition test (P3). For 5 randomly-selected rule pairs from different categories, we measure pass rates under four conditions: no rules, rule A only, rule B only, and A+B together. Under PBRS additivity, P (A+B ) ≈

P (A) + P (B) − P (none ).

Finding 6: Partial additivity. Three of 5 pairs are ap-proximately additive (error <10pp), but 2 pairs show strong non-linearity (Figure 4b, Table 5). In the most extreme case (Pair 1), rule A alone hurts (−17pp) but A+B together returns to the no-rule baseline—B fully compensates for A’s distortion. In Pair 0, rules A and B each individually help (+3pp, +6pp) but together hurt (−3pp), suggesting interference. PBRS superposition (P3) thus holds only approximately. Rules can interact through compensation (one rule rescues another’s damage) or interference (two helpful rules con-flict), mechanisms that have no direct analogue in the linear PBRS framework. 

## 7. Discussion 

The priming hypothesis. Our most robust finding is that rule content matters far less than rule presence . Random rules match curated rules; shuffled match ordered; wrong-domain match right-domain. We hypothesize that rule files function primarily as context primes : structured text that activates a latent “careful software engineering” mode in the model’s generation. This is analogous to findings in prompt sensitivity research that LLMs respond to format and framing more than semantic content (Webson & Pavlick, 2022; Min et al., 2022). The convergence of four indepen-dent conditions—random, shuffled, mismatched, and native-only—all yielding similar gains makes content-specific ex-planations difficult to maintain. A strong test of the priming hypothesis would compare full rule files against minimal primes (e.g., a single sentence like “follow best practices”); we leave this to future work. This has implications beyond coding agents: any system that uses persistent instructions—system prompts, tool descrip-tions, persona definitions—may benefit from the priming effect independently of instruction quality. 

Guardrails over guidance: implications for safe deploy-ment. The negative-constraint pattern from Experiment 4a offers actionable advice: write rules that prevent bad behavior, not rules that prescribe good behavior . The agent’s base capabilities are already strong; rules add value by constraining the search space, not by providing new knowledge. This is consistent with the PBRS framework: negative constraints are more likely to be state-dependent (they activate only when the agent is about to do something undesirable), making them closer to valid potential-based shaping. This principle is well-established in human expert performance: aviation checklists prevent errors more effec-tively than procedural instructions for pilots who already know how to fly. Our distortion analysis (Section 6.4) under-scores the safety concern: 14 of 18 well-intentioned rules individually break previously-solved tasks, with the worst degrading 24% of baseline capabilities. Rule files thus rep-resent an unaudited channel through which agent reliability can be silently degraded. Consider a team that adds “follow code style” and “handle edge cases” to their agent’s config-uration, expecting improved quality. Our data shows these rules would individually break 12–24% of previously-solved tasks—a silent regression invisible to standard testing. Orga-nizations deploying agents should audit rule files for positive directives and prioritize negative constraints. 

PBRS as explanatory lens. The PBRS vocabulary proves useful: P1 is directionally supported (state-dependent rules outperform state-independent), and the shaping/distorting distinction maps cleanly onto rule polarity (Figure 1b). How-ever, P2 and P3 reveal where the analogy breaks down—and these failures are themselves informative. The absence of a phase transition (P2) implies that LLMs actively filter conflicting instructions rather than passively accumulating bias, a qualitatively different mechanism from RL policy updates. The non-additivity (P3) reveals compensation and interference dynamics that have no RL analogue. PBRS thus provides useful vocabulary and one validated hypothe-sis, while its failures delineate a concrete boundary between RL reward shaping and LLM instruction processing. 

Implications for automated rule optimization. Our find-ings have direct consequences for systems that automati-cally optimize agent rules (Khattab et al., 2023; Zhou et al., 2023; Wu et al., 2026). First, the priming effect means that content-level optimization (rephrasing rules for clarity or specificity) has diminishing returns—optimizers should focus on structural properties: rule polarity (constraint vs. directive), category composition, and count. Second, the constraint-over-directive finding provides a concrete search prior: auto-generated principles should be biased toward negative constraints that prevent over-engineering. The shelf life of rules. As models grow stronger through RL training (Cursor, 2026), procedural rules are increasingly internalized into model capabilities. Our data supports this: all three shaping rules in Table 1 are value constraints (“do not refactor,” “do not modify unrelated files”), while three of four distorting rules are procedural directives the agent has already internalized. The long-term value of rule files lies not in telling agents how to code, but in telling them what not to do—precisely the rules that RL training cannot easily absorb. 

Practical recommendations. Based on our findings, we offer four concrete recommendations: 1. Having rules matters more than which rules. Any reasonable set of coding instructions activates the prim-ing effect. Don’t agonize over phrasing. 2. Prefer “do not” over “do.” Negative constraints are the most consistently beneficial and most durable rule type. 3. Don’t worry about rule count. Up to 50 rules shows no degradation; there is no need to aggressively prune rule files. 4. Beware of well-intentioned directives. “Follow code style” and “handle edge cases” sound helpful but can actively distract a capable agent from the core task. 

Limitations. Our results are specific to Claude Code with Claude Opus 4.6 on SWE-bench Verified (Python bug fixes only); weaker models may be more sensitive to rule content, and the “guardrails over guidance” finding may be specific to tasks where the agent already has strong prior knowledge. That said, the direction of our core finding—rules help— is consistent with independent results on a different agent: Arize AI (2025) observed >15% improvement from rules on SWE-bench Lite using the Cline agent, suggesting the priming effect may generalize across agent architectures. At n=58 , our McNemar tests detect ∼15pp effects; the 14pp improvement from rules approaches but does not cross 

p < 0.05 (closest: p = 0 .077 ), though the consistent direc-tion across all 7 conditions ( p = 0 .008 , binomial sign test) reduces the chance this is noise. By selecting discriminative tasks (30–70% baseline pass rate), we enrich for signal but may not represent the full distribution of real-world tasks. Importantly, our screening data confirms that rules do not degrade performance on the remaining tasks: always-pass tasks (47%) remain solved and always-fail tasks (27%) re-main unsolved regardless of rule condition, so the selection bias does not mask hidden harm. Key extensions include cross-agent replication, constraint-aware rule evolution (Wu et al., 2026), and converting text rules into auxiliary RL reward signals (Su et al., 2025; Wang et al., 2026). 

## 8. Conclusion 

Agent rule files are the first widely-adopted mechanism for humans to persistently shape AI agent behavior, and auto-mated systems are beginning to optimize them. Our study of 25,532 rules across 679 files, evaluated through over 5,000 7Do Agent Rules Shape or Distort? 

agent runs, reveals that rules help primarily through con-text priming rather than specific instruction, that negative constraints outperform positive directives, and that rule en-sembles are surprisingly resilient. These findings provide the empirical foundation for safe agent configuration: well-intentioned rules routinely degrade performance, and the distinction between beneficial constraints and harmful direc-tives is systematic, not arbitrary. As models grow stronger, the enduring value of rule files will lie not in teaching agents how to code, but in encoding the safety constraints and or-ganizational values that training alone cannot capture. 

## Impact Statement 

This paper studies how natural language rule files affect AI coding agent behavior and reliability. Our finding that most individual rules silently degrade agent performance has direct safety implications for the growing ecosystem of community-authored rule files. We provide actionable guidance—preferring negative constraints over positive directives—that helps practitioners configure agents more safely. We see no specific ethical concerns beyond those generally associated with improving AI coding agents.

Understand the full context of the issue before proposing a fix. 4. Make minimal, targeted changes that directly address the issue. 5. Follow the existing code style and conventions of the reposi-tory. 6. Do not refactor unrelated code or add unnecessary features. 7. Preserve backward compatibility unless the issue specifically requires breaking changes. 8. Run the existing test suite to verify your changes don’t break anything. 9. If tests fail, investigate and fix the root cause rather than modifying tests. 10. Ensure your changes are complete—don’t leave TODOs or partial implementations. 11. Write clear, readable code. Prefer explicit over clever. 12. Handle edge cases that are relevant to the issue. 13. Keep functions focused and concise. 14. Use meaningful variable and function names consistent with the codebase. 15. Do not modify files unrelated to the issue. 16. Do not add type stubs, documentation, or formatting changes unless requested. 17. Do not install new dependencies unless absolutely necessary. 18. Do not change the project’s build or test configuration.
