## 摘要
基于大语言模型（LLM）的编程智能体日益依赖模型上下文协议（MCP）等工具调用协议，在开发者的工作站上进行文件读写。当写入因内容过滤、截断或会话中断而失败时，智能体通常无法接收到结构化信号，导致草稿丢失，并因盲目重试而浪费 Token。我们提出了 Resilient Write，这是一个 MCP 服务器，它在智能体与文件系统之间介入了一个六层持久化写入表面。这六层——预检风险评分、事务性原子写入、断点续传安全分块、结构化类型错误、带外草稿板存储以及任务连续性交付信封——是正交的且可独立采用。每一层都对应于 2026 年 4 月一次真实智能体会话中观察到的具体失败模式，在该会话中，内容安全过滤器静默拒绝了一份包含脱敏 API 密钥前缀的草稿。在使用该系统撰写本文的过程中，还衍生出了三个额外工具：分块预览、格式感知验证和日志分析。包含 186 个测试的套件验证了每一层的正确性，与朴素基准和防御性基准的定量对比显示，该系统使恢复时间缩短了 5 倍，智能体自我修复率提升了 13 倍。Resilient Write 已基于 MIT 许可证开源。

## Abstract
LLM-powered coding agents increasingly rely on tool-use protocols such as the Model Context Protocol~(MCP) to read and write files on a developer's workstation. When a write fails -- due to content filters, truncation, or an interrupted session -- the agent typically receives no structured signal, loses the draft, and wastes tokens retrying blindly. We present \textbf{Resilient Write}, an MCP server that interposes a six-layer durable write surface between the agent and the filesystem. The layers -- pre-flight risk scoring, transactional atomic writes, resume-safe chunking, structured typed errors, out-of-band scratchpad storage, and task-continuity handoff envelopes -- are orthogonal and independently adoptable. Each layer maps to a concrete failure mode observed during a real agent session in April~2026, in which content-safety filters silently rejected a draft containing redacted API-key prefixes. Three additional tools -- chunk preview, format-aware validation, and journal analytics -- emerged from using the system to compose this paper. A 186-test suite validates correctness at each layer, and quantitative comparison against naive and defensive baselines shows a 5x reduction in recovery time and a 13x improvement in agent self-correction rate. Resilient Write is open-source under the MIT license.

### 1. 核心问题与整体含义（研究动机和背景）
随着 LLM 编程智能体（如 Claude Code, Cursor）从建议引擎转向自主操作，它们频繁通过工具调用协议（如 MCP）在本地文件系统执行读写。然而，**写入路径极其脆弱**：
*   **静默失败**：内容安全过滤器可能因代码中包含类似 API 密钥的字符串而静默拦截写入，智能体收不到明确信号。
*   **数据丢失与截断**：大文件写入可能因传输限制被截断，或因会话中断导致草稿丢失。
*   **缺乏结构化反馈**：当写入失败时，智能体通常只收到模糊的字符串错误，导致其盲目重试，浪费大量 Token。
该研究旨在通过在智能体与文件系统之间建立一个“持久化写入表面”，解决写入过程中的不可靠性问题。

### 2. 方法论：核心思想与关键技术
Resilient Write 作为一个 MCP 服务器，采用了**六层正交架构**，每一层针对特定的失败模式：
*   **L0: 预检风险评分 (Pre-flight Risk Scoring)**：在写入前使用确定性分类器（正则匹配）扫描草稿，识别 API 密钥、JWT、PII 等敏感模式，并根据公式计算风险得分 $s$，提前预警可能触发的过滤器拦截。
*   **L1: 事务性原子写入 (Transactional Atomic Writes)**：采用“临时文件 -> fsync -> 哈希校验 -> 原子重命名”流程，确保文件要么完整写入，要么保持原样，防止半截文件产生。
*   **L2: 断点续传安全分块 (Resume-safe Chunking)**：支持将大文件分解为编号分块，支持幂等重试和顺序组合校验。
*   **L3: 结构化类型错误 (Typed Error Envelopes)**：将错误封装为 JSON 对象，包含错误类型（如 `blocked`, `stale_precondition`）、原因提示及重试配额，使智能体能根据错误类型采取不同策略。
*   **L4: 带外草稿板 (Out-of-band Scratchpad)**：提供内容寻址存储，将不宜进入代码库的敏感数据（如捕获的凭据）存储在工作区之外。
*   **L5: 任务连续性交付 (Task-Continuity Handoff)**：生成 `HANDOFF.md` 文件，记录任务状态、文件哈希和后续步骤，以便在会话中断后让新智能体快速接手。
