## 摘要
有状态的工具调用 LLM 智能体将上下文窗口视为工作内存，然而目前的智能体框架对驻留性（residency）与持久性（durability）的管理仅是尽力而为，导致了反复出现的故障：压缩后的状态丢失、重置时跳过刷新以及破坏性写回。我们提出了 ClawVM，这是一个虚拟内存层，它将状态管理为具有最小保真度不变性的有类型页，在 Token 预算下提供多分辨率表示，并在每个生命周期边界进行经过验证的写回。由于框架已经负责组装提示词、协调工具并观察生命周期事件，因此它是天然的执行点；将契约置于此处可使驻留性与持久性变得具有确定性且可审计。在合成工作负载、12 个真实会话追踪以及对抗性压力测试中，经离线真值（oracle）确认，只要最小保真度集合符合 Token 预算，ClawVM 就能消除所有策略可控的故障，且每轮增加的策略引擎开销中位数小于 50 微秒。

## Abstract
Stateful tool-using LLM agents treat the context window as working memory, yet today's agent harnesses manage residency and durability as best-effort, causing recurring failures: lost state after compaction, bypassed flushes on reset, and destructive writeback. We present \textsc{ClawVM}, a virtual memory layer that manages state as typed pages with minimum-fidelity invariants, multi-resolution representations under a token budget, and validated writeback at every lifecycle boundary. Because the harness already assembles prompts, mediates tools, and observes lifecycle events, it is the natural enforcement point; placing the contract there makes residency and durability deterministic and auditable. Across synthetic workloads, 12 real-session traces, and adversarial stress tests, \textsc{ClawVM} eliminates all policy-controllable faults whenever the minimum-fidelity set fits within the token budget, confirmed by an offline oracle, and adds median <50 microseconds of policy-engine overhead per turn.

*   **研究背景**：现有的智能体框架（Harness）对内存的管理是“尽力而为”的。这导致了三类典型故障：
    1.  **驻留性故障**：上下文压缩（Compaction）时丢失了关键指令或计划步骤。
    2.  **持久性故障**：在会话重置或保存时，未提交的状态被静默丢弃，或在写回持久化存储时发生破坏性覆盖。

*   **整体含义**：论文借鉴操作系统的虚拟内存（Virtual Memory）概念，提出在智能体框架层建立一套强制性的内存契约，使状态管理从“启发式猜测”转向“确定性控制”。

### 2. 论文提出的方法论
ClawVM 的核心思想是将智能体状态视为受管理的虚拟内存页，其关键技术包括：
*   **类型化页面（Typed Pages）**：将状态分为 Bootstrap（引导/策略）、Constraint（约束）、Plan（计划）、Preference（偏好）、Evidence（证据）和 Conversation（对话片段）等类型，每种类型都有其最小保真度不变性（Invariants）。
*   **多分辨率表示（Multi-resolution Representations）**：每个页面支持四种保真度级别：Full（全文）、Compressed（压缩文本）、Structured（结构化字段）、Pointer（指向外部存储的指针）。系统根据 Token 预算动态调整页面保真度，而非简单丢弃。
*   **确定性选择策略**：采用两阶段算法：
    1.  **第一阶段**：安装所有硬固定（Hard-pinned）页面和满足最小保真度要求的表示。
    2.  **第二阶段**：根据单位 Token 的边际效用（考虑重计算成本、近期性等），贪心地升级页面保真度。
*   **验证写回协议（Validated Writeback）**：在生命周期边界（如重置、压缩）执行三阶段事务：结构化暂存、确定性验证（检查模式、范围和非破坏性）、作用域提交。
*   **可观测故障模型**：定义了 Refetch（重新获取）、Duplicate tool（重复工具调用）、Flush-miss（刷新遗漏）等可监测的故障类型。

### 6. 论文的主要结论与发现
*   **故障消除**：在所有 Token 预算级别下，ClawVM 彻底消除了所有“策略可控”的故障（如引导丢失、刷新遗漏），而基准方法在紧凑预算下故障频发。
*   **性能接近最优**：ClawVM 的在线策略在故障计数上达到了与离线 Oracle 相同的水平。
*   **鲁棒性**：故障消除能力主要源于其结构化约束（如硬固定和生命周期写回），而非对升级启发式算法（如 LRU vs 效用评分）的微调。
*   **任务成功率**：在紧凑预算下，ClawVM 保持了 100% 的任务成功率，而现有最佳实践（Comp-Hybrid）下降至 76.7%。

### 7. 优点
*   **系统性创新**：首次将成熟的虚拟内存管理理论完整引入 LLM 智能体领域，解决了长时运行智能体的状态一致性痛点。
*   **确定性保障**：通过框架层的强制执行，使智能体行为不再依赖于模型是否“记得”去保存状态。
*   **极低开销**：算法高效，几乎不增加推理延迟，适合生产环境部署。
*   **可审计性**：详细的故障模型和写回日志使智能体的内存决策过程变得透明、可追溯。

### 8. 不足与局限
*   **语义正确性**：ClawVM 只能保证状态“不丢失”且“格式正确”，但无法验证模型生成的更新内容在语义上是否真实或正确。
*   **工具模拟依赖**：实验中的重放机制假设工具调用是可模拟的，对于具有高度非确定性副作用的外部服务，其重放保真度可能受限。
*   **应用限制**：目前主要针对单智能体场景进行了深度验证，在复杂的多智能体协作环境下的内存一致性仍需进一步研究。
*   **在线部署数据**：虽然做了实时 Hook 验证，但大规模、长期的用户在线测试数据在文中相对较少。
