## 摘要
开发者越来越多地通过自然语言指令文件（如 CLAUDE.md、.cursorrules）来引导 AI 编程智能体，但目前尚无受控研究衡量这些规则是否真的提升了智能体性能，或者哪些属性使规则有益。我们从 GitHub 抓取了 679 个此类文件（包含 25,532 条规则），并进行了首次大规模实证评估，在 SWE-bench Verified 基准上使用最先进的编程智能体运行了 5,000 多次。研究发现，规则将性能提升了 7 到 14 个百分点，但随机规则的效果与专家策划的规则相当——这表明规则是通过上下文启动（context priming）而非具体指令发挥作用的。负面约束（如“不要重构无关代码”）是唯一在单独使用时有益的规则类型，而正面指令（如“遵循代码风格”）反而会产生负面影响——我们通过基于势能的奖励塑造（PBRS）视角分析了这一模式。此外，单条规则在孤立状态下大多有害，但集体使用时却有帮助，且在规则数量达到 50 条之前性能没有下降。这些发现揭示了一个隐藏的可靠性风险——出于好意的规则往往会降低智能体性能——并为安全的智能体配置提供了一个明确的原则：约束智能体不得做的事情，而不是规定它们应该做的事情。

## Abstract
Developers increasingly guide AI coding agents through natural language instruction files (e.g., CLAUDE.md, .cursorrules), yet no controlled study has measured whether these rules actually improve agent performance or which properties make a rule beneficial. We scrape 679 such files (25,532 rules) from GitHub and conduct the first large-scale empirical evaluation, running over 5,000 agent runs with a state-of-the-art coding agent on SWE-bench Verified. Rules improve performance by 7--14 percentage points, but random rules help as much as expert-curated ones -- suggesting rules work through context priming rather than specific instruction. Negative constraints ("do not refactor unrelated code") are the only individually beneficial rule type, while positive directives ("follow code style") actively hurt -- a pattern we analyze through the lens of potential-based reward shaping (PBRS). Moreover, individual rules are mostly harmful in isolation yet collectively helpful, with no degradation up to 50 rules. These findings expose a hidden reliability risk -- well-intentioned rules routinely degrade agent performance -- and provide a clear principle for safe agent configuration: constrain what agents must not do, rather than prescribing what they should.

## 论文详细总结（自动生成）

这篇论文对 AI 编程智能体（Coding Agents）中广泛使用的自然语言规则文件（如 `.cursorrules`）进行了首次大规模实证研究，探讨了这些规则究竟是提升了性能还是引入了干扰。

### 1. 核心问题与研究动机
随着 Claude Code 和 Cursor 等工具的流行，开发者习惯编写自然语言规则来引导智能体行为。然而，目前缺乏受控研究来验证：
*   这些规则是否真的有效？
*   规则的哪些属性（内容、数量、类型）决定了其效果？
*   是否存在“好心办坏事”的情况，即规则反而降低了智能体的成功率？

### 2. 核心方法论
研究者结合了大规模语料分析、实证实验和强化学习理论：
*   **语料库构建与分类**：从 GitHub 抓取了 679 个规则文件，提取出 25,532 条规则，并将其分为六类：项目特定、行为/人格、工具/流程、代码风格、架构、安全。
*   **理论框架（PBRS）**：引入强化学习中的**基于势能的奖励塑造（Potential-Based Reward Shaping, PBRS）**理论。将规则分为“状态相关”（如：如果测试失败，则修复根因）和“状态无关”（如：编写清晰的代码）。理论预测状态相关的规则（护栏）更能保持策略的最优性。
*   **规则注入机制**：通过在智能体工作目录放置 `CLAUDE.md` 文件，在不修改智能体底层代码的情况下，观察不同规则组合对智能体解决实际编程任务的影响。

### 3. 实验设计
*   **基准测试**：使用 **SWE-bench Verified**（包含 500 个真实的 GitHub Issue）。
*   **任务筛选**：为了提高信号强度，筛选出 58 个“判别性任务”（即基准智能体成功率为 30%-70% 的任务），排除掉总是成功或总是失败的任务。
*   **对比方法**：
    *   **规则来源对比**：专家策划规则 vs. 随机规则 vs. 错误领域规则 vs. 乱序规则。
    *   **规则数量缩放**：测试 0 到 50 条规则的影响。
    *   **消融实验**：逐一移除规则以观察单条规则的贡献。
*   **智能体模型**：使用 Claude Code 配合 Claude 3.5/4.6 Opus 模型。

### 4. 资源与算力
*   **算力消耗**：论文未提及具体的 GPU 训练算力，因为实验主要基于 API 调用。
*   **实验成本**：总计运行了超过 **5,000 次**智能体任务，API 调用成本约为 **2,000 美元**。

### 5. 实验数量与充分性
*   **实验规模**：总计 5,000 多次运行，在 SWE-bench 这种高成本基准上属于大规模实验。
*   **统计严谨性**：采用了配对实验设计，并使用 McNemar 检验、Cochran's Q 检验和二项式符号检验来验证统计显著性。
*   **充分性**：涵盖了来源、数量、类型、单条消融、组合叠加等多个维度，实验设计非常全面且客观。

### 6. 主要结论与发现
*   **上下文启动效应（Priming Effect）**：规则能提升 7-14% 的性能，但**具体内容并不重要**。随机规则、乱序规则甚至错误领域的规则提升效果与专家规则相当。这表明规则的存在主要是为了激活智能体的“认真模式”。
*   **护栏优于引导（Guardrails Beat Guidance）**：
    *   **负向约束（护栏）**：如“不要重构无关代码”，是唯一能显著提升性能的类型。
    *   **正向指令（引导）**：如“遵循代码风格”，往往会分散智能体注意力，导致性能下降。
*   **群体韧性（Ensemble Resilience）**：单条规则往往是有害的（18 条规则中有 14 条在单独使用时会破坏原本能解决的任务），但当它们组合在一起时，负面影响会相互抵消，整体表现依然优于无规则状态。
*   **数量无害性**：规则数量增加到 50 条时，性能并未出现明显下降。

### 7. 优点与亮点
*   **首次大规模实证**：填补了 AI 编程智能体配置领域的研究空白。
*   **反直觉的发现**：揭示了“内容不重要”和“正向引导有害”等违反直觉的现象，具有极高的实践指导价值。
*   **理论与实践结合**：利用 PBRS 理论解释了为什么负向约束比正向指令更安全。

### 8. 不足与局限
*   **模型单一性**：实验主要基于 Claude 系列模型，更弱的模型是否对规则内容更敏感尚不可知。
*   **语言局限**：SWE-bench 主要针对 Python 任务，结论在其他编程语言或非编程任务中的泛化性有待验证。
*   **任务筛选偏差**：虽然筛选判别性任务能增强信号，但也可能忽略了规则在极简单或极困难任务中的潜在影响。
*   **静态规则限制**：研究关注的是静态规则文件，未涉及动态生成的指令或多轮对话中的实时引导。
