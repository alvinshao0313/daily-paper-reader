## 摘要
在现代数据流系统中，除了传统程序外，出现了一种能够与流数据交互的新型实体：AI 智能体。与传统程序不同，AI 智能体利用大语言模型（LLM）推理，在流数据上完成以自然语言指定的高级任务。遗憾的是，当前的数据流系统无法完全支持智能体：它们缺乏基础机制来避免由智能体任务引起的性能干扰，并安全地处理智能体写入。我们认为，作为流数据核心抽象的共享日志必须支持创建自身的分叉，而这种可分叉的共享日志是智能体在流数据上运行的理想底层架构。我们提出了 AgileLog，这是一种新型共享日志抽象，为智能体应用场景提供了创新的分叉原语。我们设计了 Bolt，它是 AgileLog 抽象的一种实现，利用新技术降低了分叉成本，并提供了逻辑与性能隔离。

## Abstract
In modern data-streaming systems, alongside traditional programs, a new type of entity has emerged that can interact with streaming data: AI agents. Unlike traditional programs, AI agents use LLM reasoning to accomplish high-level tasks specified in natural language over streaming data. Unfortunately, current streaming systems cannot fully support agents: they lack the fundamental mechanisms to avoid the performance interference caused by agentic tasks and to safely handle agentic writes. We argue that the shared log, the core abstraction underlying streaming data, must support creating forks of itself, and that such a forkable shared log serves as a great substrate for agents acting on streaming data. We propose AgileLog, a new shared log abstraction that provides novel forking primitives for agentic use cases. We design Bolt, an implementation of the AgileLog abstraction, that uses novel techniques to make forks cheap, and provide logical and performance isolation.

*   **整体含义**：本文提出将“分叉（Forking）”作为共享日志的核心原语，通过 AgileLog 抽象和 Bolt 系统，为智能体提供低成本、隔离且安全的流数据操作环境。

### 2. 方法论：核心思想与关键技术
*   **核心思想**：借鉴数据库和文件系统的分叉概念，提出**可分叉共享日志（Forkable Shared Log）**。智能体在独立的分叉上运行，而传统应用在根日志（Root Log）上运行。
*   **AgileLog 抽象原语**：
    *   **cFork (持续分叉)**：创新点。子日志不仅共享分叉点前的历史，还能**持续继承**父日志在分叉点后新写入的数据，实现单向写入隔离。
    *   **sFork (切断分叉)**：传统的快照分叉，分叉后父子互不影响。
    *   **Promote (晋升)**：经过验证后，将子日志的写入合并回父日志。
    *   **Squash (压缩)**：删除不再需要的分叉。
*   **Bolt 系统实现技术**：
    *   **无盘架构（Diskless Architecture）**：采用计算与存储分离，利用云对象存储（如 S3）保证数据持久性，通过无状态 Broker 处理读写，实现零数据拷贝分叉。
    *   **分层日志索引（HLI）**：实现**零元数据拷贝**分叉。子日志初始为空，通过递归查找父日志索引来获取历史数据。
    *   **延迟尾部树（LTT）**：利用 Euler Tour 技术和平衡二叉搜索树，以对数级复杂度将父日志的更新传播给成百上千个 cFork，避免同步更新带来的性能瓶颈。

### 6. 主要结论与发现
*   **极低开销**：Bolt 创建分叉的延迟仅为 ~50μs，且不受日志长度影响，比元数据拷贝快 3 个数量级。
*   **强隔离性**：在智能体进行高强度分析任务时，Bolt 能保持根日志的延迟稳定，而 Kafka 的延迟会增加 2.5 倍甚至更多。
*   **高扩展性**：得益于 LTT 技术，即使存在 1000 个持续继承的分叉，父日志的写入性能也几乎不受影响。
*   **安全性**：通过 promotable cFork，系统能有效拦截智能体的错误写入，避免下游消费者崩溃。

### 7. 优点
*   **原语创新**：首次提出 `cFork` 概念，解决了流数据环境下“既要隔离又要实时数据”的矛盾。
*   **架构先进**：结合了现代无盘（Diskless）架构，充分利用云存储的扩展性来实现高性能分叉。
*   **实用性强**：针对智能体“探索性”和“易错性”的特点量身定制，解决了实际生产中的痛点。

### 8. 不足与局限
*   **读取阻塞风险**：为了保证晋升（Promote）后的线性一致性，当存在可晋升分叉时，父日志在分叉点之后的读取会被暂时阻塞，这可能影响某些实时消费者的进度。
*   **元数据层压力**：虽然 LTT 优化了传播，但极端大量的分叉（如数万个）仍可能对元数据层的内存和 CPU 造成压力。
*   **应用限制**：目前主要针对单父单子的晋升逻辑，尚不支持将多个分叉的修改复杂合并（Merge）回父日志。
