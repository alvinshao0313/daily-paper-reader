## 摘要
智能体工作流通过编排多个大语言模型（LLM）和工具来执行复杂任务。在保证目标吞吐量的同时实现低延迟服务具有挑战性，因为这些工作流可以使用任意智能体框架定义，且执行时间不可预测：执行过程可能根据数据依赖性产生分支、扇出或递归。由于工作流中的 LLM 数量通常超过可用 GPU 的数量，其执行还会导致 GPU 超额订阅。我们介绍了 Scepsy，这是一种新型智能体服务系统，能够高效地将任意多 LLM 智能体工作流调度到 GPU 集群上。Scepsy 利用了这样一个洞察：虽然智能体工作流具有不可预测的端到端延迟，但每个 LLM 在总执行时间中所占的比例在不同执行过程中相对稳定。Scepsy 根据这些聚合比例决定 GPU 分配：首先，它在不同并行度下对 LLM 进行性能分析。然后，它利用这些统计数据构建聚合 LLM 流水线，这是一种用于分配方案的轻量级延迟/吞吐量预测器。为了找到在实现目标吞吐量的同时最小化延迟的 GPU 分配方案，Scepsy 使用聚合 LLM 流水线在分数级 GPU 份额、张量并行度和副本数量构成的搜索空间中进行探索。它采用分层启发式算法将最佳分配方案部署到 GPU 集群上，在遵守网络拓扑约束的同时最小化碎片化。我们在真实智能体工作流上的评估表明，与独立优化 LLM 或依赖用户指定分配的系统相比，Scepsy 实现了高达 2.4 倍的吞吐量提升和 27 倍的延迟降低。

## Abstract
Agentic workflows carry out complex tasks by orchestrating multiple large language models (LLMs) and tools. Serving such workflows at a target throughput with low latency is challenging because they can be defined using arbitrary agentic frameworks and exhibit unpredictable execution times: execution may branch, fan-out, or recur in data-dependent ways. Since LLMs in workflows often outnumber available GPUs, their execution also leads to GPU oversubscription.   We describe Scepsy, a new agentic serving system that efficiently schedules arbitrary multi-LLM agentic workflows onto a GPU cluster. Scepsy exploits the insight that, while agentic workflows have unpredictable end-to-end latencies, the shares of each LLM's total execution times are comparatively stable across executions. Scepsy decides on GPU allocations based on these aggregate shares: first, it profiles the LLMs under different parallelism degrees. It then uses these statistics to construct an Aggregate LLM Pipeline, which is a lightweight latency/throughput predictor for allocations. To find a GPU allocation that minimizes latency while achieving a target throughput, Scepsy uses the Aggregate LLM Pipeline to explore a search space over fractional GPU shares, tensor parallelism degrees, and replica counts. It uses a hierarchical heuristic to place the best allocation onto the GPU cluster, minimizing fragmentation, while respecting network topology constraints. Our evaluation on realistic agentic workflows shows that Scepsy achieves up to 2.4x higher throughput and 27x lower latency compared to systems that optimize LLMs independently or rely on user-specified allocations.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）应用从单一对话转向复杂的**智能体工作流（Agentic Workflows）**，系统面临严峻的服务挑战。这些工作流通常编排多个不同规模的模型（如生成器、验证器、重排序器）和外部工具。
*   **挑战：** 
    *   **不可预测性：** 执行路径依赖于数据，存在分支、循环和扇出，导致端到端延迟波动剧烈。
    *   **资源过载：** 工作流中的模型数量往往超过物理 GPU 数量，导致资源竞争。
    *   **优化目标冲突：** 增加副本（数据并行）提升吞吐量，但会挤占张量并行（降低延迟）的资源。
*   **核心动机：** 现有的服务系统（如 K8s 自动扩缩容或单一模型优化器）缺乏对工作流整体结构的感知，导致 GPU 利用率低下或延迟过高。

### 2. 方法论：核心思想与关键技术
Scepsy 的核心洞察是：**尽管工作流的绝对执行时间不可预测，但每个 LLM 在总执行时间中所占的“份额（Share）”在统计上是相对稳定的。**

*   **聚合 LLM 流水线 (Aggregate LLM Pipeline, ALP)：**
    *   **追踪与聚合：** 拦截 LLM 调用，提取每个模型在工作流中的平均调用次数 ($n_m$) 和平均并行度 ($p_m$)。
    *   **性能建模：** 通过回放真实轨迹进行离线 Profile，构建“工作流级”的吞吐量-延迟曲线。
    *   **预测公式：** 工作流延迟被建模为各模型延迟贡献的总和：$L_w = \sum (L_m \cdot n_m / p_m)$；吞吐量受限于最慢的模型瓶颈。
*   **联合 GPU 调度：**
    *   **搜索空间：** 调度器在“分数级 GPU 份额（Fractional GPU）”、“张量并行度（TP）”和“副本数”三个维度进行搜索。
    *   **剪枝策略：** 利用延迟比例进行剪枝，优先给高延迟模型分配更多资源，并限制 TP 度以符合硬件拓扑。
*   **拓扑感知的分数级放置：**
    *   使用分层启发式算法，优先放置大尺寸 TP 实例到 NVLink 域内，再将小模型的分数级份额（利用 NVIDIA MPS 实现隔离）填充到剩余空间，减少集群碎片。

### 3. 实验设计
*   **应用场景：** 
    1.  **RAG + Reranker：** 包含嵌入模型、生成模型和重排序模型。
    2.  **Beam Search（束搜索）：** 包含生成器和验证器，具有典型的推理时缩放（Inference-time scaling）特征。
    3.  **混合负载：** 同时运行上述两种工作流。
*   **Benchmark 与对比方法：**
    *   **Kubernetes HPA：** 基于资源利用率的传统自动扩缩容。
    *   **Aegaeon：** 针对多模型服务的 GPU 池化/多路复用系统。
    *   **Ayo：** 现有的工作流感知服务系统（侧重于请求调度而非资源分配）。

### 4. 资源与算力
*   **硬件环境：** 实验在一个包含 **16 个 GPU** 的集群上完成。
*   **具体配置：** 4 台服务器，每台配备 4 块 **NVIDIA RTX A6000 GPU**（48GB 显存），通过 PCIe 4.0 和 100 Gbps InfiniBand 互联，GPU 内部通过第三代 NVLink 成对连接。
*   **软件栈：** 基于 MicroK8s、vLLM (v0.17/v0.2) 和 SGLang 构建。

### 5. 实验数量与充分性
*   **实验规模：** 涵盖了从 4 GPU 到 16 GPU 的扩展性测试。
*   **消融实验：** 专门验证了“仅并行优化”、“仅分数级分配”与“Scepsy 全功能”的对比，证明了协同优化的必要性。
*   **多维度评估：** 进行了吞吐量-延迟曲线分析、多工作流公平性测试以及调度器搜索时间的微基准测试（证明搜索能在 1 分钟内完成）。
*   **客观性：** 实验使用了真实的开源模型（如 Llama-3 系列）和标准智能体框架逻辑，对比基准涵盖了工业界（K8s）和学术界（Aegaeon, Ayo）的最前沿方案，实验设计较为充分且公平。

### 6. 主要结论与发现
*   **性能提升：** 相比于 K8s HPA，Scepsy 在相同吞吐量下降低了高达 **27 倍** 的延迟，或在相同延迟预算下提升了 **2.4 倍** 的吞吐量。
*   **超越现有系统：** 相比于 Ayo，Scepsy 在高负载下表现更稳健，吞吐量提升达 2.1x-8.2x，证明了**资源分配优化**比单纯的请求调度更关键。
*   **分数级分配的价值：** 对于包含小模型（如 Embedding）的工作流，分数级 GPU 分配能显著提升资源利用率，避免资源浪费。

### 7. 优点
*   **框架无关性：** 不绑定特定的智能体框架（如 LangChain 或 AutoGen），通过底层代理捕获统计数据。
*   **细粒度资源管理：** 首次在智能体场景下结合了分数级 GPU 分配与张量并行优化。
*   **预测高效：** ALP 模型极其轻量，使得调度器能快速在巨大的搜索空间中找到近优解。

### 8. 不足与局限
*   **工具执行时间：** 目前假设工具（如 API 调用、数据库查询）的执行时间可忽略不计。如果工具执行耗时极长（如运行科学实验），ALP 的预测准确性会下降。
*   **负载分布假设：** 依赖于 Profile 阶段捕获的 Token 长度分布，如果推理时的输入输出长度发生剧烈偏移（如从短文本变为超长文档），预设的分配方案可能失效。
*   **冷启动与动态性：** 论文主要关注稳态下的资源分配，对于工作流结构频繁动态变化的场景（如在线修改代码逻辑），重新 Profile 的开销可能较大。
