## 摘要
检索增强生成（RAG）将大语言模型（LLM）的响应建立在外部证据之上，但将模型视为搜索结果的被动消费者：它从未看到语料库是如何组织的，也不知道尚未检索到什么，这限制了其回溯或整合分散证据的能力。我们提出了 Corpus2Skill，它在离线状态下将文档语料库蒸馏为层级化的技能目录，并允许 LLM 智能体在推理时对其进行导航。该编译流水线迭代地对文档进行聚类，在每个层级生成由 LLM 编写的摘要，并将结果实例化为一棵可导航技能文件树。在推理时，智能体获得语料库的全景视图，通过逐渐细化的摘要深入探索主题分支，并根据 ID 检索完整文档。由于层级结构是显式可见的，智能体可以推理寻找位置，从无效路径回溯，并跨分支整合证据。在企业级客户支持 RAG 基准测试 WixQA 上，Corpus2Skill 在所有质量指标上均优于稠密检索、RAPTOR 和智能体化 RAG 基线。

## Abstract
Retrieval-Augmented Generation (RAG) grounds LLM responses in external evidence but treats the model as a passive consumer of search results: it never sees how the corpus is organized or what it has not yet retrieved, limiting its ability to backtrack or combine scattered evidence. We present Corpus2Skill, which distills a document corpus into a hierarchical skill directory offline and lets an LLM agent navigate it at serve time. The compilation pipeline iteratively clusters documents, generates LLM-written summaries at each level, and materializes the result as a tree of navigable skill files. At serve time, the agent receives a bird's-eye view of the corpus, drills into topic branches via progressively finer summaries, and retrieves full documents by ID. Because the hierarchy is explicitly visible, the agent can reason about where to look, backtrack from unproductive paths, and combine evidence across branches. On WixQA, an enterprise customer-support benchmark for RAG, Corpus2Skill outperforms dense retrieval, RAPTOR, and agentic RAG baselines across all quality metrics.

### 论文总结：Don't Retrieve, Navigate (Corpus2Skill)

#### 1. 核心问题与整体含义
*   **研究动机**：传统的检索增强生成（RAG）将大语言模型（LLM）视为搜索结果的“被动消费者”。LLM 只能看到检索到的片段（“树木”），而无法感知整个语料库的组织结构（“森林”）。
*   **核心挑战**：在处理复杂、跨主题的企业级查询时，扁平化的检索系统容易遗漏关键上下文，且 LLM 无法根据现有证据判断是否需要回溯或探索其他分支。
*   **整体含义**：本文提出 **Corpus2Skill** 框架，主张将语料库“蒸馏”为可导航的层级化技能目录，让 LLM 智能体像人类查阅文件系统一样主动“导航”而非被动“检索”。

#### 2. 方法论
*   **核心思想**：采用“先编译、后导航”的策略。离线阶段将文档转化为结构化的技能树，在线阶段由智能体通过文件浏览工具自主寻找答案。
*   **关键技术细节**：
    *   **离线编译（Compile Phase）**：
        1.  **嵌入与聚类**：使用 `Qwen3-Embedding-0.6B` 对文档进行向量化，通过迭代 K-Means 算法进行自底向上的层级聚类。
        2.  **LLM 摘要生成**：在每个聚类层级，利用 LLM 生成该分支的主题摘要、覆盖的问题类型及关键词。
        3.  **技能实例化**：将聚类结果转化为物理文件系统。根聚类变为 `SKILL.md`（技能描述），子聚类变为 `INDEX.md`（索引），叶子节点为文档 ID。
    *   **在线推理（Serve Phase）**：
        1.  **渐进式披露（Progressive Disclosure）**：智能体初始仅加载技能名称和简短描述，节省上下文窗口。
        2.  **导航工具**：提供 `code_execution`（用于 `ls` 和 `cat` 浏览目录）和 `get_document(id)`（获取全文）两个工具。
        3.  **自主决策**：智能体根据摘要判断路径，支持深入探索、回溯和跨分支整合证据。

#### 3. 实验设计
*   **数据集**：**WixQA**，一个包含 6,221 篇企业客服文章和 200 个专家编写的复杂查询的真实基准测试。
*   **对比方法（Baselines）**：
    *   **传统检索**：BM25（稀疏）、Dense（稠密向量）、Hybrid（混合检索）。
    *   **层级检索**：RAPTOR（基于聚类摘要的向量检索）。
    *   **智能体化检索**：Agentic RAG（允许智能体多次调用 BM25/Dense 搜索工具）。
*   **评估指标**：词法重合度（F1, BLEU, ROUGE）、LLM 评判指标（事实性 Factuality, 上下文召回率 Context Recall）以及成本分析。

#### 4. 资源与算力
*   **编译算力**：在 32 核 CPU 服务器上，编译 WixQA 语料库（约 6k 文档）仅需 **6.5 分钟**。
*   **模型使用**：离线嵌入使用 `Qwen3-Embedding-0.6B`（可在 CPU 运行）；在线推理默认使用 `Claude 3.5 Sonnet`。
*   **编译成本**：离线 LLM 摘要生成的 API 调用费用约为 **$5–$10**。

#### 5. 实验数量与充分性
*   **实验规模**：针对 200 个复杂查询进行了全量对比实验。
*   **消融实验**：
    *   **聚类结构**：测试了不同分支因子 $p$（5, 10, 20）对导航效率的影响。
    *   **探索预算**：测试了不同交互轮数（5, 10, 20 轮）对质量的影响。
    *   **模型选择**：对比了 Sonnet 和 Haiku 作为导航引擎的性价比。
*   **充分性评价**：实验设计较为客观，通过标准化上下文窗口（8,000 字符）确保了不同检索方法在生成阶段的公平性，消融实验覆盖了核心超参数。

#### 6. 主要结论与发现
*   **性能领先**：Corpus2Skill 在所有质量指标上均显著优于基线。Token F1 达到 0.460，比 Agentic RAG 提升 19%，比 Dense 检索提升 27%。
*   **导航优势**：在处理需要跨领域知识的查询时，智能体能够通过导航发现隐藏的关联（如“课程设置”与“支付货币”的跨分支联系）。
*   **成本权衡**：虽然质量最高，但由于 Skills API 在每轮交互中都会包含已读文件的内容，其输入 Token 成本较高（单次查询约 $0.17）。

#### 7. 优点
*   **主动推理**：充分利用了 LLM 的逻辑推理和规划能力，而非仅仅将其作为文本生成器。
*   **结构透明**：导航路径清晰可追溯，开发者可以直观看到智能体为何选择了特定文档。
*   **无需向量数据库**：在线推理阶段完全依赖文件系统和 LLM，简化了生产环境的基础设施依赖。

#### 8. 不足与局限
*   **推理成本高**：多轮交互和文件内容的累积导致 Token 消耗远高于单次检索。
*   **硬聚类限制**：目前每个文档只能归属于一个分支，对于跨学科文档可能存在路由瓶颈。
*   **静态性**：目前不支持增量更新，语料库变动需重新编译。
*   **API 依赖**：高度依赖于类似 Anthropic Skills API 的文件管理机制，对上下文窗口和 Prompt 缓存有较高要求。
