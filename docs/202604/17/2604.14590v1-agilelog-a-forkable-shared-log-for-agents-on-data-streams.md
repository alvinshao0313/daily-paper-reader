## 摘要
在现代数据流系统中，除了传统程序外，出现了一种能够与流数据交互的新型实体：AI 智能体。与传统程序不同，AI 智能体利用大语言模型（LLM）推理，在流数据上完成以自然语言指定的高级任务。遗憾的是，当前的流系统无法完全支持智能体：它们缺乏避免智能体任务引起的性能干扰以及安全处理智能体写入的基础机制。我们认为，作为流数据底层核心抽象的共享日志必须支持创建自身的分叉，而这种可分叉的共享日志是智能体在流数据上运行的理想基座。我们提出了 AgileLog，这是一种新型共享日志抽象，为智能体应用场景提供了新颖的分叉原语。我们设计了 Bolt，它是 AgileLog 抽象的一种实现，利用新技术降低了分叉开销，并提供了逻辑与性能隔离。

## Abstract
In modern data-streaming systems, alongside traditional programs, a new type of entity has emerged that can interact with streaming data: AI agents. Unlike traditional programs, AI agents use LLM reasoning to accomplish high-level tasks specified in natural language over streaming data. Unfortunately, current streaming systems cannot fully support agents: they lack the fundamental mechanisms to avoid the performance interference caused by agentic tasks and to safely handle agentic writes. We argue that the shared log, the core abstraction underlying streaming data, must support creating forks of itself, and that such a forkable shared log serves as a great substrate for agents acting on streaming data. We propose AgileLog, a new shared log abstraction that provides novel forking primitives for agentic use cases. We design Bolt, an implementation of the AgileLog abstraction, that uses novel techniques to make forks cheap, and provide logical and performance isolation.

### 1. 核心问题与整体含义（研究动机和背景）
*   **背景**：随着大语言模型（LLM）的发展，AI 智能体（Agents）开始广泛参与流数据处理（如实时分析、自动化测试、供应链管理）。
*   **核心问题**：传统的共享日志（如 Kafka）无法很好地支持智能体，主要存在三个痛点：
    1.  **性能干扰**：智能体具有探索性和投机性，会产生大量不可预测的读写负载，干扰生产环境的延迟敏感型任务。
    2.  **写入安全**：智能体可能产生错误或虚假的写入，缺乏隔离验证机制。
    3.  **沙盒需求**：开发和测试智能体需要基于真实数据的隔离环境，而现有系统难以在不复制数据的情况下提供此类沙盒。
*   **核心思想**：提出一种**可分叉（Forkable）**的共享日志抽象——**AgileLog**，允许智能体在独立的分支上运行，实现逻辑与性能的完全隔离。

### 2. 方法论：核心思想与关键技术
*   **AgileLog 抽象**：
    *   **cFork（持续分叉）**：子分支不仅共享分叉点前的历史，还能**持续继承**父分支后续的更新，同时子分支的私有写入对父分支不可见。
    *   **sFork（切断分叉）**：传统的快照式分叉，分叉后父子互不干扰。
    *   **Promote（晋升）**：经过验证后，将子分支的写入合并回父分支。
*   **Bolt 系统设计（AgileLog 的实现）**：
    *   **无盘架构（Diskless Architecture）**：采用无状态代理（Brokers）和云对象存储（如 S3）分离的设计。分叉时无需复制实际数据，只需操作元数据。
    *   **分层日志索引（HLI）**：实现“零元数据复制”分叉。子分支索引初始为空，通过递归查找父分支索引来获取历史数据。
    *   **尾部更新与延迟尾部树（LTT）**：为了高效实现 cFork，父分支更新时仅更新子分支的“尾部指针”而非复制索引条目。利用欧拉遍历技术和平衡二叉搜索树实现对数级复杂度的批量更新。
    *   **性能隔离**：将分叉任务分配到不同的 Broker 节点，利用云存储的高扩展性避免资源竞争。

### 3. 实验设计
*   **应用场景**：
    1.  **IoT 实时分析智能体**：在 sFork 上执行复杂的自然语言查询。

    3.  **供应链补货智能体**：利用 cFork + Promote 机制安全地集成智能体生成的补货指令。

### 6. 主要结论与发现
*   **极低开销**：Bolt 创建分叉的延迟仅为 ~50μs，且与日志长度无关。
*   **强隔离性**：在 Agent 执行重度分析任务时，Bolt 保护了生产任务的延迟不受影响，而 Kafka 的延迟激增了 14-130 倍。
*   **高扩展性**：得益于延迟尾部更新技术，即使存在 1000 个 cFork，父分支的写入性能也几乎没有下降。
*   **安全性**：Promote 机制成功拦截了智能体产生的格式错误数据，避免了下游消费者的崩溃。

### 7. 优点
*   **首创性**：首次提出了专门为 AI 智能体设计的“可分叉共享日志”概念，特别是 **cFork（持续继承）** 原语非常契合流处理需求。
*   **架构优雅**：巧妙利用无盘架构和分层索引，实现了真正的“零数据复制”和“零元数据复制”。
*   **实用性强**：通过三个具体的 Agent 应用展示了该系统如何解决实际开发中的安全和性能痛点。

### 8. 不足与局限
*   **读取阻塞风险**：为了保证 Promote 的一致性，当存在“可晋升分叉”时，父分支在分叉点之后的读取会被暂时阻塞，这可能在长时间验证场景下影响实时性。
*   **对象存储依赖**：系统性能高度依赖云对象存储的延迟和吞吐，在私有化部署或极低延迟需求下可能受限。
*   **元数据层压力**：虽然 LTT 优化了更新，但极深的分叉嵌套（如几十层）仍会带来递归查找的微小开销。
