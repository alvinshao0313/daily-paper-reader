## 摘要
LLM 智能体越来越多地采用“技能”作为可重用的组合单元。尽管技能在不同的智能体平台间共享，但现有系统将其视为原始上下文，导致同一技能在不同智能体上的表现不一致。这种脆弱性损害了技能的可移植性和执行效率。为应对这一挑战，我们分析了 118,000 个技能，并从传统编译器设计中汲取灵感。我们将技能视为代码，将 LLM 视为异构处理器。为了使可移植性具有可操作性，我们将技能的需求分解为一组原语能力，并衡量每个“模型-框架”组合对这些能力的支持程度。基于这些能力概况，我们提出了 SkVM，这是一个专为可移植且高效的技能执行而设计的编译与运行时系统。在编译时，SkVM 执行基于能力的编译、环境绑定和并发提取。在运行时，SkVM 应用 JIT 代码固化和自适应重编译进行性能优化。我们在八种不同规模的 LLM 和三个智能体框架上评估了 SkVM，涵盖了 SkillsBench 和代表性的技能任务。结果表明，SkVM 显著提高了不同模型和环境下的任务完成率，同时将 Token 消耗降低了高达 40%。在性能方面，SkVM 通过增强并行性实现了高达 3.2 倍的加速，并通过代码固化实现了 19-50 倍的延迟降低。

## Abstract
LLM agents increasingly adopt skills as a reusable unit of composition. While skills are shared across diverse agent platforms, current systems treat them as raw context, causing the same skill to behave inconsistently for different agents. This fragility undermines skill portability and execution efficiency.   To address this challenge, we analyze 118,000 skills and draw inspiration from traditional compiler design. We treat skills as code and LLMs as heterogeneous processors. To make portability actionable, we decompose a skill's requirements into a set of primitive capabilities, and measure how well each model-harness pair supports them. Based on these capability profiles, we propose SkVM, a compilation and runtime system designed for portable and efficient skill execution. At compile time, SkVM performs capability-based compilation, environment binding, and concurrency extraction. At runtime, SkVM applies JIT code solidification and adaptive recompilation for performance optimization.   We evaluate SkVM across eight LLMs of varying scales and three agent harnesses, covering SkillsBench and representative skill tasks. Results demonstrate that SkVM significantly improves task completion rates across different models and environments while reducing token consumption by up to 40%. In terms of performance, SkVM achieves up to 3.2x speedup with enhanced parallelism, and 19-50x latency reduction through code solidification.

#### 1. 核心问题与研究动机
*   **核心问题**：随着 LLM 智能体（Agent）生态的发展，“技能”（Skills，包含自然语言描述和脚本的重用单元）已成为核心组成部分。然而，当前系统将技能视为“原始上下文”（Raw Context）直接喂给模型，导致了**可移植性差**（同一技能在不同模型上表现迥异）、**执行效率低**（Token 消耗大、延迟高）以及**环境依赖脆弱**（缺少库或工具导致失败）等问题。
*   **研究动机**：作者通过分析 118,000 个真实技能发现，技能本质上是“自然语言编写的代码”，而 LLM 是“异构处理器”。受传统编译器设计启发，需要一套编译和运行时系统来弥合静态技能描述与动态执行环境（模型、框架、硬件）之间的鸿沟。

#### 2. 方法论：SkVM 系统架构
SkVM 将技能处理分为**编译时（AOT）**和**运行时（JIT）**两个阶段：
*   **AOT 编译（三阶段优化）**：
    1.  **基于能力的编译（Capability-Based Compilation）**：定义了 26 种原子能力（如代码生成、算术推理等）及其等级。通过微测例对目标模型进行画像，根据“能力差距”对技能进行**补偿**（增加提示词/示例）或**替换**（换一种实现路径）。
    2.  **环境绑定（Environment Binding）**：提取技能对外部库、工具的依赖，生成幂等的安装脚本，在执行前确保环境就绪。
    3.  **并发提取（Concurrency Extraction）**：分析技能工作流中的数据依赖，提取数据级（DLP）、指令级（ILP）和线程级（TLP）并行机会，并重写技能以支持并发执行。
*   **运行时优化（JIT 与调度）**：
    1.  **代码固化（Code Solidification）**：识别高频出现的脚本模板，将其固化为可执行函数。后续调用直接运行脚本，绕过 LLM 推理，大幅降低延迟和 Token 消耗。
    2.  **自适应重编译**：根据执行失败日志动态修正技能描述。
    3.  **资源感知调度**：根据 API 速率限制和系统负载动态调整并发子智能体的执行。

*   **非确定性风险**：由于技能编译输入是自然语言，编译过程本身存在一定的随机性，虽然有回滚机制，但仍可能产生不稳定的编译产物。

*   **依赖画像准确性**：如果初始的能力画像（Profiling）不够准确，可能会导致错误的补偿或替换策略。
