## 摘要
功耗已成为AI推理的核心瓶颈。随着智能体AI（agentic AI）成为一种主要的负载类型，这一问题变得愈发紧迫，然而先前的功耗管理技术几乎完全集中在单轮LLM服务上。我们的分析表明，智能体服务的行为有着根本性的不同：每个请求都携带长寿命的上下文，并在工具交织的轮次中不断演进；降低GPU频率可能会使系统进入抖动（thrashing）状态，此时内存压力会急剧恶化性能和能效。这些观察结果表明，智能体服务的功耗优化需要重新思考。我们提出了KAIROS，一个用于智能体AI服务的上下文感知功耗优化系统。KAIROS将智能体上下文作为一级控制信号，共同管理GPU频率、单实例并发量以及多实例请求放置。这使得KAIROS能够在存在内存余量时节省功耗，同时避免抖动并维持性能目标。在高层面上，KAIROS以智能体粒度跟踪请求，根据上下文增长和智能体进度调整本地控制，并在实例间路由智能体，以共同提高能效和内存稳定性。在多种软件和数据工程智能体任务上的评估显示，KAIROS在满足性能目标的同时，实现了平均27%（最高达39.8%）的功耗降低。

## Abstract
Power has become a central bottleneck for AI inference. This problem is becoming more urgent as agentic AI emerges as a major workload class, yet prior power-management techniques focus almost entirely on single-turn LLM serving. Our analysis shows that agentic serving behaves fundamentally differently: each request carries long-lived context that evolves across tool-interleaved turns, and lowering GPU frequency can push the system into a thrashing regime where memory pressure sharply worsens both performance and power efficiency. These observations show that power optimization for agentic serving requires rethinking.   We present KAIROS, a context-aware power optimization system for agentic AI serving. KAIROS uses agent context as a first-class control signal to jointly manage GPU frequency, per-instance concurrency, and multi-instance request placement. This enables KAIROS to save power when memory headroom exists while avoiding thrashing and preserving performance targets. At a high level, KAIROS tracks requests at agent granularity, adapts local control to context growth and agent progress, and routes agents across instances to jointly improve power efficiency and memory stability. Evaluated across diverse software and data engineering agentic tasks, KAIROS achieves an average of 27% (up to 39.8%) power reduction while meeting the performance targets.

### 1. 核心问题与研究动机
*   **研究背景**：AI 推理的功耗已成为数据中心扩展的瓶颈。新兴的“智能体 AI”（如 ReAct 模式）与传统单轮 LLM 推理不同，它涉及多轮工具交互，每轮都会产生并累积长寿命的上下文（Context）。
*   **核心挑战**：
    *   **状态性（Stateful）**：智能体请求具有跨轮次增长的上下文，内存压力随时间动态变化。
    *   **内存抖动（Thrashing）**：在智能体场景下，简单降低 GPU 频率（DVFS）会延长任务执行时间，导致上下文在内存中滞留更久。当内存耗尽时，系统会陷入“抖动”状态（频繁重新计算或内存换入换出），反而导致功耗激增和性能崩溃。
*   **研究目的**：设计一个能感知上下文状态、在保证性能目标（SLO）的前提下最大化节能的推理服务系统。

### 2. 方法论：KAIROS 系统设计
KAIROS 将“上下文”视为一级控制信号，通过三个核心组件协同工作：
*   **智能体 ID 追踪器（Agent ID Tracker）**：通过轻量级包装器为每个智能体分配唯一 ID，使后端能识别属于同一长流程的多轮请求，从而实现针对智能体的调度。
*   **上下文感知单实例控制器**：
    *   **频率控制**：根据当前 GPU 内存中已占用的上下文比例动态调整频率。内存余量大时降低频率节能；内存压力大时提高频率加速任务完成以释放内存。
    *   **SLO 频率增强**：实时监控每个智能体的吞吐量（tokens/s）。若低于预设 SLO 目标，则强制将频率提升至最高，以确保性能。
    *   **防抖动并发控制**：设置内存阈值（如 95%），当预测新请求会导致内存溢出进入抖动区时，延迟新智能体的准入。
*   **全局上下文感知路由器**：
    *   在多实例场景下，低负载时将请求整合（Consolidation）到少数 GPU 上，让其余 GPU 处于低功耗待机态；高负载或高内存压力时，将请求分散（Spreading）以平衡内存压力。

### 3. 实验设计
*   **数据集/场景**：
    *   **SWE-bench Verified**：软件工程任务（修复 GitHub Issue）。
    *   **DABStep**：数据分析任务。
    *   **Terminal-Bench 2.0**：复杂的命令行交互任务。
*   **智能体框架**：使用 mini-swe-agent 和 Terminus-2。
*   **对比方法（Baselines）**：
    *   无频率控制（始终最高频率）。
    *   固定低频率（如 810MHz）。
    *   轮询调度（Round Robin）路由。
*   **性能指标**：P5 吞吐量（tokens/s）、平均功耗（W）、每任务能耗（kJ）、SLO 达成率。

### 4. 资源与算力
*   **硬件环境**：使用 **NVIDIA H100 NVL** GPU（80GB 显存，400W TDP），配以 Intel Xeon Platinum 8592+ CPU 和 2TB DRAM。
*   **模型**：主要使用 **Qwen3-Coder-30B**（FP8 量化），辅助测试了 Ministral-3-14B。
*   **软件**：基于 vLLM (v0.14.0) 构建，使用 Zeus 框架进行频率控制和功耗测量。

### 5. 实验数量与充分性
*   **实验规模**：实验涵盖了 3 个不同领域的基准测试、2 种智能体类型、2 种模型规模。
*   **时长与负载**：每组实验运行 3 小时以捕捉长期的上下文动态变化，并使用泊松分布模拟真实的请求到达率。
*   **充分性评价**：实验设计较为充分，不仅对比了功耗，还深入分析了内存抖动对性能的影响，并验证了不同 SLO 目标下的系统适应性。对比实验考虑了单实例和多实例两种架构，具有较强的说服力。

### 6. 主要结论与发现
*   **节能效果**：在单实例上平均降低功耗 **27%**（最高 **39.8%**）；在多实例场景下通过路由优化，功耗降低可达 **46.3%**。
*   **性能维持**：在大幅节能的同时，KAIROS 能够稳定维持 20-45 tokens/s 的人感知吞吐量 SLO，避免了固定低频方案导致的严重性能违约。
*   **防抖动价值**：证明了在智能体推理中，单纯的 DVFS 是危险的，必须配合并发控制和上下文感知才能避免系统崩溃。

### 7. 优点与亮点
*   **视角独特**：首次指出智能体推理中“功耗-性能-内存稳定性”之间的强耦合关系，并定义了“抖动区”。
*   **指标创新**：提出使用“每智能体吞吐量”而非传统的 TTFT/TBT 作为智能体服务的 SLO 指标，更符合自动化任务的特性。
*   **实用性强**：系统设计为非侵入式，无需修改智能体程序代码即可实现 ID 追踪和调度。

### 8. 不足与局限
*   **模型覆盖**：虽然测试了 30B 和 14B 模型，但未讨论在超大规模模型（如 70B+ 或混合专家模型 MoE）上的表现，这些模型可能面临更严峻的内存带宽瓶颈。
*   **智能体类型限制**：主要针对 ReAct 风格的智能体，对于具有复杂树搜索（如 Tree-of-Thought）或大规模并行规划的智能体，其上下文增长模式可能更复杂。
*   **硬件依赖**：实验高度依赖 NVIDIA H100 的 DVFS 特性，在其他架构（如国产芯片或专用 ASIC）上的迁移效果有待验证。
*   **离线重放**：为了公平对比，实验使用了请求重放，虽然保证了变量受控，但可能忽略了真实在线环境中智能体因延迟改变而产生的行为反馈（即延迟增加可能导致智能体逻辑变化）。
