## 摘要
AI 智能体正越来越多地部署在现实世界的应用中，包括 Manus、OpenClaw 和编程智能体等系统。现有研究主要集中在“服务端”效率，提出了缓存、投机执行、流量调度和负载均衡等方法，以降低提供智能体工作负载服务的成本。然而，随着用户越来越多地通过组合本地工具、远程 API 和多样化模型来构建智能体，客户端也出现了一个同样重要的优化问题。客户端优化探讨的是开发者应如何在应用特定的质量、成本和延迟约束下，在流水线各阶段分配可用资源，包括模型选择、本地工具和 API 预算。由于这些目标取决于任务和部署环境，因此无法仅由服务端系统决定。我们推出了 AgentOpt，这是首个与框架无关的用于客户端智能体优化的 Python 软件包。我们首先研究了模型选择，这是多步智能体流水线中一个具有高影响力的优化杠杆。给定一个流水线和一个小型评估集，目标是找到将模型分配给流水线角色的最具成本效益的方案。这个问题在实践中具有重大意义：在准确率相当的情况下，我们的实验表明，最佳和最差模型组合之间的成本差距可达 13-32 倍。为了高效探索呈指数级增长的组合空间，AgentOpt 实现了八种搜索算法，包括 Arm Elimination、Epsilon-LUCB、Threshold Successive Elimination 和贝叶斯优化。在四个基准测试中，Arm Elimination 在其中三个任务上恢复了接近最优的准确率，同时相对于暴力搜索减少了 24-67% 的评估预算。代码和基准测试结果可在 https://agentoptimizer.github.io/agentopt/ 获取。

## Abstract
AI agents are increasingly deployed in real-world applications, including systems such as Manus, OpenClaw, and coding agents. Existing research has primarily focused on \emph{server-side} efficiency, proposing methods such as caching, speculative execution, traffic scheduling, and load balancing to reduce the cost of serving agentic workloads. However, as users increasingly construct agents by composing local tools, remote APIs, and diverse models, an equally important optimization problem arises on the client side. Client-side optimization asks how developers should allocate the resources available to them, including model choice, local tools, and API budget across pipeline stages, subject to application-specific quality, cost, and latency constraints. Because these objectives depend on the task and deployment setting, they cannot be determined by server-side systems alone. We introduce AgentOpt, the first framework-agnostic Python package for client-side agent optimization. We first study model selection, a high-impact optimization lever in multi-step agent pipelines. Given a pipeline and a small evaluation set, the goal is to find the most cost-effective assignment of models to pipeline roles. This problem is consequential in practice: at matched accuracy, the cost gap between the best and worst model combinations can reach 13--32$\times$ in our experiments. To efficiently explore the exponentially growing combination space, AgentOpt implements eight search algorithms, including Arm Elimination, Epsilon-LUCB, Threshold Successive Elimination, and Bayesian Optimization. Across four benchmarks, Arm Elimination recovers near-optimal accuracy while reducing evaluation budget by 24--67\% relative to brute-force search on three of four tasks. Code and benchmark results available at https://agentoptimizer.github.io/agentopt/.

*   **研究动机**：不同的模型组合在准确率、成本和延迟之间存在巨大的权衡空间。由于 Agent 任务具有多步依赖性，简单的“每步选最强模型”往往不是最优解，且搜索空间随步骤增加呈指数级增长。

### 2. 方法论
*   **核心思想**：将 Agent 模型选择视为一个**端到端的黑盒优化问题**。AgentOpt 框架通过拦截 HTTP 请求，在不修改原始 Agent 代码的前提下，自动搜索并评估不同的模型组合（Combo）。
*   **关键技术细节**：
    *   **框架无关的拦截机制**：利用 Python 的 `contextvars` 和对 `httpx` 库的补丁（Patching），在传输层捕获所有 LLM 调用，记录 Token 使用、延迟和成本。
    *   **组合（Combo）抽象**：将 $N$ 个角色的 Agent 流水线视为一个整体，模型组合 $c = (m_1, m_2, \dots, m_N)$ 是评估的最小单元。
    *   **缓存与并行执行**：内置响应缓存以减少重复 API 开销，并支持受控的并发评估以绕过速率限制。
*   **搜索算法**：实现了 8 种算法以应对组合爆炸，包括：
    *   **臂消除（Arm Elimination）**：默认算法，通过置信区间逐步剔除表现差的组合。
    *   **贝叶斯优化（BO）**：利用高斯过程拟合性能表面，引导搜索。
    *   **其他**：Epsilon-LUCB、阈值逐次消除、爬山法（Hill Climbing）、随机搜索、暴力搜索及 LM 建议法。

*   **实验规模**：针对 4 个基准测试进行了详尽的暴力搜索（Brute Force）以获取真实的最优解（Ground Truth）。
*   **统计显著性**：在对比搜索算法时，每个算法均在 **50 个随机种子**下取平均值，以评估搜索行为的方差和稳定性。
*   **公平性**：所有模型均通过 AWS Bedrock 统一调用，使用相同的提示词和 Agent 逻辑，实验设计较为客观、公平。

### 6. 主要结论与发现
*   **模型能力不等于组合性能**：最强的模型（如 Claude Opus 4.6）在某些角色中表现极差。例如在 HotpotQA 中，Opus 作为“规划者”时因过于自信而跳过工具调用，导致其在 81 种组合中排名垫底。
*   **巨大的成本差距**：在准确率持平的情况下，最佳组合与最差组合的成本差异可达 **13-32 倍**。
*   **算法效率**：**臂消除（Arm Elimination）** 表现最稳健，在保持接近最优准确率的同时，比暴力搜索减少了 **24-67%** 的评估预算。
*   **LM 建议的局限性**：仅靠 LLM 预测哪些组合好（LM Proposal）在复杂多步任务中往往失效，必须进行实证评估。

### 7. 优点
*   **实用性强**：AgentOpt 采用非侵入式设计，开发者无需重写代码即可优化现有的 LangGraph 或 AutoGen 项目。
*   **视角独特**：首次系统性地定义了“客户端优化”这一层级，填补了服务端优化与应用开发之间的空白。
*   **算法丰富**：提供了多种搜索策略，适应不同预算和搜索空间规模的需求。

### 8. 不足与局限
*   **版本局限**：目前 v0.1 版本主要关注静态的模型分配，尚未涉及动态路由或工具选择的深度优化。
*   **候选池规模**：实验中每个角色的候选模型数量较少（9 个），在更大规模的模型池中，搜索算法的收敛性仍需验证。
*   **环境依赖**：优化结果高度依赖于 API 价格和模型版本的实时状态，具有一定的时效性。
*   **评估成本**：虽然减少了预算，但对于小型开发者来说，为了寻找最优组合而进行的初始评估开销仍然存在。
