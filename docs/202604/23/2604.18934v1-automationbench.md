## 摘要
现有的软件自动化 AI 基准测试很少能同时结合跨应用协同、自主 API 发现以及策略遵循。真实的业务工作流对这三者都有需求：单个任务可能跨越 CRM、收件箱、日历和即时通讯平台，这要求智能体（agent）能够找到正确的端点、遵循策略文档，并将正确的数据写入各个系统。为了填补这一空白，我们推出了 AutomationBench，这是一个用于评估 AI 智能体通过 REST API 进行跨应用工作流编排能力的基准测试。该基准测试借鉴了 Zapier 平台的真实工作流模式，任务涵盖销售、市场、运营、支持、财务和人力资源等领域。智能体必须自行发现相关的端点，遵循分层的业务规则，并在包含无关甚至误导性记录的环境中进行操作。评分采用程序化方式且仅针对最终状态：即正确的数据是否最终进入了正确的系统。即使是目前最顶尖的前沿模型，得分也低于 10%。AutomationBench 提供了一个具有挑战性且真实的衡量标准，反映了当前模型在企业实际所需的智能体能力方面所处的水平。

## Abstract
Existing AI benchmarks for software automation rarely combine cross-application coordination, autonomous API discovery, and policy adherence. Real business workflows demand all three: a single task may span a CRM, inbox, calendar, and messaging platform - requiring the agent to find the right endpoints, follow a policy document, and write correct data to each system. To address this gap, we introduce AutomationBench, a benchmark for evaluating AI agents on cross-application workflow orchestration via REST APIs. Drawing on real workflow patterns from Zapier's platform, tasks span Sales, Marketing, Operations, Support, Finance, and HR domains. Agents must discover relevant endpoints themselves, follow layered business rules, and navigate environments with irrelevant and sometimes misleading records. Grading is programmatic and end-state only: whether the correct data ended up in the right systems. Even the best frontier models currently score below 10%. AutomationBench provides a challenging, realistic measure of where current models stand relative to the agentic capabilities businesses actually need.

*   **数据集/场景**：共包含 1200 多个任务，其中 600 个公开用于研究，600 多个私有用于榜单评估。

    *   引入了负向断言（Negative Assertions）防止智能体通过“暴力尝试”或“刷屏”来骗取分数。
    *   使用了私有测试集以防止模型过拟合。

### 6. 主要结论与发现
*   **性能极低**：目前最顶尖的模型得分均**低于 10%**（最高分 Opus 4.7 仅为 9.9%），说明现有模型远未达到商业级自动化的要求。
*   **模型差异**：Gemini 和 Opus 解决的任务集重合度较低（Jaccard 相似度仅 0.17），说明不同模型在处理复杂逻辑时的擅长点不同。
*   **主要失败模式**：
    *   **虚假自信**：72%-91% 的失败案例中，模型宣称已成功但实际未完成。
    *   **搜索不力**：无法在复杂环境中持久、有条理地寻找数据。
    *   **列表处理缺失**：在处理多个条目（如 12 封邮件）时，往往只处理部分就停止。
    *   **指令遵循不严**：忽略特定的格式要求或政策细节。

### 7. 优点
*   **高度真实性**：直接源自 Zapier 的真实业务逻辑，比合成的 API 任务更具实战意义。
*   **端到端评估**：不限制解题路径，只看结果，这符合商业自动化的本质需求。
*   **确定性评分**：不使用“LLM-as-a-judge”，而是通过硬性的字符串和结构化检查，保证了评分的客观性和可重复性。
*   **多层难度设计**：通过加入政策冲突、时区陷阱等，有效区分了模型的推理深度。

### 8. 不足与局限
*   **合成数据风险**：尽管基于真实模式，但任务仍是合成的，可能存在与现实世界细微的脱节。
*   **人工审查限制**：由于任务规模大且复杂，难以对每一个生成的任务进行深度的人工逐一核查。
*   **主观性限制**：为了保证评分的确定性，基准测试限制了需要高度主观判断的任务（要求模型必须将判断转化为结构化状态）。
*   **静态环境**：虽然模拟了 API，但环境仍是相对静态的，无法完全模拟真实互联网中复杂的网络延迟或动态权限变化。
