## 摘要
UI 到代码生成要求视觉语言模型 (VLM) 从单张截图中生成数千个结构化 HTML/CSS Token，这使得视觉 Token 的效率变得至关重要。现有的压缩方法要么在推理时使用与任务无关的启发式方法选择 Token，要么将低注意力特征置零而不实际缩短序列长度——这两种方式都无法真正降低预填充 (prefill) 延迟，也无法适应 UI 截图非均匀的信息密度。与此同时，光学（编码器端学习的）压缩在文档 OCR 中表现出强大的效果，但尚未有先前的工作将这一范式应用于 UI 到代码生成。我们提出了 UIPress，这是一个插入在冻结的 ViT 编码器和 Qwen3-VL-8B 的 LLM 解码器之间的轻量级学习压缩模块。UIPress 结合了深度可分离卷积、元素引导的空间重加权以及 Transformer 细化，将约 6,700 个视觉 Token 压缩到 256 个的固定预算中。配合解码器上的低秩自适应 (LoRA) 以弥合表示差距，整个系统仅增加了约 21.7M 个可训练参数（仅占 8B 基础模型的 0.26%）。在 Design2Code 数据集上，基于相同基础模型与四个基准方法的公平比较显示，使用 256 个 Token 的 UIPress 实现了 0.8127 的 CLIP 分数，优于未压缩基准 7.5%，优于最强的推理时方法 4.6%，同时实现了 9.1 倍的首个 Token 生成时间加速。据我们所知，UIPress 是首个用于 UI 到代码生成任务的编码器端学习压缩方法。

## Abstract
UI-to-Code generation requires vision-language models (VLMs) to produce thousands of tokens of structured HTML/CSS from a single screenshot, making visual token efficiency critical. Existing compression methods either select tokens at inference time using task-agnostic heuristics, or zero out low-attention features without actually shortening the sequence -- neither truly reduces prefill latency or adapts to the non-uniform information density of UI screenshots. Meanwhile, optical (encoder-side learned) compression has shown strong results for document OCR, yet no prior work has adapted this paradigm to UI-to-Code generation. We propose UIPress, a lightweight learned compression module inserted between the frozen ViT encoder and the LLM decoder of Qwen3-VL-8B. UIPress combines depthwise-separable convolutions, element-guided spatial reweighting, and Transformer refinement to compress ${\sim}$6{,}700 visual tokens to a fixed budget of 256. Together with Low-Rank Adaptation (LoRA) on the decoder to bridge the representation gap, the entire system adds only ${\sim}$21.7M trainable parameters (0.26\% of the 8B base model). Under a fair comparison on the same base model against four baselines on Design2Code, UIPress at 256 tokens achieves a CLIP score of 0.8127, outperforming the uncompressed baseline by +7.5\% and the strongest inference-time method by +4.6\%, while delivering 9.1$\times$ time-to-first-token speedup. To the best of our knowledge, UIPress is the first encoder-side learned compression method for the UI-to-Code task.

### 1. 核心问题与整体含义
*   **研究背景**：将 UI 截图自动转换为 HTML/CSS 代码（UI-to-Code）是前端工程中的重要任务。现代视觉语言模型（VLM）通常将高分辨率截图表示为数千个视觉 Token（如 Qwen3-VL 在处理典型页面时产生约 6,700 个 Token），这导致了极高的预填充（Prefill）延迟和显存占用。
*   **核心问题**：现有的 Token 压缩方法存在局限：
    1.  **推理时方法（如 FastV）**：仅将特征置零而不缩短序列，无法降低延迟。
    2.  **启发式选择（如 VisionZip）**：与任务无关，忽略了 UI 截图中信息分布极不均匀的特性（文字和按钮比背景重要得多）。
    3.  **领域空白**：虽然“光学压缩”（编码器端学习压缩）在 OCR 领域有效，但尚未应用于复杂的 UI 到代码生成任务。
*   **研究意义**：本文旨在通过在编码器端引入轻量级学习模块，实现极高倍率的 Token 压缩，同时保持甚至提升代码生成的质量。

### 2. 方法论
UIPress 在冻结的 ViT 编码器和 LLM 解码器之间插入了一个**光学压缩器（Optical Compressor）**，并配合 **LoRA** 进行解码器适配。
*   **光学压缩器（Cθ）的三阶段流程**：
    1.  **空间卷积压缩**：使用两层级联的**深度可分离卷积（Depthwise-Separable Convolutions）**进行 4 倍空间下采样，利用卷积的归纳偏置捕捉 UI 布局的局部相关性。
    2.  **元素引导的 Token 重加权**：利用轻量级检测器（OmniParser V2）识别 UI 元素。根据类别分配权重（文字/按钮=1.0，图标/输入框=0.5，背景=0.2），使模型在随后的**自适应平均池化**中将表示能力集中在关键区域，最终固定输出 $K=256$ 个 Token。
    3.  **Transformer 精炼**：通过单层 Transformer 编码器恢复因空间池化而受损的远距离 UI 组件（如导航栏与页脚）之间的依赖关系。
*   **解码器适配（LoRA）**：在 LLM 解码器的所有层中对 Query 和 Value 投影矩阵应用 LoRA（秩 $r=16$），以弥合压缩特征与原始特征空间之间的分布差异。
*   **联合训练**：在 50K 个 WebSight 样本上，通过自回归语言建模损失函数对压缩器和 LoRA 进行端到端训练，而 ViT 和 LLM 主体保持冻结。

*   **数据集**：
    *   **训练集**：WebSight（50K 对合成截图-HTML）。
    *   **评估集**：Design2Code（485 个真实网页，包含博客、电商、仪表盘等）。
*   **Benchmark 与对比方法**：
    *   **基准模型**：Qwen3-VL-8B-Instruct。

        2.  **分辨率缩放**：调整输入分辨率至 480px（约 845 Token）。

        4.  **元素引导剪枝**：EfficientUICoder（基于边缘检测的剪枝）。
        5.  **特征置零**：FastV（注意力得分低者置零）。
*   **评价指标**：CLIP Score（衡量生成页面与原图的视觉相似度）、首字生成时间（TTFT）、端到端延迟、峰值显存。

### 6. 主要结论与发现
*   **超越无损（Beyond Lossless）**：UIPress 在仅使用 256 个 Token（压缩率 25.5x）的情况下，CLIP 分数达到 0.8127，比未压缩的基线（0.7563）高出 **7.5%**。这表明学习到的压缩能有效过滤视觉噪声，强化结构信息。
*   **极速提升**：相比基线，实现了 **9.1 倍** 的首字生成时间（TTFT）加速。
*   **适配至关重要**：消融实验显示，LoRA 适配是性能提升的最大贡献者（贡献了约 10.8% 的 CLIP 提升），证明了弥合表示间隙的必要性。
*   **最优预算**：$K=256$ 是性能与延迟的最佳平衡点，进一步增加 Token 数量收益递减。

### 7. 优点
*   **任务感知性**：通过元素引导重加权，解决了 UI 截图信息密度非均匀的问题。
*   **真正的效率提升**：不同于置零法，UIPress 实际缩短了序列长度，显著降低了计算开销。
*   **轻量且易迁移**：仅需极少的可训练参数和少量的训练数据即可在强大的基础模型上实现显著改进。
*   **开源贡献**：提供了首个 UI 到代码任务的编码器端压缩基准。

### 8. 不足与局限
*   **数据依赖**：虽然仅需 50K 数据，但仍依赖于高质量的截图-代码对，对于移动端 App 或特定设计工具（如 Figma）的泛化可能需要额外的数据微调。
*   **检测器开销**：虽然 OmniParser 运行较快且可并行，但引入了额外的预处理步骤。
*   **固定预算**：目前使用固定的 $K=256$，未来若能根据 UI 复杂度动态分配 Token 数量（如简单页面用更少 Token）会更理想。
*   **评价指标局限**：CLIP Score 侧重全局视觉相似度，可能无法完全反映 CSS 属性精度或字符级文本准确率。
