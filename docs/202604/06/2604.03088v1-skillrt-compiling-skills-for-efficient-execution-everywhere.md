## 摘要
LLM 智能体越来越多地采用“技能”作为可重用的组合单元。尽管技能在不同的智能体平台间共享，但现有系统将其视为原始上下文，导致同一技能在不同智能体上的表现不一致。这种脆弱性损害了技能的可移植性和执行效率。为应对这一挑战，我们分析了 118,000 个技能，并从传统的编译器设计中汲取灵感。我们将技能视为代码，将 LLM 视为异构处理器。为了使可移植性具有可操作性，我们将技能的需求分解为一组原语能力，并衡量每个“模型-框架”组合对这些能力的支持程度。基于这些能力画像，我们提出了 SkillRT，这是一个专为可移植且高效的技能执行而设计的编译与运行时系统。在编译时，SkillRT 执行基于能力的编译、环境绑定和并发提取；在运行时，SkillRT 应用即时（JIT）代码固化和自适应重编译来进行性能优化。我们在八种不同规模的 LLM 和三个智能体框架上评估了 SkillRT，涵盖了 SkillsBench 和代表性的技能任务。结果表明，SkillRT 显著提高了不同模型和环境下的任务完成率，同时将 Token 消耗降低了高达 40%。在性能方面，SkillRT 通过增强并行性实现了高达 3.2 倍的加速，并通过代码固化实现了 19-50 倍的延迟降低。

## Abstract
LLM agents increasingly adopt skills as a reusable unit of composition. While skills are shared across diverse agent platforms, current systems treat them as raw context, causing the same skill to behave inconsistently for different agents. This fragility undermines skill portability and execution efficiency.   To address this challenge, we analyze 118,000 skills and draw inspiration from traditional compiler design. We treat skills as code and LLMs as heterogeneous processors. To make portability actionable, we decompose a skill's requirements into a set of primitive capabilities, and measure how well each model-harness pair supports them. Based on these capability profiles, we propose SkillRT, a compilation and runtime system designed for portable and efficient skill execution. At compile time, SkillRT performs capability-based compilation, environment binding, and concurrency extraction. At runtime, SkillRT applies JIT code solidification and adaptive recompilation for performance optimization.   We evaluate SkillRT across eight LLMs of varying scales and three agent harnesses, covering SkillsBench and representative skill tasks. Results demonstrate that SkillRT significantly improves task completion rates across different models and environments while reducing token consumption by up to 40%. In terms of performance, SkillRT achieves up to 3.2x speedup with enhanced parallelism, and 19-50x latency reduction through code solidification.

#### 1. 核心问题与整体含义（研究动机和背景）
随着 LLM 智能体（Agent）生态的发展，“技能”（Skills，通常由自然语言描述和脚本组成）成为可复用的核心单元。然而，现有系统将技能视为“原始上下文”（Raw Context），直接喂给模型。这导致了三个核心问题：
*   **模型不匹配（Model Mismatch）：** 不同模型对技能指令的理解和执行能力差异巨大，导致同一技能在不同模型上表现不一，甚至导致性能下降。
*   **框架不匹配（Harness Mismatch）：** 不同的智能体运行框架（如 OpenCode, OpenClaw）提供的工具和环境不同，技能难以跨平台移植。
*   **环境不匹配（Environment Mismatch）：** 技能往往依赖特定的系统库或工具，若运行环境未配置，会导致执行失败或 Token 浪费。
**研究动机：** 借鉴传统编译器设计思想，将“技能视为代码”，将“LLM 视为异构处理器”，通过编译和运行时优化解决技能的可移植性和执行效率问题。

#### 2. 方法论：核心思想与关键技术
论文提出了 **SkillRT** 系统，包含 **AOT（预编译）** 和 **JIT（即时优化）** 两个阶段：

*   **AOT 编译阶段（Install Time）：**
    *   **基于能力的编译（Capability-Based Compilation）：** 定义了 26 种原始能力（如代码生成、逻辑推理等）及其等级。系统会剖析目标模型的能力画像，对比技能需求，通过“补偿”（增加示例/约束）或“替换”（换一种实现路径）来适配模型。
    *   **环境绑定（Environment Binding）：** 自动提取技能的依赖项（如 Python 库、CLI 工具），生成安装脚本，确保执行前环境就绪。
    *   **并发提取（Concurrency Extraction）：** 分析技能工作流中的数据依赖，提取数据级（DLP）、指令级（ILP）和线程级（TLP）并行机会，并映射到框架原语。
*   **运行时与 JIT 优化阶段（Execution Time）：**
    *   **代码固化（Code Solidification）：** 识别技能中高频出现的参数化脚本模板。一旦模式匹配成功，后续执行将绕过 LLM 推理，直接运行固化代码，大幅降低延迟和 Token 消耗。
    *   **自适应重编译（Adaptive Recompilation）：** 监控执行失败日志，利用模型自我修复的轨迹对技能进行迭代优化。
    *   **资源感知调度：** 根据系统 CPU、内存和 API 速率限制动态调整并发子智能体的执行。

#### 3. 实验设计
*   **数据集/场景：** 涵盖了 SkillsBench、PinchBench 以及从主流平台（clawhub.ai, skills.sh）收集的代表性任务，共计 118 个任务。

*   **硬件平台：** 实验主要在一台配备 **16GB 内存的 Mac Mini M4** 上完成。
*   **算力消耗：** 论文未涉及大规模模型训练，主要开销在于 AOT 编译时的模型调用（Profiling）。对于小型模型（如 Qwen 3-30B），完成全套能力剖析约需 31 分钟，成本约为 0.079 美元。

*   **实验规模：** 分析了超过 11.8 万个真实技能，针对 118 个任务在 8 种模型和 3 种框架上进行了交叉验证。
*   **充分性：** 实验设计非常全面，包含了消融实验（验证 AOT/JIT 各组件贡献）、Token 消耗分析、端到端延迟测试以及并发性能测试。
*   **客观性：** 引入了多种规模的模型和不同特性的框架，证明了 SkillRT 对弱模型的显著提升作用，也展示了对强模型的效率优化，评估维度客观。

#### 6. 主要结论与发现
*   **成功率提升：** SkillRT 平均提升任务完成率 **15.3%**。对于弱模型，优化效果尤为显著。
*   **Token 节省：** 在任务成功的前提下，最高可减少 **40%** 的 Token 消耗。
*   **性能加速：** 通过并发提取实现最高 **3.2 倍** 的墙钟时间加速；通过代码固化实现 **19–50 倍** 的延迟降低。

#### 7. 优点
*   **视角新颖：** 首次系统性地将传统编译器架构（AOT/JIT/能力剖析）引入 LLM 技能执行领域。
*   **效率极高：** 代码固化技术（JIT）通过绕过 LLM 推理，解决了 Agent 系统响应慢、成本高的痛点。
*   **实用性强：** 解决了技能在异构环境下的“脆弱性”问题，对构建工业级 Agent 生态具有重要参考价值。

#### 8. 不足与局限
*   **非确定性风险：** 技能编译本身依赖 LLM，可能引入自然语言处理的非确定性，虽然有回滚机制，但仍存在编译失败的可能。

*   **能力覆盖：** 目前定义的 26 种原始能力虽覆盖了 95% 的分析样本，但随着多模态技能或更复杂技能的出现，能力库可能需要持续维护和扩充。
