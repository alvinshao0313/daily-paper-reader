## 摘要
当智能体 A 委托给智能体 B，而智能体 B 代表用户 X 调用工具 C 时，现有框架无法回答：是谁的授权链导致了这一操作，以及它在何处违反了策略？本文介绍了 SentinelAgent，这是一个用于联邦多智能体 AI 系统中可验证委托链的形式化框架。委托链演算（DCC）定义了七项属性——六项确定性属性（权限收窄、策略保持、取证可重构性、级联遏制、范围-动作一致性、输出模式一致性）和一项概率性属性（意图保持）——并通过四个元定理和一个命题确立了确定性意图验证在实践中的不可行性。意图保持委托协议（IPDP）通过一个非大语言模型（non-LLM）的委托授权服务（DAS）在运行时强制执行所有七项属性。三点验证生命周期在 DelegationBench v4（包含 516 个场景、10 个攻击类别、13 个联邦领域）上实现了 100% 的综合真阳性率（TPR）和 0% 的假阳性率（FPR）。在黑盒对抗条件下，DAS 拦截了 30/30 次攻击，且误报率为零。在对抗性压力测试下，确定性属性不可破；面对复杂的改写，意图验证的性能下降至 13%。在 190 个政府委托示例上对自然语言推理（NLI）模型进行微调，将 P2 的 TPR 从 1.7% 提升至 88.3%（5 折交叉验证，F1=82.1%）。属性 P1、P3-P7 通过 TLA+ 模型检测在 270 万个状态下进行了机械验证，未发现任何违规。即使意图验证被规避，其余六项属性仍能将攻击者限制在允许的 API 调用、符合规范的输出、可追溯的操作、受限的级联以及合规的行为之内。

## Abstract
When Agent A delegates to Agent B, which invokes Tool C on behalf of User X, no existing framework can answer: whose authorization chain led to this action, and where did it violate policy? This paper introduces SentinelAgent, a formal framework for verifiable delegation chains in federal multi-agent AI systems. The Delegation Chain Calculus (DCC) defines seven properties - six deterministic (authority narrowing, policy preservation, forensic reconstructibility, cascade containment, scope-action conformance, output schema conformance) and one probabilistic (intent preservation) - with four meta-theorems and one proposition establishing the practical infeasibility of deterministic intent verification. The Intent-Preserving Delegation Protocol (IPDP) enforces all seven properties at runtime through a non-LLM Delegation Authority Service. A three-point verification lifecycle achieves 100% combined TPR at 0% FPR on DelegationBench v4 (516 scenarios, 10 attack categories, 13 federal domains). Under black-box adversarial conditions, the DAS blocks 30/30 attacks with 0 false positives. Deterministic properties are unbreakable under adversarial stress testing; intent verification degrades to 13% against sophisticated paraphrasing. Fine-tuning the NLI model on 190 government delegation examples improves P2 from 1.7% to 88.3% TPR (5-fold cross-validated, F1=82.1%). Properties P1, P3-P7 are mechanically verified via TLA+ model checking across 2.7 million states with zero violations. Even when intent verification is evaded, the remaining six properties constrain the adversary to permitted API calls, conformant outputs, traceable actions, bounded cascades, and compliant behavior.

#### 1. 核心问题与整体含义（研究动机和背景）
随着 AI 从对话式向智能体化（Agentic）转变，攻击面发生了根本性变化。智能体能够调用 API、执行代码并自主决策，但在复杂的多跳委托链（如智能体 A 委托 B 调用工具 C）中，现有的安全框架无法回答：**谁授权了该操作？原始意图是否在传递中被篡改？操作是否符合联邦安全策略（如 NIST 800-53）？** 论文指出，现有的 IAM（身份访问管理）工具无法处理智能体间的动态授权，导致了“委托问责制缺口”。

#### 2. 方法论：核心思想与关键技术
论文提出了 **SentinelAgent** 框架，其核心由以下三部分组成：
*   **委托链演算 (DCC)：** 一个形式化模型，定义了 7 项核心属性。
    *   **确定性属性 (P1, P3-P7)：** 包括权限单调收窄（子智能体权限不能超过父智能体）、策略保持、取证可重构性（O(n) 时间复杂度）、级联遏制、范围-动作一致性（P6）和输出模式一致性（P7）。
    *   **概率性属性 (P2)：** 意图蕴含保持，即验证子任务是否背离了初始意图。
*   **委托授权服务 (DAS)：** 一个**非大模型（Non-LLM）**的受信任组件，负责在运行时强制执行 DCC 属性。它通过 HMAC 签名发放委托令牌，避免了针对防御系统本身的提示词注入攻击。
*   **三点验证生命周期：**
    1.  **执行前 (P2)：** 三层意图检查（关键词过滤、上下文增强的 NLI 自然语言推理、良性覆盖启发式）。
    2.  **执行时 (P6)：** 范围执行代理，拦截 API 调用并比对工具清单（Manifest）。
    3.  **执行后 (P7)：** 输出模式验证器，检查返回数据是否符合预定义的类型标签。

#### 3. 实验设计

#### 5. 实验数量与充分性
*   **实验规模：** 516 个基准测试场景、2.7M 个 TLA+ 状态检测、126 种规避组合的元定理验证。
*   **充分性：** 实验设计非常全面，不仅包含静态基准测试，还引入了**黑盒红队测试**、**独立红队测试**以及**实时 LangChain 智能体集成测试**。

#### 6. 主要结论与发现
*   **综合防御力：** P2+P6+P7 的组合在基准测试中达到了 **100% 的真阳性率 (TPR) 和 0% 的假阳性率 (FPR)**。
*   **意图验证的局限：** 证明了在自然语言环境下，实现 100% 确定性的意图验证在实践中是不可行的（类比莱斯定理），因此必须采用概率性设计。
*   **微调效果：** 针对政府领域的微调将 NLI 模型的恶意检测率从 1.7% 提升至 88.3%。
*   **纵深防御：** 即使攻击者绕过了意图检查，P6 和 P7 仍能将其限制在合规的 API 调用和输出范围内。

#### 7. 优点
*   **形式化与工程化的结合：** 既有严谨的 TLA+ 证明和演算定义，又有可运行的 HTTP DAS 原型。
*   **非 LLM 核心：** 安全决策不依赖 LLM，从根源上免疫了针对防御者自身的提示词注入。
*   **合规性映射：** 首次将 NIST 800-53 的 20 项控制措施直接映射到智能体委托的技术实现上，对联邦机构极具参考价值。
*   **低延迟：** 20ms 的开销对于大多数智能体应用几乎可以忽略不计。

#### 8. 不足与局限
*   **对抗性改写问题：** 意图验证（P2）在面对高级改写攻击时表现较弱，高度依赖后续的 P6/P7 补位。
*   **DAS 中心化风险：** DAS 是单点信任中心，若被攻破则全盘崩溃（虽然论文提出了 2-of-3 签名等缓解方案）。
*   **清单维护成本：** P6 依赖于工具清单（Manifest）的准确性，随着 API 的演进，维护这些清单可能带来行政负担。
*   **语义盲区：** P7 仅验证输出类型标签，无法识别同一类型下的细微语义偏差（如合法的“拒绝原因”中包含的隐性偏见）。
