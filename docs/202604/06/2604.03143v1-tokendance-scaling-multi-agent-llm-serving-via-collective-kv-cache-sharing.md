## 摘要
多智能体大语言模型（LLM）应用通常以同步轮次的方式组织执行，其中中央调度器收集所有智能体的输出，并重新分发合并后的上下文。这种全收集（All-Gather）通信模式产生了海量的 KV 缓存冗余，因为每个智能体的提示词都包含相同的共享输出块，但现有的重用方法无法高效地利用这一点。我们提出了 TokenDance，这是一个通过利用全收集模式进行集合式 KV 缓存共享，从而扩展并发智能体数量的系统。TokenDance 的 KV 收集器（KV Collector）通过一个集合步骤在整个轮次中执行 KV 缓存重用，因此无论智能体数量多少，重用共享块的开销仅需支付一次。其差异感知存储（Diff-Aware Storage）将同级缓存编码为相对于单个主副本的块稀疏差异（block-sparse diffs），在代表性工作负载上实现了 11-17 倍的压缩。在 GenerativeAgents 和 AgentSociety 上的评估表明，在满足服务水平目标（SLO）要求的情况下，TokenDance 支持的并发智能体数量比带有前缀缓存（prefix caching）的 vLLM 多出 2.7 倍，将每个智能体的 KV 缓存存储减少了高达 17.5 倍，并比基于请求的位置无关缓存（position-independent caching）实现了高达 1.9 倍的预填充（prefill）加速。

## Abstract
Multi-agent LLM applications organize execution in synchronized rounds where a central scheduler gathers outputs from all agents and redistributes the combined context. This All-Gather communication pattern creates massive KV Cache redundancy, because every agent's prompt contains the same shared output blocks, yet existing reuse methods fail to exploit it efficiently. We present TokenDance, a system that scales the number of concurrent agents by exploiting the All-Gather pattern for collective KV Cache sharing. TokenDance's KV Collector performs KV Cache reuse over the full round in one collective step, so the cost of reusing a shared block is paid once regardless of agent count. Its Diff-Aware Storage encodes sibling caches as block-sparse diffs against a single master copy, achieving 11-17x compression on representative workloads. Evaluation on GenerativeAgents and AgentSociety shows that TokenDance supports up to 2.7x more concurrent agents than vLLM with prefix caching under SLO requirement, reduces per-agent KV Cache storage by up to 17.5x, and achieves up to 1.9x prefill speedup over per-request position-independent caching.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：在多智能体应用（如社会模拟、协作编码）中，智能体通常遵循“全收集（All-Gather）”模式：每一轮对话中，调度器收集所有智能体的输出，并将其作为公共上下文分发给所有智能体。
*   **瓶颈**：
    *   **存储冗余**：每个智能体的提示词包含相同的共享块，但由于私有历史长度不同，这些共享块在 KV Cache 中的绝对位置不同，导致现有系统（如 vLLM）无法识别重用，内存消耗随智能体数量线性增长。
    *   **计算冗余**：现有的位置无关缓存（PIC）方法对每个请求独立进行位置编码旋转（RoPE）和差异分析，在 $N$ 个智能体的轮次中重复了 $N$ 次相同的计算。
*   **研究意义**：TokenDance 通过将优化粒度从“单个请求”提升到“多智能体轮次”，显著降低了内存压力和计算开销。

### 2. 方法论：核心思想与关键技术
TokenDance 的核心思想是**“一次计算，集体共享；存储差异，按需恢复”**。
*   **轮次感知提示词接口（Round-Aware Prompt Interface）**：在逻辑块之间插入特殊的隔离符 `<TTSEP>`，使系统能识别不同请求中位置偏移但内容相同的块，从而实现基于段（Segment）的哈希索引。
*   **集合式 KV 缓存重用（Collective KV Cache Reuse）**：
    *   **KV Collector**：将同一轮次内的 $N$ 个请求分组，仅执行一次批处理 RoPE 旋转和一次“重要位置”选择，将重用开销从 $O(N)$ 降至接近 $O(1)$。
*   **差异感知存储（Diff-Aware Storage）**：
    *   **Master-Mirror 布局**：存储一个完整的“主副本（Master）”，其余智能体的缓存仅存储相对于主副本的“块稀疏差异（Block-sparse diffs）”。在典型负载下，差异部分仅占总量的 10%-20%。
*   **融合差异恢复（Fused Diff Restore）**：
    *   在 KV Cache 从存储传输到 GPU 内存的流水线中，利用 CUDA Kernel 直接将稀疏差异合并到主副本中，避免了额外的内存读写开销。

*   **容量提升**：在相同的延迟 SLO（服务水平目标）下，TokenDance 支持的并发智能体数量比 vLLM 提升了 **2.7 倍**。
*   **存储压缩**：相比存储全量 KV Cache，TokenDance 将每个智能体的平均存储开销降低了 **11-17.5 倍**。

*   **模型规模效应**：模型越大（如 14B 相比 7B），TokenDance 的存储压缩优势越明显。

### 7. 优点
*   **架构创新**：首次将分布式计算中的“集合通信（Collective Communication）”思想引入到 LLM 推理服务的 KV Cache 管理中。
*   **软硬结合**：通过自定义 CUDA Kernel 实现融合恢复，确保了压缩带来的收益不会被解压延迟抵消。
*   **实用性强**：对现有框架（如 vLLM）改动较小，且能显著提升单卡能承载的智能体规模。

### 8. 不足与局限
*   **应用场景限制**：该系统高度依赖于“All-Gather”这种特定的通信模式。如果多智能体应用不遵循同步轮次或共享上下文极少，收益将大幅下降。
*   **精度风险**：由于底层使用了 PIC 方法（选择性重计算），虽然实验显示对模拟类任务影响较小，但在对数值精度极度敏感的任务中可能存在微小偏差。
*   **硬件依赖**：融合恢复等优化针对 NVIDIA GPU 架构设计，在其他硬件平台上的移植性和表现尚未验证。
