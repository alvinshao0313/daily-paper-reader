## Abstract
LLM-based agents for text-to-SQL often struggle with latency-performance trade-off, where performance improvements come at the cost of latency or vice versa. We reformulate text-to-SQL generation within the lens of software test coverage where the original query is prepared with a suite of test cases with simpler, atomic SQLs that are executed in parallel and together ensure semantic coverage of the original query. After iterating on test case coverage, the final SQL is generated only when enough information is gathered, leveraging the explored test case SQLs to ground the final generation. We validated our framework on a state-of-the-art benchmark for text-to-SQL, Spider 2.0, achieving a new state-of-the-art with 70.2% execution accuracy.

### 1. 论文的核心问题与整体含义
*   **研究动机**：在处理如 Spider 2.0 这样具有真实世界复杂性（跨域大数据库、嵌套数据、长 SQL）的 Text-to-SQL 任务时，现有的 Agent 框架通常依赖复杂的串行推理、多次工具调用和迭代修正。这虽然提升了准确率，但导致了极高的响应延迟，难以满足交互式分析的需求。
*   **核心问题**：如何在不牺牲（甚至提升）SQL 生成准确率的前提下，通过并行化处理显著降低 Agent 的执行延迟？

### 2. 方法论：软件测试视角的并行探索
PExA 将 Text-to-SQL 任务重新表述为一个**软件测试覆盖率**问题。
*   **核心思想**：不直接尝试一次性解决复杂的自然语言（NL）查询，而是将其视为一组需要被“覆盖”的语义需求。Agent 首先生成一系列简单的、自包含的“测试用例”（原子 SQL），通过并行执行这些测试来探索数据库并收集证据，最后整合这些证据生成最终 SQL。
*   **关键子代理（Sub-Agents）**：
    1.  **Planner（规划者）**：将原始查询分解为多个独立的原子测试问题（如：检查某个过滤条件、验证 Join 关系）。
    2.  **Test Case Generator（测试生成器）**：执行 Schema 链接和数据库压缩，将测试问题转化为 SQL 并并行执行。它利用 SQL 执行器的反馈（报错、空结果、成功）进行自我修正。
    3.  **SQL Proposer（提议者）**：整合所有已执行测试用例的结果（包括成功的、失败的和探索性的），利用长上下文推理能力合成最终的长 SQL。
*   **并行化技术**：
    *   **并行规划**：单次前向传递生成整套测试计划。
    *   **并发执行**：所有测试用例同时运行，延迟由最慢的单个操作决定。
    *   **单步多路径搜索**：在生成单个测试 SQL 时，单次调用产生多个候选路径，提高搜索效率。

### 6. 主要结论与发现
*   **性能领先**：PExA 在 Spider 2.0 上取得了 **70.2%** 的执行准确率，刷新了 SOTA 记录。
*   **效率优化**：通过并行探索，PExA 在保持高性能的同时，延迟与 ReFoRCE 等非 Agent 方法相当，远优于传统的串行 Agent。
*   **理论增益**：理论分析显示并行模式比串行模式在相同搜索空间下延迟降低了约 2 倍。
*   **模型差异**：GPT 系列模型在 SQL 生成和长上下文推理上优于 Claude 系列；Claude 在规划阶段表现尚可，但在长 SQL 合成上较弱。

### 7. 优点与亮点
*   **范式迁移**：巧妙地将软件工程中的“测试驱动”和“覆盖率”概念引入 SQL 生成，使复杂问题模块化。
*   **高度并行**：打破了 Agent 框架必须“慢”的固有印象，通过单步多路径和并发执行优化了帕累托前沿。
*   **鲁棒性**：通过“过度生成”测试用例，不仅探索目标信息，也探索非目标信息，为最终生成提供了极强的上下文支撑。

### 8. 不足与局限
*   **经济成本**：由于涉及大量的并行调用和推理 token（尤其是 o3 这种推理模型），运行成本非常高昂。
*   **闭源依赖**：性能高度依赖于顶尖的闭源模型（如 GPT-o3），在开源模型上的泛化能力尚未得到充分验证。
*   **应用限制**：虽然并行化降低了延迟，但 5 分钟以上的平均响应时间对于实时交互场景仍然偏长，更适合离线分析或复杂报表生成。
