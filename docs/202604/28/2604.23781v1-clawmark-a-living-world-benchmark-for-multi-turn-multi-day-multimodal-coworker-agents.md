## 摘要
语言模型智能体正越来越多地被用作持久的同事，在多个工作日内协助用户。在这些工作流中，周围环境可能会独立于智能体发生变化：新邮件到达、日历条目变动、知识库记录更新，且证据会出现在图像、扫描版PDF、音频、视频和电子表格中。现有的基准测试无法充分评估这一场景，因为它们通常在单个静态回合内运行，且主要以文本为中心。我们推出了 ClawMark，这是一个针对同事智能体的基准测试，围绕多轮多天任务、状态随轮次演进的有状态沙箱服务环境以及基于规则的验证而构建。当前版本包含跨 13 个专业场景的 100 个任务，在五个有状态沙箱服务（文件系统、电子邮件、日历、知识库、电子表格）上执行，并由 1537 个确定性 Python 检查器根据执行后的服务状态进行评分；评分过程中不调用 LLM 作为裁判。我们对七个前沿智能体系统进行了基准测试。最强的模型达到了 75.8 的加权分数，但最佳的严格任务成功率仅为 20.0%，这表明局部进展很常见，而完整的端到端工作流完成仍然很少见。轮次级分析显示，在第一次外生环境更新后性能有所下降，突显了适应不断变化的状态是一个关键的开放挑战。我们发布了该基准测试、评估工具和构建流水线，以支持可重复的同事智能体评估。

## Abstract
Language-model agents are increasingly used as persistent coworkers that assist users across multiple working days. During such workflows, the surrounding environment may change independently of the agent: new emails arrive, calendar entries shift, knowledge-base records are updated, and evidence appears across images, scanned PDFs, audio, video, and spreadsheets. Existing benchmarks do not adequately evaluate this setting because they typically run within a single static episode and remain largely text-centric. We introduce \bench{}, a benchmark for coworker agents built around multi-turn multi-day tasks, a stateful sandboxed service environment whose state evolves between turns, and rule-based verification. The current release contains 100 tasks across 13 professional scenarios, executed against five stateful sandboxed services (filesystem, email, calendar, knowledge base, spreadsheet) and scored by 1537 deterministic Python checkers over post-execution service state; no LLM-as-judge is invoked during scoring. We benchmark seven frontier agent systems. The strongest model reaches 75.8 weighted score, but the best strict Task Success is only 20.0\%, indicating that partial progress is common while complete end-to-end workflow completion remains rare. Turn-level analysis shows that performance drops after the first exogenous environment update, highlighting adaptation to changing state as a key open challenge. We release the benchmark, evaluation harness, and construction pipeline to support reproducible coworker-agent evaluation.

#### 1. 核心问题与整体含义
*   **研究动机**：当前的 AI 智能体（Agent）基准测试大多局限于**单次、静态、以文本为中心**的会话。然而，真实的办公场景要求智能体作为“同事”（Coworker）能够跨越多个工作日持续存在。
*   **核心挑战**：在真实世界中，环境会独立于智能体发生变化（如收到新邮件、日程冲突、知识库更新），且证据往往分散在图像、扫描 PDF、音视频和电子表格等**多模态**载体中。
*   **论文贡献**：提出了 **ClawMark**，这是首个结合了多天任务流、动态演进环境、全多模态证据以及完全基于规则（非 LLM 裁判）评分的智能体基准测试。

#### 2. 方法论
*   **核心思想**：模拟一个“生活化世界”（Living-World），智能体需在 2-6 个虚拟工作日（Turn）内完成任务，环境在轮次之间会发生**外生性变化**。
*   **关键技术细节**：
    *   **五大沙箱服务**：文件系统（Docker 挂载）、电子邮件（GreenMail）、日历（Radicale）、知识库（Notion 兼容）、电子表格（Google Sheets 兼容）。
    *   **环境动态性**：分为“响亮事件”（Loud events，在提示词中告知）和“静默突变”（Silent mutations，不告知，需智能体主动检查发现）。
    *   **多模态集成**：提供原始音视频、图片和复杂 PDF，不提供预先转录的文本，要求智能体调用工具（如 ffmpeg, whisper）自行处理。
*   **评分机制**：
    *   **加权得分 (Score)**：基于 1537 个确定性 Python 检查器，计算通过的检查器权重之和与总权重之比。
    *   **严格任务成功率 (Task Success)**：只有当一个任务的所有检查器全部通过时，才计为成功（All-or-nothing）。
    *   **红线约束 (Red-line)**：设定了 55 个合规性检查点（如严禁在报告出来前批准赔付），违反红线会导致高额扣分。

#### 3. 实验设计
*   **数据集/场景**：包含 100 个任务，涵盖 13 个专业领域（医疗助理、保险理赔、法律助理、投资分析、EDA 电子设计自动化、人力资源等），涉及 87 种不同的职业角色。
*   **对比方法**：评估了 7 个前沿模型：
    *   **闭源模型**：Claude Sonnet 4.6, Claude Opus 4.6, GPT-5.4 (high), Gemini 3.1 Pro Preview, Qwen 3.6 Plus。
    *   **开源模型**：Kimi K2.6, Kimi K2.5。
*   **测试框架**：所有模型均在统一的 **OpenClaw** 框架下运行，使用相同的工具 Schema，不进行针对性提示词工程。

#### 4. 资源与算力
*   **算力说明**：论文未提及模型训练所需的 GPU 资源（因为是基准测试评估）。
*   **推理开销**：详细记录了每个模型完成全量测试的**墙钟时间**（Wall time）和 **Token 消耗**。例如，Claude Sonnet 4.6 完成 100 个任务耗时 22.3 小时，消耗约 2.58 亿个 Token。

#### 5. 实验数量与充分性
*   **实验规模**：共进行了 10,759 次检查器评估。
*   **充分性分析**：
    *   **稳定性测试**：对 Kimi K2.6 和 GPT-5.4 进行了 3 次独立全量运行，结果波动较小（1.0 - 2.8 个百分点），证明了测试的稳定性。
    *   **多维度分析**：除了总分，还进行了轮次级轨迹分析（Day 1 vs Day 2 vs Day 3）和失败模式分类统计。
*   **客观性**：采用“发布门禁”（Release Gate）机制，确保每个任务的检查器在两次独立运行中产生位对位一致（Bit-identical）的结果，排除了 LLM 裁判的主观偏差。

#### 6. 主要结论与发现
*   **性能瓶颈**：最强模型（Claude Sonnet 4.6）加权得分仅 75.8，而**严格任务成功率最高仅为 20.0%**，说明完成端到端复杂工作流对当前模型仍极具挑战。
*   **环境适应性差**：大多数模型在 Day 2（环境首次发生外生变化）时性能显著下降，难以识别和适应“静默突变”。
*   **失败模式**：**静默变化检测**（失败率 56.5%）和**后端状态写入**（失败率 53.6%）是两大主要失败点。
*   **合规性风险**：即使是高分模型也会触碰“红线”，在某些任务中所有测试模型均出现了违规行为。

#### 7. 优点
*   **真实性高**：模拟了多天办公的动态性，比静态基准更贴近实际应用。
*   **评估严谨**：完全摒弃 LLM 裁判，使用确定性代码检查沙箱服务的最终状态，保证了评估的可重复性和客观性。
*   **多模态深度**：强调原始多模态证据的处理，而非简单的图文配对。
*   **专业领域覆盖**：引入了 EDA、保险、医疗等垂直行业场景，提升了测试的专业深度。

#### 8. 不足与局限
*   **任务规模有限**：100 个任务虽然质量高，但相比某些包含数千个任务的基准，样本量仍较小。
*   **场景分布不均**：部分场景（如 EDA）仅包含 1 个任务，难以代表该领域的稳定趋势。
*   **运行成本高**：由于涉及多轮交互和复杂沙箱环境，单次全量评估的时间和 Token 成本较高，不利于频繁迭代测试。
*   **框架依赖**：虽然宣称框架无关，但实验主要基于 OpenClaw，不同框架的实现细节可能影响模型表现。
