## Abstract
Training effective software engineering agents requires large volumes of task-specific trajectories, incurring substantial data construction costs. Inspired by the "Less-Is-More" hypothesis in mathematical reasoning, we investigate its extension to agentic scenarios and propose an end-to-end training framework that achieves superior agentic capabilities with fewer but higher-quality training trajectories. This is achieved via STITCH (Sliding-memory Trajectory Inference and Task Chunking Heuristic), a coarse-to-fine mechanism that filters low-value noise and retains decision-critical tokens to maximize training signal quality. We conduct experiments across multiple agent frameworks (e.g., mini-SWE-agent, MSWE-agent), model scales (30B to 355B), and multilingual settings (Python, Java, and ArkTS). On SWE-bench Verified, models trained with STITCH achieve up to 63.16% relative improvement over base models. On Multi-SWE-bench (Java), MiniMax-M2.5-STITCH achieves 43.75% with our CodeArts Agent scaffold (+16.67%). On HarmonyOS (ArkTS), GLM-4.7-STITCH improves the compilation pass rate to 61.31% (+43.34%) with less than 1K training trajectories. Our results confirm that the "Less-Is-More" paradigm generalizes effectively to complex agentic tasks across diverse languages and model scales.

### 1. 核心问题与研究动机
*   **核心问题**：训练能够处理复杂软件工程任务（如修复 Bug、跨文件导航、工具调用）的智能体通常需要海量的任务轨迹数据，这导致了极高的数据构建成本和标注负担。
*   **研究动机**：受数学推理领域 LIMO（Less-Is-More）研究的启发，作者提出：如果基础模型已经具备了足够的代码理解和工具使用能力，是否可以通过极少量（如几百条）但极高质量的轨迹数据，高效地激发其智能体（Agentic）能力？

### 2. 方法论：STITCH 框架
论文提出了一个端到端的训练框架，包含两个核心组件：

*   **SandForge（数据构建与评估平台）**：
    *   将真实的 GitHub Issue 转化为可执行的任务。
    *   自动收集智能体执行过程中的轨迹、奖励信号、测试输出和补丁（Patch）。
    *   支持多语言（Python, Java, ArkTS）和多种智能体架构。
*   **STITCH（粗到精的轨迹筛选机制）**：
    *   **宏观预筛选（Macro-Level）**：利用轻量级的逻辑回归（LR）模型，基于自动发现的特征（如代码产量、工具调用成功率、效率、错误恢复尝试等）快速过滤掉统计学上的低质量轨迹。
    *   **微观语义提取（Micro-Level）**：
        *   **启发式安全切分**：针对长轨迹，在不破坏“动作-观察”逻辑链的前提下寻找安全切分点。
        *   **滑动记忆 Map-Reduce**：在 Map 阶段，利用 LLM 评估器对轨迹片段打分，并通过“滑动记忆”机制保持跨片段的上下文连贯性；在 Reduce 阶段，汇总局部得分以识别全局高质量轨迹，甚至从失败的全局轨迹中提取出局部高价值的子任务片段。

### 3. 实验设计
*   **数据集与基准测试（Benchmark）**：
    *   **Python**：使用 **SWE-bench Verified**（500个案例）。
    *   **Java**：使用 **Multi-SWE-bench (Java)**（128个真实 GitHub 问题）。
    *   **ArkTS（鸿蒙开发语言）**：自建基准测试（168个需求），评估编译通过率和视觉预览通过率。
*   **对比方法**：
    *   **Base Models**：原始指令微调模型。
    *   **RFT (Reject Sampling Fine Tuning)**：传统的拒绝采样微调。
    *   **STITCH (Ours)**：经过精细筛选后的少量数据微调。
*   **涵盖模型**：从 30B 到 355B 不等，包括 Qwen3-Coder-30B、GLM-4.5/4.7 系列、MiniMax-M2.5 等。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了三种主流编程语言、多个模型参数量级（30B, 100B+, 200B+, 300B+）以及多种智能体框架（mini-SWE-agent, MSWE-agent, CodeArts Agent）。
*   **充分性评价**：实验设计较为充分。作者不仅在公开榜单上进行了对比，还针对低资源语言（ArkTS）进行了泛化性验证。通过对比 RFT 和 STITCH，客观地证明了“质量优于数量”的观点。

### 6. 主要结论与发现
*   **“少即是多”范式有效**：仅需不到 1000 条高质量轨迹，即可显著提升大模型的智能体能力。
*   **性能提升显著**：
    *   在 SWE-bench Verified 上，30B 模型相对基座提升了 **63.16%**。
    *   在 Java 场景下，MiniMax-M2.5 配合 CodeArts Agent 达到了 **43.75%** 的解决率，超越了 GPT-5.2 等闭源模型。
    *   在 ArkTS 场景下，编译通过率提升了 **43.34%**。
*   **模型规模效应**：STITCH 在中小规模模型（30B-100B）上提升最为剧烈；在超大规模 SOTA 模型上虽有提升，但由于基座能力已接近饱和，提升幅度相对较小。

### 7. 优点与亮点
*   **数据效率极高**：挑战了“大数据量微调”的传统认知，极大地降低了智能体训练的门槛。
*   **STITCH 机制创新**：引入滑动记忆的 Map-Reduce 评估方法，解决了长轨迹难以被 LLM 完整评估且易断裂语义的痛点。
*   **多语言泛化**：证明了该方法在 ArkTS 等新兴、低资源语言上的强大迁移能力。
*   **工程落地性强**：SandForge 框架实现了从任务构建到自动化视觉评估（针对 UI 任务）的全流程闭环。

### 8. 不足与局限
*   **SOTA 模型边际效应**：在 200B+ 的顶尖模型上，STITCH 带来的绝对增益变小，暗示该方法可能更侧重于“能力激发”而非“能力创造”。
*   **评估成本**：虽然训练数据量小，但 STITCH 的微调筛选过程（尤其是 Micro-level 的 LLM 打分）仍需消耗一定的推理算力。
*   **潜在的泄露风险**：尽管作者采取了禁用 Git 历史查询和 Web 搜索等措施，但由于 SWE 任务源于公开 GitHub 仓库，预训练阶段的数据污染风险依然难以完全排除。
