## 摘要
能够预测用户需求并自主执行任务的主动式智能体作为数字助手具有巨大潜力，但缺乏真实的交互用户模拟框架阻碍了其发展。现有方法将应用程序建模为扁平的工具调用 API，未能捕捉到数字环境中用户交互的有状态和顺序特性，使得真实的用户模拟变得不可行。我们推出了主动式智能体研究环境（Pare），这是一个用于在数字环境中构建和评估主动式智能体的框架。Pare 将应用程序建模为具有有状态导航和状态依赖动作空间的有限状态机，供用户模拟器使用，从而实现了活跃用户模拟。在此基础上，我们提出了 Pare-Bench，这是一个包含 143 个多样化任务的基准测试，涵盖了通信、生产力、日程安排和生活方式类应用，旨在测试上下文观察、目标推理、干预时机以及多应用编排能力。

## Abstract
Proactive agents that anticipate user needs and autonomously execute tasks hold great promise as digital assistants, yet the lack of realistic user simulation frameworks hinders their development. Existing approaches model apps as flat tool-calling APIs, failing to capture the stateful and sequential nature of user interaction in digital environments and making realistic user simulation infeasible. We introduce Proactive Agent Research Environment (Pare), a framework for building and evaluating proactive agents in digital environments. Pare models applications as finite state machines with stateful navigation and state-dependent action space for the user simulator, enabling active user simulation. Building on this foundation, we present Pare-Bench, a benchmark of 143 diverse tasks spanning communication, productivity, scheduling, and lifestyle apps, designed to test context observation, goal inference, intervention timing, and multi-app orchestration.

*   **核心挑战**：现有的评估框架将应用程序简化为扁平的 API 调用，忽略了真实用户在数字设备上必须通过“屏幕导航”和“顺序操作”来完成任务的特征。这种简化导致无法真实模拟用户与助手的互动过程，从而难以评估助手在目标推理、干预时机和多应用协作方面的表现。

### 2. 论文提出的方法论
*   **核心思想**：引入**非对称模拟**机制。用户模拟器必须像真人一样在有限状态机（FSM）建模的应用界面中导航，而主动助手则拥有直接调用后端 API 的权限。
*   **关键技术细节**：
    *   **有状态应用建模（FSM）**：将每个应用（如邮件、日历、打车）建模为有限状态机。用户必须执行“打开应用 -> 搜索 -> 点击 -> 填写”等一系列状态转换动作。
    *   **Stackelberg POMDP 建模**：将用户与助手的交互形式化为领导者-追随者博弈。用户（领导者）执行动作，助手（追随者）观察并决定是否提出建议（Proposal）。
    *   **观察非对称性**：用户只能看到当前屏幕和截断的通知；助手能看到完整的系统日志和用户的所有操作历史。
    *   **观察-执行（Observe-Execute）架构**：助手分为两个阶段：
        1.  **观察模式**：持续监测，仅使用只读工具收集信息，决定是否发送建议。
        2.  **执行模式**：在用户接受建议后，利用扁平 API 高效完成跨应用任务。

### 7. 优点（亮点）
*   **首创性环境设计**：通过 FSM 引入了用户与助手之间的“能力不对称性”，这比以往的扁平 API 模拟更贴近现实世界中“助手拥有系统权限，用户操作 UI”的场景。
*   **Observe-Execute 架构**：这种设计在保护用户自主权（必须确认后执行）的同时，提高了任务处理的效率。
*   **自动生成流水线**：利用 LLM 自动生成场景、数据和验证逻辑，保证了基准测试的可扩展性。

### 8. 不足与局限
*   **缺乏视觉感知**：目前 Pare 仍基于文本/API 交互，未涉及多模态（如屏幕截图）观察，这与真实的手机使用环境仍有差距。
*   **用户模型单一**：模拟用户虽然能模拟操作，但缺乏人类的复杂情感、疲劳感或对助手的信任度差异（Theory of Mind 建模不足）。
*   **长程任务覆盖有限**：目前的任务主要集中在短时间内完成，尚未覆盖跨越数天或需要复杂外部世界知识的长周期任务。
