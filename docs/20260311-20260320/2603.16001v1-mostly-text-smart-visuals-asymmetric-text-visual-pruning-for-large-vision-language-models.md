## 摘要
网络剪枝是实现轻量化大视觉语言模型（LVLMs）的一种有效技术，它主要将权重和激活值都纳入重要性度量中。然而，现有的研究通常以统一的方式处理来自不同模态的校准数据，忽略了模态特定的行为。这提出了一个关键挑战：如何处理文本和视觉 token 的差异化行为，以实现 LVLMs 的精确剪枝。为此，我们通过解耦相应的权重，系统地研究了视觉和文本 token 对剪枝操作的敏感性，结果表明：(i) 文本路径应通过文本 token 进行校准，因为它比视觉路径表现出更高的敏感性；(ii) 视觉路径表现出高度冗余，甚至允许 50% 的稀疏度。受这些见解的启发，我们为 LVLMs 提出了一种简单而有效的不对称文本-视觉权重剪枝方法，称为 ATV-Pruning，该方法通过从文本和视觉路径中选择信息丰富的 token 来建立精确权重剪枝的重要性度量。具体而言，ATV-Pruning 整合了两项主要创新：首先，通过利用所有文本 token 和视觉 token 的子集自适应地构建校准池；其次，我们设计了一种层自适应选择策略来产生重要的视觉 token。最后，在标准多模态基准测试上的广泛实验验证了我们的 ATV-Pruning 优于现有最先进的方法。

## Abstract
Network pruning is an effective technique for enabling lightweight Large Vision-Language Models (LVLMs), which primarily incorporates both weights and activations into the importance metric. However, existing efforts typically process calibration data from different modalities in a unified manner, overlooking modality-specific behaviors. This raises a critical challenge: how to address the divergent behaviors of textual and visual tokens for accurate pruning of LVLMs. To this end, we systematically investigate the sensitivity of visual and textual tokens to the pruning operation by decoupling their corresponding weights, revealing that: (i) the textual pathway should be calibrated via text tokens, since it exhibits higher sensitivity than the visual pathway; (ii) the visual pathway exhibits high redundancy, permitting even 50% sparsity. Motivated by these insights, we propose a simple yet effective Asymmetric Text-Visual Weight Pruning method for LVLMs, dubbed ATV-Pruning, which establishes the importance metric for accurate weight pruning by selecting the informative tokens from both textual and visual pathways. Specifically, ATV-Pruning integrates two primary innovations: first, a calibration pool is adaptively constructed by drawing on all textual tokens and a subset of visual tokens; second, we devise a layer-adaptive selection strategy to yield important visual tokens. Finally, extensive experiments across standard multimodal benchmarks verify the superiority of our ATV-Pruning over state-of-the-art methods.

#### 1. 核心问题与整体含义（研究动机和背景）
随着大型视觉语言模型（LVLMs）规模的不断扩大，其在资源受限设备上的部署面临巨大挑战。网络剪枝是减轻模型负担的关键技术，但现有方法（如 Wanda, SparseGPT）通常将文本和视觉 Token 视为同质数据进行统一校准。
*   **研究动机**：作者发现文本和视觉模态在激活分布和剪枝重要性上存在显著差异（模态异质性）。
*   **核心发现**：通过解耦实验证明，文本路径对校准数据高度敏感，必须使用文本 Token 校准；而视觉路径具有极高的冗余性，即使在 60% 的稀疏度下，仅靠文本校准也能保留绝大部分性能。

#### 2. 方法论：核心思想与关键技术
论文提出了 **ATV-Pruning（不对称文本-视觉剪枝）**，其核心思想是“文本全保留，视觉精细选”。
*   **不对称校准池构建**：校准集不再盲目使用所有 Token，而是包含**全部文本 Token** 和一个**自适应筛选的视觉 Token 子集**。
*   **层自适应视觉选择策略**：
    *   **视觉漂移（Visual Drift）**：利用 Token 在经过一个 Transformer 块前后的余弦距离作为显著性得分。漂移越大，说明该层对该 Token 的处理越活跃。
    *   **自适应预算分配**：根据每一层平均的视觉漂移程度，动态决定该层保留多少视觉 Token。活跃层保留更多，稳定层保留更少。
*   **重要性度量**：在构建的校准池上计算激活范数，结合权重幅值（遵循 Wanda 方案）计算重要性得分，最后进行非结构化或半结构化剪枝。

#### 3. 实验设计
*   **数据集/基准测试**：涵盖了 9 个标准多模态基准，包括 GQA、MMBench、MME、MMMU、OK-VQA、POPE、ScienceQA-IMG、TextVQA 和 VizWiz。
*   **对比方法**：
    *   **SparseGPT**：基于 Hessian 矩阵的权重更新剪枝。
    *   **Wanda**：经典的激活感知权重剪枝。
    *   **TAMP**：现有的多模态感知剪枝 SOTA 方法。
*   **测试模型**：LLaVA-NeXT (8B)、Qwen2-VL (7B)、LLaVA-OneVision (7B) 和 Qwen2.5-VL (7B)。

*   **硬件**：实验在一台 **NVIDIA A100 GPU** 上完成。
*   **效率**：ATV-Pruning 具有极高的效率。剪枝 LLaVA-NeXT (8B) 仅需约 **99.6 秒**，虽然比基础的 Wanda (73.7s) 略慢，但远快于 SparseGPT (666s) 和 TAMP (1418.4s)。

*   **实验规模**：论文在 4 种主流模型架构上进行了验证，涵盖了 50% 和 60% 的非结构化稀疏度，以及 2:4 和 4:8 的半结构化稀疏度。

*   **充分性评价**：实验设计客观公平，使用了统一的校准集（来自 ShareGPT4V 的 128 个样本）和官方评估工具（lmms-eval），覆盖了感知、推理、知识等多个维度，实验结果具有高度的说服力。

#### 6. 主要结论与发现
*   **非对称敏感性**：文本路径是 LVLM 的灵魂，校准时不可或缺；视觉路径参数冗余严重，50%-60% 的稀疏度几乎不损性能。
*   **文本 Token 的不可替代性**：实验显示，减少校准池中的文本 Token 会导致性能显著下降，证明了文本信息在维持跨模态对齐中的核心作用。
*   **性能领先**：ATV-Pruning 在高稀疏度（60%）下优势尤为明显，平均性能保留率显著高于 Wanda 和 TAMP，甚至在某些任务上超过了计算密集的 SparseGPT。

#### 7. 优点
*   **见解深刻**：通过解耦实验揭示了 LVLM 内部不同模态路径的敏感度差异，为多模态模型压缩提供了新视角。
*   **简单高效**：无需复杂的迭代优化或海量计算，仅通过改进校准池构建逻辑即可获得显著增益。
*   **通用性强**：在 LLaVA 系列和 Qwen 系列模型上均表现出稳定的提升。

#### 8. 不足与局限
*   **组件覆盖**：研究主要集中在 LLM 主干网络（Backbone）的剪枝，尚未深入探讨视觉编码器（Vision Encoder）与主干网络协同剪枝的影响。
*   **模态局限**：目前仅针对图像-文本模态，尚未扩展到音频、视频或点云等更多模态的大型多模态模型（LMMs）。
*   **硬件加速依赖**：虽然测试了半结构化剪枝，但非结构化剪枝在普通硬件上仍难以直接获得推理加速，需要专门的稀疏计算库支持。
